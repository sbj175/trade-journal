# OPT-168: Covered Calls misclassified as Short Call + Shares due to equity/option group separation

## Problem

The Ledger page shows zero "Covered Call" groups despite 84 Covered Call chains existing in `order_chains` (which the Reports page correctly displays). The PostgreSQL `position_groups` table has 154 "Short Call" entries and 54 "Shares" entries that should be paired as Covered Calls.

**Root cause:** The `assign_lots_to_groups()` function in `src/pipeline/group_manager.py` (lines 114-123) deliberately separates equity and option lots into different groups:

* Equity lots only update the `au_to_group` index
* Option lots only update `chain_to_group` and `aue_to_group`

This means shares and short calls for the same underlying never end up in the same group, so the strategy engine (which correctly detects Covered Calls in `patterns_combo.py`) never sees them together.

## Three gaps to fix

### 1\. Grouping: Covered Call rule in `assign_lots_to_groups()`

The equity-first separation was intentional to prevent cross-contamination (e.g., exercise-derived stock pulling option groups apart). But it's too aggressive — it prevents legitimate Covered Call grouping.

**Fix:** When a short call lot is being assigned and there's an open equity group for the same (account, underlying), merge them. The strategy engine will then see the combined legs and label it "Covered Call". The tricky part is doing this without breaking the separation for exercise/assignment-derived equity that should stay separate.

**Key file:** `src/pipeline/group_manager.py` — `assign_lots_to_groups()`

### 2\. Strategy re-evaluation on lot moves

When the user moves lots between groups on the Ledger page (`POST /api/ledger/move-lots`), `_refresh_group_status()` recalculates status and dates but **does not re-run the strategy engine**. This means:

* User drags shares into a Short Call group → label stays "Short Call" instead of updating to "Covered Call"
* User splits an Iron Condor into two spreads → labels don't update

This is the critical piece for the user regrouping goal — the whole point of lot moves is that strategy labels should adjust automatically.

**Fix:** After lot moves, load the group's lots, run `recognize(lots_to_legs(lots))`, and update `strategy_label` (unless `user_override` is set — see #3).

**Key files:** `src/services/ledger_service.py` — `_refresh_group_status()`, `src/routers/ledger.py` — `move_lots()`

### 3\. `user_override` flag for manual labels

From the OPT-123 design (never implemented): when a user manually edits a strategy label on the Ledger, flag it with `user_override = true` so that:

* Reprocessing/sync doesn't overwrite it
* Strategy re-evaluation after lot moves respects it
* The user can clear the override to let auto-detection take over again

**Database change:** Add `user_override` boolean column to `position_groups` (Alembic migration).

**Key files:** `src/database/models.py`, `src/routers/ledger.py` — `update_ledger_group()`, `src/pipeline/group_manager.py` — `GroupPersister.process_groups()`

## Implementation order

1. **#2 first** — strategy re-evaluation on lot moves (immediate value: users can manually fix Covered Calls by dragging shares into the group, and the label updates)
2. **#3 next** — `user_override` flag (protects manual fixes from being overwritten on next sync)
3. **#1 last** — auto-grouping fix (eliminates the need for manual fixes entirely)

## Verification

* After fix: PG `position_groups` should have \~84 "Covered Call" groups matching `order_chains`
* Lot moves should trigger strategy label updates
* Manual label edits should survive sync/reprocessing
