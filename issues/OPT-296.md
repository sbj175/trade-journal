# OPT-296: Chain cumulative realized — compute on-the-fly so Analysis B/E matches Roll Chain modal exactly

The new OPT-295 Analysis panel's "Chain B/E Close" is derived from `group.roll_chain.cumulative_realized_pnl`, which `src/routers/positions.py:171` reads straight out of the materialized `roll_chain_summaries` table. That table is only refreshed when the orchestrator's Stage 7 (`populate_roll_chain_summaries`) runs, so its value can lag the actual leg-by-leg sum the Roll Chain modal already displays correctly.

A live example surfaced 2026-05-18 on the Trad IRA IBIT 41.5 covered call (`5WZ26959`):

- Roll Chain modal summed 7 prior closed legs to **-$14,791** (and after today's leg-8 close, "Chain Realized: **-$71**", which is the same sum + leg 8).
- `roll_chain_summaries.cumulative_realized_pnl` at the same instant: **-$14,351** — stale by $440.
- Premium on the open leg: $30,320. With the stale cumulative the panel showed Chain B/E = **$2.00**; the true B/E using the modal's sum was **$1.94**.
- The user closed at $1.95 expecting a 5¢ cushion (small chain profit); actual result was a $71 chain loss.

## Scope

`src/routers/positions.py:get_open_chains` currently does:

```python
roll_chain_by_group[rc.current_group_id] = {
    ...,
    "cumulative_realized_pnl": rc.cumulative_realized_pnl,
}
```

Replace `cumulative_realized_pnl` with a fresh sum computed at request time:

1. For the current open group's chain (`rc.root_group_id` → all groups linking forward via `rolled_from_group_id`), collect every closed group's lots and sum the `realized_pnl` of all `lot_closings` rows. This is the same number the modal computes when it adds up each leg's "Realized" column.
2. Keep all other cached fields (`chain_length`, `roll_count`, `first_opened`, `last_rolled`, `cumulative_premium`) from the materialized row — they're not affected by the staleness bug.
3. If the chain has no prior closed groups, the on-the-fly sum is `0` (same as today's behavior).

`populate_roll_chain_summaries` continues to run as a write-time cache for everything else; we just stop trusting its `cumulative_realized_pnl` column for read-time displays of open-leg chain math.

## Out of scope

- Diagnosing why `populate_roll_chain_summaries` produced the stale value in the first place. The on-the-fly fix supersedes the cached value, so the bug stops mattering. Worth a separate look only if other consumers of `roll_chain_summaries.cumulative_realized_pnl` exist.
- Caching the on-the-fly result. The lots+closings batch already happens per-request inside `get_open_chains`; we just reuse those rows. No new DB round-trips.
- Live BTC-quote integration ("predicted chain total at current mid") — that was floated during OPT-295 discussion but is its own follow-up if useful.

### 2026-05-18 — implementation + verification

`src/routers/positions.py:get_open_chains` now walks each open group's `rolled_from_group_id` chain at request time and sums `lot_closings.realized_pnl` for every group in the chain. The result overrides `roll_chain_summaries.cumulative_realized_pnl` on the wire payload. Same semantics as `populate_roll_chain_summaries` but always fresh.

Verified end-to-end via DB walk on the user's IBIT 41.5 Trad IRA chain (now closed): traversed 8 groups, summed to exactly **-$71**, matching the Roll Chain modal's "Chain Realized". The Roth IBIT 41.5 closed at $1.90 with chain P&L -$12, i.e. the displayed Chain B/E of $1.90 was within a third of a cent of true B/E — no staleness on that side, but the fix protects against future occurrences.

Both 41.5 chains are now fully closed, so live in-browser verification on a still-open CC will have to wait for the next applicable position.
