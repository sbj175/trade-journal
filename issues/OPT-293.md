# OPT-293: Roll Analysis — show loss-target and profit-target exit prices for vertical spreads

The Roll Analysis panel today surfaces *state* (P&L, Greeks, R:R, EV, proximity). It does not surface the **prices** at which the trader's standard close-out rules would trigger. Add two derived numbers per vertical so the trader can place the closing order without doing the math in their head.

The percentages are **driven by the existing `StrategyTarget` rows** (`profit_target_pct` / `loss_target_pct`, already user-scoped per `strategy_name` and already editable in Settings). No new schema, no new settings UI — the Roll Analysis panel just reads the same values the rest of the app uses, and the trader can already tune them per strategy.

Default to `50%` / `50%` when no target row exists for a strategy.

Scoped initially to **debit spreads** (Bull Call, Bear Put). Credit spreads (Bull Put, Bear Call) are a natural symmetric extension — see "Open question" below.

## Numbers to compute (debit spreads)

For a debit spread with entry debit `D` per share, strike width `W`, and `StrategyTarget` row resolving to `loss_pct ∈ [0,100]` and `profit_pct ∈ [0,100]`:

- **Max loss** = `D` (paid up front)
- **Max profit** = `W − D`
- **Loss-target exit price** = `D × (1 − loss_pct/100)`  — sell-to-close order price; e.g. with `loss_pct = 50` the trader recovers half of the debit
- **Profit-target exit price** = `D + (W − D) × (profit_pct/100)`  — sell-to-close order price; e.g. with `profit_pct = 50` the trader captures half the upside

Both prices are quoted per share (matches how the option chain displays). Multiply by 100 × contract count for total dollars if needed for a separate "total credit on close" display, but per-share is what gets typed into the order ticket.

Label each row with the *resolved* percentage so the trader sees the math without opening Settings — e.g. "50% Loss Exit · $0.50" or "60% Profit Exit · $1.30".

## Where the values come from

- `D` (entry debit per share) — already derivable from the group's open transactions; same source the P&L Status section uses today. `usePositionsAnalysis.js` computes the current debit/credit basis when building `rollAnalysis`.
- `W` (strike width) — the absolute difference between the two leg strikes. Both legs are already in `group.optionLegs` after the enrichment loop in `useLedgerGroups.js` / equivalent positions composable.
- `loss_pct` / `profit_pct` — read from the `strategyTargets[strategy]` map that `usePositionsAnalysis.js` already destructures (`const targets = strategyTargets?.[strategy] || {}`). Fields are `loss_target_pct` / `profit_target_pct` on `StrategyTarget`. Default both to `50` when missing.

`D` and `W` are static for the life of the spread (no quote dependency, no recompute per tick). `loss_pct` / `profit_pct` only change when the trader edits them in Settings — the existing `targets` Pinia store already reacts to that.

## Display

Add to `frontend/src/components/PositionsRollAnalysis.vue`. Either:

1. **Inline in the P&L Status column** — add two rows below Reward:Risk: "50% Loss Exit" and "50% Max-Profit Exit", each showing the per-share order price. Keeps the column compact, no new section.

2. **New "Exit Targets" column** — a fourth column between Greeks and Context. More room for labels, possibly room for a credit-spread mirror.

Recommend **option 1** — fewer pixels, matches the diagnostic-style P&L Status section, and the values don't need much room (e.g., `$0.50` / `$1.25`).

Visual treatment:
- Same label-on-left, value-on-right pattern as the existing P&L Status rows.
- Color cue is optional; if used, keep it muted (these are *targets*, not *current state*). A light tv-blue would distinguish them from the colored P&L rows.

## Out of scope (for this issue)

- Computing the same targets for *roll candidates* (i.e., "if I roll up & out, what would the 50% loss exit price be on the new spread?"). That's an OPT-292-class follow-on — interesting but bigger.
- Order entry integration. The number is for the trader to type/paste into Tastytrade themselves.
- Custom thresholds (e.g., the trader prefers 40% of max profit). If demand emerges, surface as a setting under `/settings`.

## Open question

Should this ship with credit spreads at the same time, or in a follow-up?

For a credit spread with entry credit `C` per share, width `W`, and the same target percentages:
- **Profit-target exit price** = `C × (1 − profit_pct/100)` (buy-back; lower price closer to zero = more captured profit)
- **Loss-target exit price** = `C + (W − C) × (loss_pct/100)` (buy-back when that fraction of max-loss damage is realized)

Same shape, opposite direction (buy-to-close vs sell-to-close). Trader's mental model is "% of max profit" / "% of max loss," so the labels should normalize across debit/credit:

| Label (debit + credit share) | Debit spread formula | Credit spread formula |
|---|---|---|
| `profit_pct`% Profit Exit | `D + (W − D) × (profit_pct/100)` | `C × (1 − profit_pct/100)` |
| `loss_pct`% Loss Exit | `D × (1 − loss_pct/100)` | `C + (W − C) × (loss_pct/100)` |

The same `StrategyTarget` row applies to both — and the four supported vertical strategies already each get their own row in the existing Settings UI, so per-strategy tuning is already available the moment this lands.

Default recommendation: ship both at once with the normalized labels.

## Acceptance

- A Bull Call or Bear Put spread row, when expanded with Roll Analysis on, shows the two new prices.
- Prices respect the `StrategyTarget` row for that strategy. Editing the target in Settings updates the panel without a reload.
- Missing target falls back cleanly to `50%` / `50%`.
- The row label shows the resolved percentage (e.g. "60% Profit Exit"), so the trader doesn't have to remember what they configured.
- The prices match a hand-calculation against a known entry debit/credit, width, and target percentages.
- When the spread is currently trading at or past either threshold, the row's price still renders correctly (no NaN, no flip in sign).
- If extended to credit spreads in scope, both work; the labels carry the same meaning in both cases.
