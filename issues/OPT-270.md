# OPT-270: Roll chain breaks when two independent positions on the same underlying are rolled on the same day

## Repro (real case: IBIT, 2026-04-24)

User had two independent covered-call positions on IBIT, each with its own 8-long roll chain. Today both were rolled to different expirations (one 40.5C → 41C, one 43C → 43C). Expected: each roll continues its own chain, producing two independent 9-long chains. Actual: both chains terminate at yesterday's group, and today's new legs land in a single 2-leg group mis-labeled "Diagonal Spread."

## Root cause (two compounding bugs in `src/pipeline/group_manager.py`)

### Bug 1 — Independent chains fused into one group

When the pipeline processes today's opens, both new short-call lots get assigned to the same existing position_group even though their `lot.chain_id` values trace back to different parents.

Evidence from DB:

* Group 5519 now contains three lots:
  * lot 13686 — May-1 43C (CLOSED today) — chain `..._45354037`
  * lot 13688 — May-1 41C (OPEN) — chain `..._43949151` ← **different chain, should not be here**
  * lot 13691 — May-8 43C (OPEN) — chain `..._45354037`
* Same pattern in group 5520 with the 80-contract tranche.

The 41C belongs to the 40.5C's lineage. The 43C belongs to the original 43C's lineage. They are independent and should be in separate groups.

Group assignment appears to be keyed off "most recent open group on this underlying" rather than off `lot.chain_id`, which is the actual ground truth for roll lineage.

### Bug 2 — Roll-link detection keyed by `strategy_label`

Once a group ends up 2-leg, the recognizer labels it "Diagonal Spread" (two short calls, different strikes, different expirations). `_detect_roll_links` in `group_manager.py` (\~L461–550, specifically L512) then can't connect this child group back to its "Covered Call" predecessor because it uses `strategy_label` as a matching key. `rolled_from_group_id` is left `None`, terminating the chain.

Verification query: no group in the DB has `rolled_from_group_id` pointing to gid=5514 or gid=5516 (the last two "Covered Call" CLOSED groups in the prior chain).

## Proposed fix

1. **Group assignment follows** `lot.chain_id`. Each chain_id gets its own group. If a new open lot's chain_id matches an existing open group's chain_id, extend that group; otherwise create a new group whose `rolled_from_group_id` points to the group of the closed lot on the same chain.
2. **Roll-link detection uses chain/lineage, not** `strategy_label`. A closed lot's group is the predecessor of any new open lot derived from that close (same chain_id), regardless of what the recognizer labels either side.

This should also incidentally fix similar mis-labels on chains where a covered call roll got tagged "Diagonal Spread" in earlier syncs (e.g., today's groups 5519/5520 were backdated to opening_date 2026-04-17 — they were mis-fused at that earlier roll too; we just didn't notice because each chain was only 1–2 rolls old at that point).

## Deferred (follow-up)

Recognizer awareness of underlying share holdings so that "short call + shares" favors Covered Call over Diagonal Spread. Not strictly needed to fix the chain linkage, since fix (2) above makes the roll-link robust to label drift.

## Test plan

* Run `/api/reprocess` on IBIT and verify:
  * Each of the two roll lineages has its own 9-long chain.
  * New groups have `rolled_from_group_id` pointing to gid=5514 / gid=5516 (or equivalent predecessors).
  * Strategy labels on the new groups are "Covered Call," not "Diagonal Spread."
* Confirm the RollChainModal for either chain shows 9 entries, with today's roll at the top.
* Confirm Ledger UI still renders the old closed groups as part of the same chain (no regression on reports / pnl_events).

## Comments

### 2026-04-25 — Steve Johnson

Fix committed on `opt-270-chain-aware-group-routing` (`50e64ce`), pushed to origin.

**Changes (`src/pipeline/group_manager.py`):**
- Rule 1 in both `assign_lots_to_groups` and `GroupPersister` now requires the candidate group to be open. Without the check, a closed lot's stale expiration anchor pulled new opens from a different chain into the wrong group.
- `_detect_roll_links` no longer keys on `strategy_label`. Lineage is determined by `lot.chain_id` overlap + same-day open/close. Strategy label is a heuristic interpretation; it must not gate roll continuity.
- Added a docstring note codifying "Tier 4 (roll link) must not depend on Tier 3 (strategy label)."

**Verified on full reprocess:**
- Both 8-roll IBIT Covered Call chains (32-tranche and 80-tranche) reformed to length 9 with today's roll on top.
- Independent 43C lineage formed its own 3-roll chain, correctly separate.
- All four new groups for today's rolls labeled "Covered Call" by the recognizer (no longer "Diagonal Spread").
- No regressions on multi-leg strategies: Bull/Bear spreads still group correctly via shared `chain_id` from the single-order entry. 6 each of Bull Call / Bear Put / Bull Put on the books, all 4-lot groups with chain_count=1.

**Deferred follow-up (separate issue):** recognizer awareness of underlying share holdings so single short-call rolls always identify as "Covered Call" without relying on the existing upgrade path.

Branch ready for review/merge.
