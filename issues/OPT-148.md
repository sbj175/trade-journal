# OPT-148: Enforce tenant isolation on DELETE/UPDATE ORM operations

The ORM tenant filter in `tenant.py` only fires on SELECT queries. Bulk DELETE and UPDATE operations (`session.query(Model).filter(...).delete()` / `.update()`) bypass tenant scoping entirely.

While all existing call sites now have explicit `user_id` filters (fixed in OPT-147), new code could easily forget this. Extend the `do_orm_execute` listener to also inject `WHERE user_id = ?` on DELETE and UPDATE statements.

**Changes:**

1. Extend `do_orm_execute` listener to handle `is_delete` and `is_update` in addition to `is_select`
2. Change `.filter()` → `.where()` since Delete/Update statement objects don't support `.filter()`
3. Add unit tests for bulk DELETE and UPDATE tenant isolation
