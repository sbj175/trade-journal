# OPT-206: Improvements to Strategy detection

What you’re running into is essentially a **pattern recognition / partitioning problem** over a small set of option legs. The key difficulty is that **the same legs can belong to multiple valid strategies**, and if you greedily group them you can misclassify (exactly the vertical + extra leg problem you mentioned).

A reliable way to solve this **in one pass without user correction** is to treat strategy identification as a **scoring / matching problem**, not a greedy grouping problem.

Below is a practical approach used in trading systems.

---

## 1\. Normalize the Legs First

Group legs by:

* **Underlying**
* **Expiration**

Most multi-leg strategies only combine legs with the same expiration (except diagonals/calendars which you can treat separately).

So step one:

```
for each underlying:
    for each expiration:
        analyze legs in this bucket
```

This dramatically reduces complexity.

---

## 2\. Represent Legs Structurally

Convert each leg into a normalized structure:

```
{
    type: CALL | PUT
    side: LONG | SHORT
    strike: float
    qty: int
}
```

Then sort by:

```
(type, strike)
```

---

## 3\. Define Strategy Patterns (Templates)

Instead of grouping legs first, **detect strategies by templates**.

Examples:

### Vertical Spread

```
CALL or PUT
1 long
1 short
different strikes
```

### Jade Lizard

```
short put
short call
long call (higher strike)
```

### Iron Condor

```
long put
short put
short call
long call
```

### Straddle

```
short call
short put
same strike
```

Each template becomes a **predicate function** that checks if a set of legs matches.

---

## 4\. Generate Candidate Combinations

For a bucket of *n legs*, generate combinations of size:

```
2 legs
3 legs
4 legs
```

Because strategies rarely exceed 4 legs.

For each combination:

```
for combo in combinations(legs, 2..4):
    for strategy in strategy_templates:
        if strategy.matches(combo):
            candidates.append(...)
```

---

## 5\. Score Each Candidate

This is the crucial part.

Give each detected strategy a **confidence score**.

Example heuristic scoring:

| Feature | Points |
| -- | -- |
| Exact strike symmetry | +5 |
| Equal quantities | +3 |
| Common retail strategy | +2 |
| Leaves no orphan legs | +2 |
| Strike ordering correct | +3 |

Example:

```
Jade Lizard = score 12
Vertical + naked call = score 7
```

---

## 6\. Solve the Partition

Now you choose the **set of strategies that covers all legs with the highest total score**.

This is basically a **maximum weighted set packing problem**.

But because there are rarely more than \~6 legs, brute force works.

Pseudo:

```
best_solution = None
best_score = -inf

for partition in all_valid_partitions(candidates):
    if no_leg_overlap(partition):
        score = sum(strategy.score)
        if score > best_score:
            best_solution = partition
```

This ensures:

* Jade Lizard wins over
* Vertical + random call

because it scores higher.

---

## 7\. Why This Works

Your failure mode today:

```
short put
short call
long call
```

Greedy detection might do:

```
vertical call spread
+ naked short put
```

But scoring will prefer:

```
jade lizard
```

because the template score is higher.

---

## 8\. Performance

This sounds expensive but it isn't.

Typical retail positions:

```
2–6 legs
```

Possible combos:

```
C(6,4) = 15
```

Total candidate checks ≈ trivial.

---

## 9\. Implementation Tip (Important)

Create a **canonical signature** for strategies.

Example Jade Lizard signature:

```
PUT: -1
CALL: -1
CALL: +1 higher strike
```

Vertical call spread:

```
CALL: +1
CALL: -1
```

Matching signatures makes detection extremely clean.

---

## 10\. Bonus: How Brokerages Do It

Broker systems (TDA, IBKR) generally:

1. Detect **maximal strategies first** (iron condor, jade lizard, butterfly).
2. Then fall back to smaller spreads.
3. Finally classify leftovers as **naked legs**.

This ordering alone solves many cases.
