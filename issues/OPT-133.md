# OPT-133: Research: AI Insights integration for OptionLedger

## Overview

Investigate adding AI-powered insights to OptionLedger, starting with trade/roll analysis on the Ledger page and potentially expanding to other pages.

## Current State

* No AI/LLM integration exists in the app today
* No dedicated Roll Analysis feature — rolls are tracked in order chains (`ROLLING` order type, `linked_order_id`, sequence numbers) and visible in Chains/Ledger UI, but no analysis layer sits on top
* Rich structured data already available: complete trade chains with temporal ordering, leg-level P&L, Greeks via Black-Scholes, strategy categorization, win/loss outcomes

## Potential Integration Points

### 1\. Trade Chain Narrative / Post-Mortem

Generate plain-English summaries of position groups: opening → rolls → close, with P&L context. Pure text generation from structured data, no market prediction needed.

### 2\. Roll Analysis Panel (Ledger or new page)

Feed a chain's full history to the LLM and get:

* Was rolling optimal vs. closing and re-entering?
* Roll cost analysis (credits/debits at each step)
* Strike adjustment pattern assessment
* DTE management evaluation

### 3\. Position Risk Commentary (Positions or Risk page)

Live positions + current Greeks → actionable observations about tested sides, DTE urgency, delta drift, etc.

### 4\. Performance Pattern Recognition (Reports page)

Aggregate trade history → insights about win rate vs. risk/reward imbalances, strategy-specific patterns, position sizing observations.

## Implementation Options

| Approach | Pros | Cons |
| -- | -- | -- |
| **Anthropic API (Claude)** | Best reasoning for trade analysis, tool use | Requires API key + cost per call |
| **OpenAI API** | Well-known, good structured output | Same cost/key concerns |
| **Local LLM (Ollama)** | Free, private, no API key | Weaker reasoning, needs GPU |

Cloud API (user provides own key) and local Ollama could both be supported given the app's privacy-first design.

## Suggested Starting Point

A **"Get AI Insight" button on the Ledger page** for a selected position group:

1. Gather the group's lots, orders, P&L, and roll history
2. Send to Claude API with a trading-analysis system prompt
3. Display the response in a panel below the group

## Open Questions

* Which LLM provider(s) to support initially?
* Where should the API key be stored? (Settings page, `.env`, or both)
* Should insights be cached/persisted or generated on demand?
* Scope of first iteration — single group analysis only, or broader?

## Data Requirements for AI Insights

### Already available in the app (pipe into LLM context)
- **Delta** — spread responsiveness to price moves; near-ATM vs far OTM risk profile
- **Theta** — exact daily decay cost; critical for "can I afford to wait?" decisions
- **Gamma** — delta shift rate; most important inside 14 DTE
- **Vega** — IV exposure; most important for 45+ DTE positions or pre-earnings
- **IVR** — current implied volatility rank (shown on Positions page)

All four Greeks are computed via Black-Scholes on the Risk page already. Need to make them available in the position data passed to the LLM.

### Not currently in the app (would strengthen analysis)
- **IV Rank/Percentile trend** — is IV expanding or contracting? Determines whether vega exposure is helping or hurting
- **Probability of Profit (PoP)** — derivable from delta (~`1 - delta` of the long strike for put spreads). Turns "5.2% away from strike" into "18% chance of profit"
- **Expected Value (EV)** — `PoP × max_profit` vs `(1-PoP) × max_loss`. The real decision metric: is staying in the trade +EV or -EV?

### Priority for first iteration
Delta, theta, and IV trend cover ~90% of cases. Gamma is critical inside 14 DTE. Vega matters most at 45+ DTE or pre-earnings. PoP and EV are the most actionable derived metrics to add.

### Data pipeline summary
The Risk page already computes per-position Greeks via Black-Scholes. The main work is:
1. Making Greeks available in the Ledger/Positions API response (or a dedicated insights endpoint)
2. Computing PoP and EV from existing delta + max profit/loss values
3. Passing all of this as structured context to the LLM prompt

## Correction: Greeks Data Source

The previous comment incorrectly stated Greeks are "computed via Black-Scholes on the Risk page." In reality:

**Primary source: Real broker Greeks from Tastytrade (DXFeed streaming)**
- Delta, gamma, theta, vega, rho, IV, IVR streamed via `DXLinkStreamer` in `tastytrade_client.py`
- Cached in `quote_cache` table with full Greeks columns
- Delivered to frontend via WebSocket at `/ws/quotes`

**Fallback only: Black-Scholes (client-side)**
- Used when DXFeed Greeks subscription is unavailable (depends on user's TT market data tier)
- Uses broker IV as input when available, hardcoded 30% default otherwise
- Risk page shows source indicator: "DXFeed + Black-Scholes" vs "Black-Scholes"

**Implication for AI Insights:** We have real broker-provided Greeks available in most cases. The insights endpoint can pull them from `quote_cache` — no need to compute them server-side. This makes the data pipeline simpler and the analysis more accurate.
