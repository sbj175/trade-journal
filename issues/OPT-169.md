# OPT-169: Migrate Reports page from order_chains to position_groups + lots

## Goal

Eliminate the Reports page's dependency on `order_chains` by querying `position_groups` + `position_lots` + `lot_closings` instead. This is a key step toward eventually removing the chains system entirely.

## Current state

The Reports page (`/api/reports/performance`) queries `OrderChain` directly for:

* `strategy_type` — strategy label
* `total_pnl` — realized P&L
* `opening_date` / `closing_date` — date filtering
* `chain_status` — CLOSED filter
* `account_number` — account filtering

The Dashboard endpoint (`/api/dashboard`) also queries chains via `order_manager.get_order_chains()`.

## What position_groups already has

| Field | Status |
| -- | -- |
| `strategy_label` | ✓ Exists, but incorrect for some strategies (OPT-168 fixes this) |
| `status` (OPEN/CLOSED) | ✓ Exists |
| `opening_date` | ✓ Exists (from lots) |
| `closing_date` | ✓ Exists (from lot_closings) |
| `account_number` | ✓ Exists |
| `realized_pnl` | ✗ Missing — must be computed from lots |

## What needs to happen

### 1\. Compute group P&L from lots

Position groups don't store P&L. It needs to be derived:

```
For each group:
  realized_pnl = sum of lot_closings.realized_pnl for all lots in the group
```

Two options:

* **Option A:** Compute on the fly via SQL join (group → group_lots → lots → lot_closings)
* **Option B:** Add a `realized_pnl` column to `position_groups` and update it during `_refresh_group_status()` and `GroupPersister.process_groups()`

Option B is simpler for queries and keeps the Reports endpoint fast. The value is already computable — `lot_closings` has `realized_pnl` per closing.

### 2\. Rewrite `/api/reports/performance`

Replace:

```python
q = session.query(OrderChain).filter(OrderChain.chain_status == 'CLOSED')
```

With:

```python
q = session.query(PositionGroup).filter(PositionGroup.status == 'CLOSED')
```

Map fields: `strategy_type` → `strategy_label`, `total_pnl` → `realized_pnl`, dates and account stay the same.

### 3\. Rewrite `/api/dashboard`

Replace `order_manager.get_order_chains()` with a position_groups query. The dashboard computes summary stats (total P&L, win rate, strategy breakdown) — all derivable from groups.

### 4\. Rewrite `/api/reports/strategies`

Currently queries `OrderChain.strategy_type`. Switch to `PositionGroup.strategy_label`.

### 5\. Max risk/reward calculation

`report_service.calculate_max_risk_reward()` currently takes a `chain_id` and queries `OrderChainMember` → `Order` → `OrderPosition` to find the legs. This needs to be rewritten to derive risk/reward from the group's lots instead (lot strikes, types, and quantities).

## Depends on

* **OPT-168** — Covered Calls and strategy labels must be correct on position_groups first
* Lot-based P&L must be accurate (already the case — lots are source of truth)

## What this unblocks

After this, `order_chains` is only used by:

* Sync/reprocessing (chain_graph.py builds them, group_manager.py reads them for grouping)
* Positions page (OrderChainCache for order history display)
* Debug endpoint

A follow-up issue could then remove chains from the pipeline entirely, using lot connections (`order_id` / `closing_order_id`) for grouping instead of chain membership.
