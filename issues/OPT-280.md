# OPT-280: Roll detection misses same-day legged-out-and-back-in rolls (chain-only filter)

## Bug

The roll-chain modal's "learn more" section explicitly documents:

> A roll is automatically detected when a position is closed and a new one is opened matching on: Same account, Same underlying, Same type (Call/Put), Same day. Whether you're rolling your Covered Calls or **legging out and back into a new Iron Condor**, the roll is detected.

Today's `_detect_roll_links` requires shared `chain_id` between source and target groups. That filter only succeeds when the close and the open happen in a single ROLLING broker order (so the new lots inherit chain identity through Stage 3). When a trader closes one position and opens another the same day as **separate broker orders** — exactly the documented "legged out and back in" case — no chain overlap exists and the link doesn't form.

## Real-world example (USO Iron Condors)

Four 4-leg Iron Condors that the user rolled month-to-month, each as a fresh broker order:

```
gid=8723  Jul 18 exp  closed Jun 27
gid=8741  Aug 15 exp  opened Jun 27, closed Jul 25     ← should rolled_from gid 8723
gid=8768  Sep 19 exp  opened Jul 25, closed Aug 22     ← should rolled_from gid 8741
gid=8793  Oct 17 exp  opened Aug 22, closed Sep 26     ← should rolled_from gid 8768
```

All four `rolled_from_group_id = NULL` today.

## Why the implementation drifted

[OPT-244](https://linear.app/optionedge/issue/OPT-244/roll-not-detected-when-additional-leg-opened-alongside-roll) added the chain-overlap filter to suppress false roll links on split-off OPENING orders. [OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same) then removed `strategy_label` from the matching key (recognizer is heuristic, must not gate roll continuity). What remained was chain_id alone — stricter than the documented behavior.

## Fix

Match groups by **leg-shape signature** as a fallback when chain overlap doesn't fire. The signature is a multiset of `(option_type, sign-of-quantity)` across the group's lots. Two groups with the same signature are structurally the same shape (e.g., both 4-leg ICs with 2 puts + 2 calls + matching directions).

Algorithm:

1. Find candidates by (account, underlying, source.closing_day == target.opening_day) — existing
2. Prefer chain-overlap candidates if any exist — existing
3. **Fall back to signature-match candidates** if no chain candidates — new
4. Tie-break by closest lot count — existing

This keeps every passing case green:

* IBIT parallel-rung rolls: chain match wins (preferred over signature)
* [OPT-261](https://linear.app/optionedge/issue/OPT-261/position-groups-dont-reset-when-all-lots-close-before-new-lots-open) Bull Call Spread → Put Butterfly: signatures differ ((C,L)+(C,S) vs (P,L)×2+(P,S)×2) → no link
* Same-exp roll within a single ROLLING order ([OPT-279](https://linear.app/optionedge/issue/OPT-279/same-expiration-rolls-create-new-groups-instead-of-staying-in-one)): chain match wins

And adds:

* USO IC → IC across separate broker orders → signature match → link forms ✓
