# OPT-121: Architecture: Redesigned data pipeline with cross-order strategy recognition

## Design Philosophy

The current pipeline was built incrementally and it shows — parallel tracking systems, a separate equity pass, fragile chain-to-lot mappings captured mid-processing, and strategy detection that only sees one order at a time. This redesign starts from the requirements in [OPT-120](https://linear.app/optionedge/issue/OPT-120/documentation-data-pipeline-from-raw-transactions-to-position-groups) and rebuilds with three guiding principles:

1. **Position-centric, not order-centric.** Strategy detection looks at *what you hold*, not *how you got there*. An Iron Condor is an Iron Condor whether you placed it as one order or four.
2. **One pipeline for everything.** Options and equities flow through the same stages. No second pass, no bolted-on netting.
3. **Chains emerge from lots, not the other way around.** Orders are linked into chains by the lots they share — a graph traversal, not a fragile pre-capture mapping.

---

## Pipeline Overview

```
Tastytrade API
     │
     ▼
┌──────────────────────────┐
│  1. Transaction Ingest   │  Fetch and store raw transactions (unchanged)
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│  2. Order Assembly       │  Group fills → orders, classify open/close/roll
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│  3. Position Ledger      │  Unified FIFO lot tracking (options + equity)
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│  4. Chain Graph          │  Derive chains by walking lot connections
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│  5. Strategy Engine      │  Cross-order strategy recognition on position snapshots
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│  6. Position Groups      │  First-class user-facing groups with live strategy labels
└──────────┴───────────────┘
```

**What's different from today:**

* Stages 3 and 4 are swapped. Today: chains are built during lot processing (tangled). New: lots are created first, chains derived after (clean separation).
* Stage 5 is entirely new. Today: strategy detection runs once on a single opening order. New: runs on position snapshots, sees all legs regardless of how many orders placed them.
* Stage 3 is unified. Today: options go through OrderProcessor, equity goes through a separate `process_equity_transactions()` pass with a netting hack. New: everything goes through the same ledger.
* `positions_inventory` is eliminated. Lots are the only position tracking system.

## Stage 1: Transaction Ingest (unchanged)

Same as today. Fetch from Tastytrade API, upsert into `raw_transactions`. Filter out futures/futures options. This stage is solid and needs no changes.

## Stage 2: Order Assembly

**Input:** Raw transaction dicts from `raw_transactions`
**Output:** `Order` objects — classified, normalized, with parsed option details

This stage is largely the same as today's OrderProcessor preprocessing + grouping + classification. The key changes:

**Include equity transactions in the main flow.** Today, pure equity buy/sell transactions are excluded from the OrderProcessor and handled in a separate pass. In the redesign, they flow through the same pipeline. The grouping key remains `(account, underlying, order_id)`.

**Classification stays the same:** OPENING, CLOSING, ROLLING — determined by whether transactions are "to open", "to close", or both.

**Special handling preserved:**

* Symbol changes: same synthetic order ID logic (SYMCHG_CLOSE / SYMCHG_OPEN)
* Expirations/assignments/exercises: same synthetic order ID generation
* Fill aggregation: same normalization by (action, symbol, option_type, strike, expiration, price)
* Assignment stock transactions (no order_id): tagged as `source: 'assignment'` instead of being excluded. They'll be handled in Stage 3 as derived lots.

**What's eliminated:**

* No `_closing_order_to_chains` pre-capture. This side-channel mapping is the most fragile part of the current system. It exists because today's pipeline tries to build chains *during* lot processing. The redesign builds chains *after*.
* No `_assignment_stock_transactions` side list. Assignment-derived equity is handled inline in Stage 3.

## Stage 3: Position Ledger (unified FIFO)

**Input:** Orders in chronological sequence
**Output:** `position_lots` and `lot_closings` records — the complete lot lifecycle

This is the core tracking engine. It replaces both `PositionInventoryManager` and the current `LotManager`, unified into one system that handles all instrument types.

### Processing loop

For each order, chronologically:

```
for order in orders_by_date:
    for tx in order.transactions:
        if tx.is_opening:
            create_lot(tx, order_id=order.id)
        elif tx.is_closing:
            close_lots_fifo(tx, order_id=order.id)
    
    if order.has_assignments:
        create_derived_lots(order)
```

### Lot creation

Same as today. Each opening transaction creates a lot with:

* `transaction_id` (stable Tastytrade ID, survives reprocessing)
* `order_id` (which order created this lot) ← **NEW field**
* `symbol`, `underlying`, `instrument_type`
* `quantity`, `remaining_quantity`, `entry_price`, `entry_date`
* `option_type`, `strike`, `expiration` (null for equity)

The new `order_id` field on lots is what enables chain derivation in Stage 4.

### FIFO closing

Same as today — match oldest open lots first, create `lot_closings` records. But now each `lot_closings` also stores:

* `closing_order_id` ← **NEW field** — which order triggered this closing

This is the second half of what enables chain derivation. With `order_id` on lots (who opened) and `closing_order_id` on closings (who closed), we can build a complete graph of which orders are connected.

### Derived lots (assignments)

When an assignment is detected:

1. Close the option lot with `closing_type='ASSIGNMENT'`
2. Match to the corresponding stock transaction (same 60-second window + quantity check as today)
3. Create a derived equity lot with `derived_from_lot_id` pointing back

**Exercises** should also create derived lots (today they don't — a known bug).

### Equity handling (no separate pass)

Regular stock buy/sell orders flow through the same loop. A Buy to Open creates a stock lot. A Sell to Close closes stock lots via FIFO. No special handling needed.

**Netting of opposing positions:** Today this is a post-hoc hack in `ledger_service.py`. In the redesign, it happens naturally: if assignment creates -800 shares (short stock lot) and an ACAT transfer creates +800 shares (long stock lot), a subsequent equity close of either direction will FIFO-match against the oldest lot. If we need explicit netting (no user-initiated close, just cancellation), a `net_opposing_lots()` sweep runs once at the end — same logic, but part of the main pipeline, not a bolted-on pass.

### What's eliminated

* `positions_inventory` table and `PositionInventoryManager` — lots are the only tracking system
* Separate `process_equity_transactions()` pass
* The `_closing_order_to_chains` pre-capture hack

## Stage 4: Chain Graph

**Input:** All lots and lot_closings (with their `order_id` / `closing_order_id` fields)
**Output:** `order_chains` with members

### The insight

A chain is a **connected component** in a graph where:

* **Nodes** = orders
* **Edges** = lots that connect an opening order to a closing order

If Order A opens a lot and Order B closes that lot, A and B are in the same chain. If Order B is a roll that also opens new lots closed by Order C, then A, B, and C are all in the same chain.

### Algorithm

```
1. Build adjacency:
   - For each lot: edge between lot.order_id (opener) and each lot_closing.closing_order_id (closer)
   
2. Find connected components (union-find or BFS/DFS):
   - Each component = one chain
   
3. Determine chain metadata:
   - chain_id: derived from earliest order's underlying + date + id
   - status: OPEN if any lot has remaining_quantity > 0; CLOSED otherwise; ASSIGNED if assignment event + open stock lots
   - open_date: earliest order date
   - close_date: latest closing date (if closed)
   - orders: all orders in the component, sorted chronologically
```

### Why this is better

* **No fragile pre-capture mapping.** Today's `_closing_order_to_chains` captures chain IDs *before* lots are closed, because after closing, the lot's chain_id can't be queried from open lots. The redesign doesn't need this — chain derivation runs after ALL lots are processed, using the `order_id` / `closing_order_id` foreign keys.
* **Chain merging is free.** If a closing order closes lots from two different "proto-chains", union-find naturally merges them into one connected component. No special merge logic needed.
* **Orphan chains are explicit.** An opening order with no closings is simply a single-node connected component. No special orphan creation needed.
* **Rolls are transparent.** A rolling order has both incoming edges (closing old lots) and outgoing edges (opening new lots), naturally bridging two parts of the chain.

### Database changes

* `position_lots`: add `order_id` column (which order created this lot)
* `lot_closings`: add `closing_order_id` column (which order closed this lot)
* `order_chains`, `order_chain_members`: same as today
* Remove: `positions_inventory` table

## Stage 5: Strategy Engine (NEW)

This is the headline feature of the redesign. Strategy detection operates on **position snapshots** — the set of all open legs for an account+underlying — regardless of how many orders were used to build the position.

### Core API

```python
@dataclass
class Leg:
    instrument_type: str    # 'EQUITY_OPTION' or 'EQUITY'
    option_type: str | None # 'Call', 'Put', or None for equity
    strike: float | None
    expiration: date | None
    direction: str          # 'long' or 'short'
    quantity: int           # always positive, direction indicates sign

@dataclass
class StrategyResult:
    name: str               # e.g. "Iron Condor", "Covered Call"
    confidence: float       # 0.0 to 1.0
    legs: dict[str, list]   # role → legs mapping, e.g. {"short_put": [...], "long_put": [...]}
    notes: str | None       # e.g. "Built across 2 orders"

def recognize(legs: list[Leg]) -> StrategyResult:
    """Pure function. Given open legs, return the best matching strategy."""
```

### Detection hierarchy (most specific first)

**4+ leg strategies:**

| \# | Strategy | Pattern |
| -- | -- | -- |
| 1 | **Iron Condor** | Bull put spread + bear call spread, same expiration. Short strikes inside long strikes. |
| 2 | **Iron Butterfly** | Same as Iron Condor but short call strike == short put strike |
| 3 | **Butterfly** (call or put) | 3 strikes in same type: buy 1 low + sell 2 mid + buy 1 high (or inverse for short butterfly) |
| 4 | **Double Calendar** | 2 calendars on same underlying at different strikes |
| 5 | **N-Leg Strategy** | Fallback for any unrecognized 4+ leg combo |

**3-leg strategies:**

| \# | Strategy | Pattern |
| -- | -- | -- |
| 6 | **Broken Wing Butterfly** | 3 strikes but unequal wing widths |
| 7 | **Jade Lizard** | Short put + bear call spread, same expiration |
| 8 | **Collar** | Long stock + long put + short call |
| 9 | **3-Leg Strategy** | Fallback |

**2-leg same-type strategies:**

| \# | Strategy | Pattern |
| -- | -- | -- |
| 10 | **ZEBRA** (bull/bear) | 2:1 ratio, same type, same expiration |
| 11 | **Calendar Spread** | Same type, same strike, different expirations |
| 12 | **Diagonal Spread** | Same type, different strike, different expirations |
| 13 | **PMCC** | Long deep-ITM far-dated call + short near-dated OTM call (a specific diagonal) |
| 14 | **Bull Call / Bear Call / Bull Put / Bear Put Spread** | Same type, same expiration, different strikes, 1:1 ratio |

**2-leg different-type strategies:**

| \# | Strategy | Pattern |
| -- | -- | -- |
| 15 | **Straddle** (long/short) | Call + put, same strike, same expiration |
| 16 | **Strangle** (long/short) | Call + put, different strikes, same expiration |

**Stock + option strategies:**

| \# | Strategy | Pattern |
| -- | -- | -- |
| 17 | **Covered Call** | Long stock (≥ 100 shares per contract) + short call |
| 18 | **Protective Put** | Long stock + long put |
| 19 | **Covered Strangle** | Long stock + short put + short call |
| 20 | **Cash-Secured Put** | Short put (cash analysis optional, can infer from account type) |

**Single-leg strategies:**

| \# | Strategy | Pattern |
| -- | -- | -- |
| 21 | Long Call / Short Call / Long Put / Short Put / Shares | Single leg |

### When does detection run?

**On every position change.** After Stage 3 processes an order that affects an underlying, the Strategy Engine runs on the updated position snapshot for that underlying+account. The result is stored on the position group.

This means strategy labels **evolve over time**:

1. You sell a put spread → group labeled "Bull Put Spread"
2. You sell a call spread on the same underlying+expiration → label updates to "Iron Condor"
3. The call spread expires → label reverts to "Bull Put Spread"

This is a feature, not a bug. It accurately reflects what the position IS at each point in time.

### Strategy history

To preserve the evolution, we store a `strategy_history` on the group:

```json
[
  {"strategy": "Bull Put Spread", "from_date": "2025-01-15", "to_date": "2025-01-20"},
  {"strategy": "Iron Condor", "from_date": "2025-01-20", "to_date": "2025-03-21"},
  {"strategy": "Bull Put Spread", "from_date": "2025-03-21", "to_date": null}
]
```

This enables the Reports page to attribute P&L to the correct strategy at the time the P&L was realized.

### Progression strategies (Wheel)

The Wheel is a repeating cycle: Short Put → Assignment → Covered Call → Called Away → repeat. It's not a position pattern — it's an **event sequence**. The Strategy Engine detects it by examining chain history:

```
if chain has:
  (1) Short Put order, followed by
  (2) Assignment event, followed by
  (3) Short Call order on the assigned stock
then:
  label = "Wheel" (with phase indicator: "Wheel: Covered Call phase")
```

This uses the chain's order history (from Stage 4) plus the strategy snapshot (from the recognize function). The Wheel label is a special overlay that takes priority when the event sequence matches, even though the current snapshot might just show "Covered Call".

## Stage 6: Position Groups (first-class)

Today, groups are a bolt-on layer seeded from chains after the fact. In the redesign, groups are a first-class concept created alongside lots.

### When groups are created

* **On first opening order for an underlying+account:** A new group is created.
* **On subsequent orders for the same underlying+account+expiration:** Lots are added to the existing group (if the position overlaps).
* **On roll:** The group persists — the old lot is closed and the new lot is added to the same group.

### Group identity

Groups are identified by `group_id` (UUID, stable) and linked to lots via `transaction_id` (stable Tastytrade ID). This is the same survivability mechanism as today, but groups are created at lot-creation time instead of after-the-fact seeding.

### Auto-grouping logic

When a new lot is created, the system decides which group it belongs to:

1. **Same chain?** If the lot's order is in the same chain as an existing open group → add to that group.
2. **Same underlying+expiration?** If there's an open group for the same underlying+account with lots sharing the same expiration → add to that group (cross-order strategy building).
3. **New group.** Otherwise, create a new group.

Rule 2 is what enables cross-order Iron Condors. When you sell a put spread (group created, labeled "Bull Put Spread") and then sell a call spread on the same underlying+expiration, the call spread lots join the existing group. The Strategy Engine then re-evaluates and updates the label to "Iron Condor".

### Manual override preserved

Users can still manually move lots between groups and override strategy labels on the Ledger page. Manual labels are flagged as `user_override: true` and are not auto-updated by the Strategy Engine.

### Group status

Same logic as today: if any lot has `remaining_quantity > 0`, the group is OPEN. Otherwise CLOSED. But computed inline, not as a post-hoc refresh.

## Edge Cases Preserved

Every edge case from the current system is accounted for in the redesign:

| Edge Case | Current Handling | Redesigned Handling |
| -- | -- | -- |
| **Symbol changes** (corporate actions) | Synthetic order IDs, underlying override | Same — Stage 2 unchanged |
| **Assignments → derived stock lots** | Two-phase: close option, match stock tx in 60s window, create derived lot | Same — Stage 3, but inline instead of post-hoc |
| **Exercises → derived stock lots** | Bug: not implemented | Fixed — same derived lot logic as assignments |
| **Expirations** | Synthetic order IDs, zero-price skip | Same — Stage 2 unchanged |
| **Cash settlement** | Uses tx.value directly instead of price × qty × 100 | Same — P&L calculator unchanged |
| **Rolls** | Chain inheritance via pre-capture mapping | Natural — roll order is a node with edges to both old and new lots, same connected component |
| **Chain merging** | Explicit merge with `_MERGED` suffix | Free — union-find produces one component |
| **Orphan chains** | Explicit orphan creation | Natural — single-node connected component |
| **Partial fills** | Aggregation by (action, symbol, price) | Same — Stage 2 normalization unchanged |
| **FIFO exhaustion** | Silent ignore of remaining quantity | Same — but with a warning log |
| **Multi-tenant user_id** | ORM events + manual dialect_insert | Same — unchanged |
| **Equity netting** | Separate post-hoc pass | End-of-pipeline sweep, same logic, integrated |
| **ACAT transfers** | Receive Deliver BTO creates lots, STC skipped | Same — Stage 3 recognizes Receive Deliver opens |
| **Covered Call stock check** | Query raw_transactions for net stock position | Lot-based: check open equity lots for the underlying (more accurate) |
| **Merged chain strategy preservation** | Pre-save strategies for MERGED chains | Not needed — strategies are on groups, not chains. Groups persist across reprocessing. |

## Module Structure

```
src/pipeline/
├── __init__.py
├── orchestrator.py          # Runs stages 2-6 in sequence, handles incremental vs full
├── order_assembler.py       # Stage 2: raw txs → classified orders
├── position_ledger.py       # Stage 3: orders → lots (unified FIFO, all instruments)
├── chain_graph.py           # Stage 4: lots → chains (connected components)
├── strategy_engine.py       # Stage 5: position snapshots → strategy labels
│   ├── patterns.py          #   Pure pattern matching functions
│   ├── progressions.py      #   Wheel and other event-sequence strategies
│   └── constants.py         #   Strategy definitions and metadata
└── group_manager.py         # Stage 6: lot + chain + strategy → groups
```

The existing `src/models/` and `src/services/` code stays in place during development. The new `src/pipeline/` module is built alongside and can be run in shadow mode against the same raw_transactions to validate output parity.

## Migration / Validation Strategy

1. **Build** `src/pipeline/` alongside existing code. Both systems read from `raw_transactions` and produce lots, chains, and groups.
2. **Shadow run.** Process the full transaction history through both pipelines. Diff the outputs:
   * Do the same lots exist with the same quantities and entry prices?
   * Do the same chains have the same members?
   * Is realized P&L identical?
3. **Strategy improvement validation.** For chains currently labeled "2-Leg Strategy" or "Unknown", verify the new engine produces a better label.
4. **Cross-order test cases.** Manually verify that multi-order Iron Condors, Covered Calls, and Wheel progressions are correctly detected.
5. **Cut over.** Once outputs match (modulo intentional improvements), swap the pipeline and remove old code.

---

## Implementation Order

If we build this, the recommended sequence:

1. **Strategy Engine** (Stage 5) — can be built and tested in isolation with mock leg data. Highest user-visible impact.
2. **Position Ledger** (Stage 3) — unified FIFO, replacing lot_manager + position_inventory + equity pass. Core infrastructure.
3. **Chain Graph** (Stage 4) — depends on Stage 3's `order_id` / `closing_order_id` fields.
4. **Order Assembly** (Stage 2) — mostly same as today, add equity inclusion.
5. **Group Manager** (Stage 6) — depends on Stages 3-5.
6. **Orchestrator** — wires it all together, handles incremental reprocessing.

### What was built
