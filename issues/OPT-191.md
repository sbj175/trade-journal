# OPT-191: Research using SnapTrade

This would provide API integration with multiple brokers.

## Research Findings

### What is SnapTrade?

A unified brokerage API aggregator (by Passiv) providing one API to connect to 25+ brokerages for data access and trade execution.

**Tastytrade support:** Yes — OAuth authentication, trading + data access.

**Other notable brokers:** Schwab, Fidelity (read-only), Interactive Brokers (read-only), Robinhood (read-only), E\*TRADE, Webull, Alpaca, Coinbase, and more. Capabilities vary significantly by broker.

### Key Gaps for OptionLedger

1. **No real-time streaming** — no WebSocket support at all. SnapTrade recommends a third-party market data provider. Dealbreaker for the Positions page live quotes.
2. **No options Greeks** — the options chain endpoint returns strikes/expirations/symbols but not delta, gamma, theta, vega, bid/ask, volume, or open interest.
3. **Daily data sync** — transactions update daily by default (manual refresh available but not instant).
4. **Transaction timestamps may be imprecise** compared to what we get directly.
5. **$2/user/month cost** on pay-as-you-go (free tier limited to 5 connections).

### What We'd Gain

* Multi-broker support with one integration
* Hosted connection portal (handles all broker OAuth/MFA complexity)
* Normalized data model across brokers
* Python SDK available (`snaptrade-python-sdk` on PyPI)

### Recommendation

The direct Tastytrade API is significantly more capable for OptionLedger's current needs (real-time quotes, Greeks, detailed transaction data). SnapTrade becomes compelling only if multi-broker support is a priority. A **hybrid approach** — keeping direct Tastytrade integration while using SnapTrade for other brokers — would preserve our current capabilities while enabling expansion.

---

## Field-Level Data Mapping: SnapTrade vs RawTransaction

### Fields That Map Cleanly

Basic fields: `id`, `description`, `quantity`, `price`, `symbol` (though nested in SnapTrade).

### Fields That Need Transformation

| Our Field | SnapTrade Equivalent | Issue |
| -- | -- | -- |
| `commission` + `regulatory_fees` + `clearing_fees` | Single `fee` field | Lose fee granularity (3 fields → 1) |
| `action` (e.g. `OrderAction.BUY_TO_OPEN`) | `type` + `option_type` (split across 2 fields) | Need mapping layer; our codebase uses TT action strings throughout |
| `instrument_type` (`Equity Option`, `Equity`) | `symbol.type.code` (`cs`, `et`, etc.) | Different taxonomy entirely |
| `executed_at` (ISO datetime) | `trade_date` | Granularity varies by broker — some only provide date, not time. Breaks order grouping. |
| `value` / `net_value` | `amount` / computed `amount - fee` | No pre-computed net value |
| `transaction_type` / `transaction_sub_type` | `type` enum (BUY, SELL, OPTIONEXPIRATION, etc.) | Significant mapping work; SnapTrade folds sub-types into main type |
| `account_number` | `account.number` (nested) | May be masked by SnapTrade |

### Critical Gap: `order_id`

**This is the biggest concern.** Tastytrade provides an `order_id` that groups legs of multi-leg orders together. Our entire pipeline — `OrderProcessor`, strategy detection, chain building — depends on it.

SnapTrade has `external_reference_id` which "groups related transactions" but it is **nullable and brokerage-dependent**. Without reliable order grouping, we would need heuristic matching (same timestamp + same underlying + complementary actions), which is error-prone for complex strategies like Iron Condors.

### Fields SnapTrade Has That We Don't Store

* `currency` (ISO code) — useful for multi-currency
* `settlement_date` — distinct from trade date
* `option_symbol.strike_price` / `expiration_date` / `option_type` — structured fields (we parse from symbol string)
* `symbol.figi_code` — standardized instrument identifier

### Integration Effort Summary

Supporting SnapTrade would require:

1. A normalization layer between raw API data and our processing pipeline
2. Fallback heuristic order grouping (for brokers without reliable `order_id`)
3. Action/type mapping throughout `order_processor.py`, `strategy_detector.py`, `lot_manager.py`, `sync_service.py`
4. Instrument type taxonomy mapping

Not trivial, but doable — and it's the price of multi-broker support.

## SnapTrade Research Findings

### 1. What is SnapTrade?

SnapTrade is a unified brokerage API aggregator built by [Passiv](https://passiv.com/snaptrade/). It provides a single API that connects to 25+ retail brokerages, offering normalized data access (positions, balances, transactions) and trade execution. Think "Plaid for brokerage accounts" but with trading capabilities, not just read-only data.

**Key value prop:** One integration gives you connectivity to Robinhood, Schwab, Fidelity, Tastytrade, Interactive Brokers, E*TRADE, Webull, and more.

---

### 2. Supported Brokers

25+ integrations including:
- **Tastytrade** -- YES, supported with OAuth auth, trading + data access
- Robinhood (read-only)
- Charles Schwab
- E*TRADE (read-only)
- Fidelity (read-only, no trading)
- Interactive Brokers (read-only, no trading)
- Webull US (read-only)
- Alpaca
- Coinbase (OAuth, trading)
- Chase (OAuth, trading)
- Moomoo (OAuth, trading)
- Public (OAuth, trading)
- Questrade (read-only)
- Wealthsimple (read-only)
- Trading 212 (read-only)
- Binance, Kraken, BUX, Commsec, AJ Bell, Empower, and others

**Important:** Capabilities vary dramatically by broker. Many are read-only. Trading support (including options) is broker-specific.

---

### 3. API Capabilities

| Capability | Available? | Notes |
|---|---|---|
| **Transaction history** | Yes | Activities endpoint covers buys, sells, dividends, deposits, withdrawals. Updates daily. May lack precise timestamps. |
| **Current positions** | Yes | Separate endpoints for stock/ETF positions and option positions |
| **Account balances** | Yes | Cash holdings, buying power, multi-currency support |
| **Options chain data** | Partial | Returns strike prices, expiration dates, call/put symbol IDs, exercise type, multiplier |
| **Options Greeks** | **NO** | Not returned by the options chain endpoint |
| **Bid/ask, volume, OI** | **NO** | Not in the options chain response |
| **Real-time/streaming quotes** | **NO** | No WebSocket streaming. Some brokers return real-time prices, others delayed. SnapTrade recommends using a third-party market data provider. |
| **Options trading** | Yes (broker-dependent) | Single-leg and multi-leg strategies supported on brokers that allow it |
| **Order history** | Yes | All historical orders + recent orders (last 24 hours) |

**Data sync model:** Holdings sync immediately on connection, refresh once daily, with manual refresh available via API call. No real-time push/streaming.

---

### 4. Authentication Model

**For your app (developer):**
- API key pair: `clientId` + `consumerKey`

**For end users (connecting their brokerage):**
- SnapTrade provides a hosted "Connection Portal" (similar to Plaid Link)
- You generate a redirect URL per user, user opens it in browser
- Portal handles: brokerage selection, OAuth redirects, username/password auth, MFA verification
- User never shares credentials with your app
- Each user has a `userId` (you choose, immutable) + `userSecret` (generated, sensitive)
- Tastytrade specifically uses OAuth -- users log into Tastytrade directly

**This is actually very similar to what we already built for OPT-53 (Tastytrade OAuth flow), but SnapTrade handles all the broker-specific OAuth complexity.**

---

### 5. Pricing

| Plan | Cost | Limits |
|---|---|---|
| **Free** | $0 | 5 brokerage connections, all features including trading |
| **Pay-as-you-go** | $2/connected user/month | Unlimited connections, 250 req/min |
| **Enterprise** | Custom (starts ~$500/mo) | Volume discounts, higher rate limits, dedicated Slack support |

- A "connected user" = unique individual, not per-connection (user with 3 brokerages = 1 charge)
- No per-API-call costs
- Free tier is sufficient for development and small deployments

---

### 6. SDK Availability

| Language | Package | Status |
|---|---|---|
| **Python** | `snaptrade-python-sdk` (PyPI, v11.0.168) | Official, actively maintained |
| **TypeScript/Node.js** | `snaptrade-typescript-sdk` / `snaptrade-node-sdk` | Official |
| Java, Ruby, C#, PHP, Go | Available | Official SDKs |

All SDKs are auto-generated from OpenAPI spec. Documentation at [docs.snaptrade.com](https://docs.snaptrade.com/).

---

### 7. Limitations / Gotchas for Options Trading

1. **No Greeks data** -- options chain returns structure (strikes, expirations, symbols) but not delta, gamma, theta, vega. Would need a separate data source (like our current Black-Scholes engine on the Risk page).
2. **No real-time streaming** -- no WebSocket support. We currently use Tastytrade's WebSocket for live quotes on the Positions page. Would need a third-party market data provider.
3. **No bid/ask, volume, or open interest** in options chain data.
4. **Transaction timestamps may be imprecise** -- activities update daily, not real-time.
5. **Broker-dependent capabilities** -- what works with Tastytrade may not work with Schwab. Each integration has different read/write/trade support levels.
6. **Data refresh is daily by default** -- manual refresh available but not instant. Our current sync fetches everything on demand.
7. **Options positions returned via separate endpoint** -- normalized but may lose broker-specific fields.

---

### 8. Comparison: SnapTrade vs Direct Tastytrade API

| Aspect | Direct Tastytrade API | SnapTrade |
|---|---|---|
| **Broker coverage** | Tastytrade only | 25+ brokers |
| **Real-time quotes** | WebSocket streaming | No streaming, recommend 3rd party |
| **Options Greeks** | Available via SDK | Not available |
| **Transaction detail** | Full detail, real-time | Daily sync, may lack precise timestamps |
| **Options trading** | Full support (complex strategies) | Single + multi-leg, broker-dependent |
| **Auth complexity** | We manage OAuth ourselves | SnapTrade handles all broker OAuth |
| **Data normalization** | Tastytrade-specific format | Normalized across all brokers |
| **Cost** | Free (direct API) | $2/user/month (pay-as-you-go) |
| **Latency** | Direct, low latency | Extra hop through SnapTrade |
| **Account data** | Rich, broker-specific fields | Normalized, may lose some fields |

#### What we'd GAIN:
- Multi-broker support with one integration (biggest win)
- Hosted connection portal (no per-broker OAuth to maintain)
- Normalized data model across brokers
- Faster time to add new brokers

#### What we'd LOSE:
- Real-time WebSocket streaming (critical for Positions page)
- Options Greeks from the API
- Precise transaction timestamps
- Fine-grained Tastytrade-specific data
- Direct API access (adds latency + dependency on SnapTrade uptime)
- Free access (adds $2/user/month cost)

---

### Recommendation

SnapTrade is a strong fit **if multi-broker support is a priority** and the app evolves toward a SaaS product serving traders across multiple brokerages. However, for the current OptionLedger use case (Tastytrade-focused, real-time quotes, options Greeks, detailed transaction history), the direct Tastytrade API is significantly more capable.

**A hybrid approach could work:** Use SnapTrade for brokers we don't directly integrate with, while keeping the direct Tastytrade integration for its superior real-time data and options-specific features. This would require an abstraction layer in the backend to normalize data from both sources.

**Next steps if we decide to pursue:**
1. Sign up for free tier and test Tastytrade connection via SnapTrade
2. Compare transaction data quality (SnapTrade vs direct API) for the same account
3. Evaluate whether the Connection Portal UX is acceptable
4. Prototype the abstraction layer for multi-source data
