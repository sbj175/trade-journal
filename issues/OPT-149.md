# OPT-149: Tags and position notes disappear after sync

## Bug

Tags and position notes (comments) on the Positions page are cleared/disappear after clicking Sync. They reappear after a page refresh, suggesting the data is still in the DB but the in-memory state is being reset during the sync flow.

## Likely Cause

`fetchPositions(includeSync=true)` rebuilds `allItems` from scratch using the `/api/open-chains` response. The chain objects returned by the API may not include `tags` or the `group_id` needed to look up notes — so after the array is replaced, the UI loses the association.

Specifically:

1. **Tags**: `allItems` is rebuilt from raw chain data which likely doesn't include the `tags` array. The tags are only fetched as part of the ledger/group API, not the open-chains endpoint.
2. **Notes**: `positionComments` ref persists, but if `group_id` values change or aren't present in the rebuilt items, `getPositionComment()` returns empty.

## Steps to Reproduce

1. Open the Positions page
2. Add a tag or note to any position
3. Click Sync
4. Observe: tags and notes disappear
5. Refresh the page — they reappear

## Expected Behavior

Tags and notes should persist across syncs.

## Comments

### 2026-02-28 — Steve Johnson

**Fix merged to main** (`b3e9f6c`)

### Root Causes Fixed

1. **`save_raw_transactions` over-counted** — incremented `saved_count` for every transaction including duplicates, so `raw_saved > 0` was always true and `reprocess()` ran on every sync even with zero new data. Fix: only count rows where `result.rowcount > 0`.

2. **`group_id` was a random UUID** — `clear_all_lots()` deleted PositionGroup rows, then GroupPersister recreated them with new UUIDs. Tags (keyed by `group_id`) and notes (keyed by `group_${group_id}`) were orphaned.

### Changes

- **`src/database/db_manager.py`**: Only count actually-inserted rows in `save_raw_transactions`
- **`src/pipeline/group_manager.py`**: Use `source_chain_id` as deterministic `group_id`; match existing groups by chain_id; auto-migrate old UUID → chain_id (cascading to PositionGroupTag, PositionNote, PositionGroupLot); add PositionGroupTag to orphan cleanup
- **`src/services/ledger_service.py`**: Use `chain_id` as `group_id` in `seed_position_groups`
- **`src/models/lot_manager.py`**: Stop deleting PositionGroup rows in `clear_all_lots()` (GroupPersister handles orphan cleanup)
- **`frontend/src/pages/positions/App.vue`**: Re-fetch notes from DB after sync

### Auto-Migration

Existing groups with old UUID group_ids will be automatically migrated to deterministic chain_id-based group_ids on first sync. No manual migration needed.
