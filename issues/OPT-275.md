# OPT-275: Multi-fill same-order opens split into separate groups when later closed

## Bug

A roll whose new opening leg filled in two pieces (different prices preventing assembler aggregation, e.g. 24 contracts at $1.02 + 8 contracts at $1.01) split into two separate position_groups instead of being treated as one logical position.

User-visible: 32x IBIT 39 calls Mar 6 → Mar 13 showing as a 24-lot group and an 8-lot group.

## Root cause

`group_manager.py` Rule 0 (chain-aware routing) requires `_is_group_open(gk)` to merge a new lot into an existing group on chain match. `_is_group_open` checks whether any lot in the group has `remaining_quantity != 0`. The routing pass reads the **final** state of every lot — so when both Mar 6 lots were eventually closed by the next roll, the first lot's freshly-created group looked "closed" the moment we tried to merge the second lot.

## Fix

Rule 0 also passes when the candidate group already contains a lot with the same `opening_order_id` as the new lot. Same-order + same-chain is the structural signal for "this is a multi-fill of one logical broker action" — and it's safe with respect to parallel-ladder rolls because [OPT-273](https://linear.app/optionedge/issue/OPT-273/same-direction-multi-leg-roll-collapses-parallel-chains-in-position) ensures parallel rungs get distinct chain_ids, so Rule 0 wouldn't fire for them in the first place.

## Test

Added `test_multi_fill_same_order_roll_stays_in_one_group` in `TestRollMechanics`. Asserts both pieces of a multi-fill new opening leg share one group.

## Impact

Reprocess group count dropped from \~860 to \~490 across the user's full dataset, with the 370 missing groups being phantom multi-fill splits like this one.
