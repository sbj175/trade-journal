# OPT-107: PostgreSQL Migration (Local Docker)

Add PostgreSQL as an alternative database backend via Docker while keeping SQLite as the default. The app stays local (not containerized) — only PostgreSQL runs in Docker.

## Approach

* **Dual dialect**: Both SQLite and PostgreSQL supported. `DATABASE_URL` env var selects the backend. No `DATABASE_URL` = SQLite default.
* **Docker scope**: Only PostgreSQL in Docker. App runs natively via `start.sh`.
* **Tests**: Continue using SQLite for speed.

## Phases

1. Engine and Configuration — dialect-aware `init_engine()`
2. Dialect-Agnostic Upserts — replace 22 `sqlite_insert` call sites with `dialect_insert`
3. Remove Legacy sqlite3 Code — delete `_add_transaction_columns()`, `get_connection()`, etc.
4. Alembic Dual-Dialect Support — conditional `render_as_batch` and URL from env
5. Docker Compose + Dependencies — PostgreSQL 16 Alpine + psycopg2-binary
6. Data Migration Script — `scripts/migrate_sqlite_to_pg.py`
7. Documentation — Update [CLAUDE.md](<http://CLAUDE.md>)

## Benefits of PostgreSQL Migration

### Immediate Benefits
- **Production-ready database**: PostgreSQL provides ACID compliance with proper FK enforcement, connection pooling (`pool_size=5, max_overflow=10`), and concurrent access — no more "database locked" errors under load
- **Data integrity**: PostgreSQL strictly enforces foreign key constraints, surfacing orphaned references that SQLite silently allowed (e.g., system-generated expiration orders). This catches data quality issues early
- **Zero-downtime switchback**: `DATABASE_URL` env var controls which backend is used. Comment it out → back on SQLite instantly. No code changes needed

### Legacy Code Removal
- **Deleted 190 lines** of raw `PRAGMA table_info()` / `ALTER TABLE` migration code (`_add_transaction_columns`)
- **Removed all `import sqlite3`** and `get_connection()` from the codebase — everything goes through SQLAlchemy ORM now
- **Eliminated the conftest.py hack** that stubbed `_add_transaction_columns` to work around SQLite locking during tests
- **Replaced 22 `sqlite_insert` call sites** with a single `dialect_insert()` that resolves the correct dialect at engine init time

### SaaS Readiness (OPT-51)
- **Dialect-agnostic data layer**: Every database operation works identically on SQLite and PostgreSQL — no dialect-specific code in business logic
- **Alembic dual-dialect**: Schema migrations work on both backends (`render_as_batch` conditional for SQLite, standard for PG)
- **Docker Compose ready**: One command (`docker compose up -d`) spins up PostgreSQL for development or deployment
- **Data migration path**: `scripts/migrate_sqlite_to_pg.py` provides a tested bulk-transfer tool for moving existing user data to PostgreSQL

### Developer Experience
- **Simpler initialization**: `DatabaseManager(db_url=...)` replaces the old `db_path` parameter; `initialize_database()` is now pure ORM (`Base.metadata.create_all`)
- **Tests remain fast**: All 71 tests continue running against SQLite (37s) — no Docker dependency for the test suite
- **Net code reduction**: 423 lines added, 466 removed = -43 lines net despite adding PostgreSQL support, Docker Compose, and a migration script
