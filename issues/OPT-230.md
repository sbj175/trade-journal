# OPT-230: Calculate and store Realized Volatility

This provides the data to compare IV to RV.

Add a derived-metrics table:

symbol_volatility_metrics

\-------------------------

symbol

date

rv10

rv20

rv30

---

The RV formula

For each trade date:

log_return = ln(P_today / P_yesterday)

Then:

RV = stdev(last_N_log_returns) × √252

Example windows:

RV10

RV20

RV30

Store them.

### 2026-03-23 — Steve Johnson

The core efficiency insight

Never compute RV on demand per trade.

Instead:

Store prices once

Compute returns once

Update RV incrementally

Lookup instantly later

That turns an expensive operation into a trivial one.

\---

Step 1 — Only store daily closes (not full candles)

For RV:

You only need daily close prices

So your table should look like:

price_history

\-------------

symbol

date

close

Example:

SPY | 2026-03-18 | 507.21

SPY | 2026-03-19 | 509.33

SPY | 2026-03-20 | 505.88

Storage footprint stays tiny even for thousands of symbols.

\---

Step 2 — Precompute log returns once

Add another table:

symbol_returns

\--------------

symbol

date

log_return

Compute nightly:

log_return = ln(close_today / close_yesterday)

Now RV becomes fast because:

RV = stdev(last N returns)

instead of recomputing returns repeatedly.

\---

Step 3 — Use rolling windows (the real speed trick)

Instead of recalculating standard deviation from scratch each time:

Maintain rolling statistics.

Example structure:

symbol_volatility_metrics

\-------------------------

symbol

date

rv10

rv20

rv30

Each night:

drop oldest return

add newest return

update stdev

That’s O(1) compute per update.

Extremely cheap.

\---

Step 4 — Only compute metrics for symbols users actually trade

This is the biggest optimization most people miss.

Don’t do:

compute RV for entire market

Do:

compute RV only for tracked symbols

Example:

If OptionLedger users trade:

SPY

QQQ

TSLA

NVDA

AAPL

Only maintain those.

\---

Step 5 — Cache symbol volatility metrics

Once computed:

SPY

2026-03-22

rv10 = 18.2

rv20 = 21.0

rv30 = 24.4

Store permanently.

Then when a trade syncs:

lookup(symbol, trade_date)

instead of recomputing anything.

This makes trade ingestion extremely fast.

\---

Step 6 — Efficient nightly update job (recommended architecture)

Run once per day after market close:

for symbol in tracked_symbols:

    fetch latest close

    compute log_return

    update rolling RV windows

    store results

Runtime:

Usually under seconds.

Even at large scale.

\---

Step 7 — Efficient historical backfill (important for your existing trades)

When a new symbol appears:

fetch last 60–90 trading days

compute returns once

compute RV history once

store results

done forever

Never repeat again.

Instead of:

Step 9 — Even better: compute RV inside Postgres directly (Supabase-friendly)

Since you're already using Postgres via Supabase, you can compute rolling RV like this:

stddev(log_return)

OVER (

  PARTITION BY symbol

  ORDER BY date

  ROWS BETWEEN 19 PRECEDING AND CURRENT ROW

)

This creates:

RV20 automatically

Database-native.

Fast.

Elegant.

No backend compute required.
