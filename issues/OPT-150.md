# OPT-150: JWT token logged in plaintext in uvicorn access log

## Bug

WebSocket upgrade lines in uvicorn's access log expose the full JWT token:

```
INFO: 172.25.0.1:60284 - "WebSocket /ws/quotes?token=eyJhbG..." [accepted]
```

This is a security concern — tokens should never appear in logs.

## Root Cause

The frontend passes the JWT as a query parameter (`?token=...`) for WebSocket auth. Uvicorn logs the full request path including query string in its access log.

## Approaches Tried (all failed)

1. `logging.Filter` on `uvicorn.access` logger at module level — Filter gets added but uvicorn reconfigures logging when it spawns the worker process (with `reload=True`), wiping the filter.
2. `logging.Filter` added in `@app.on_event("startup")` — Same result. By the time the startup event fires, uvicorn's access logger is already configured and the WebSocket accept line has already been logged.
3. **Modifying** `record.msg` in the filter — Uvicorn puts the URL path in `record.args` (a tuple), not `record.msg`. Fixed to check both, but filter still wasn't active (see #1/#2).
4. **Custom** `log_config` dict passed to `uvicorn.run()` — Works for `__main__` but Docker entrypoint runs uvicorn directly (`exec uvicorn app:app`), bypassing `__main__`.
5. `logging.config.dictConfig()` at module import time — Applied when `app.py` is imported. Still didn't take effect, likely because uvicorn applies its own `log_config` *after* importing the app module, overriding our dictConfig.

## Current State

Code in `app.py` has `AccessLogFilter` class, `_UVICORN_LOG_CONFIG` dict, and `dictConfig()` call — none of which work. All should be removed before trying a new approach.

## Possible Next Steps

* **Pass token via WebSocket subprotocol** instead of query param (eliminates the problem at the source)
* **Custom uvicorn log config file** passed via `--log-config` in `docker-entrypoint.sh`
* **ASGI middleware** that strips the token from `scope["query_string"]` before the request reaches uvicorn's logging layer (unclear if access log fires before or after ASGI)
* **Suppress WebSocket lines entirely** from access log if redaction proves too difficult
