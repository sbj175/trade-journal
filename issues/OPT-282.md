# OPT-282: OPT-282: roll_chain_summary loses branch heads (one summary per leaf, not per root)

## Symptom

User report (2026-04-27): IBIT covered calls show all rolls attributed to the 43-strike position; the 41-strike position shows zero rolls.

## Root cause

`src/pipeline/roll_chain_summary.py::populate_roll_chain_summaries` does a BFS from each root and uses `chain[-1]` as the head — i.e., one summary per root, picking whichever leaf the BFS visits last. When the `rolled_from_group_id` graph branches (spec §5.2), only one leaf gets a summary row; siblings disappear from the rollup table.

In the IBIT case, root `4079b434` (acct 8644) and root `12b569b4` (acct 6959) each branched on 2026-04-10 — the 39C/4-10 closing rolled into BOTH a 42C/4-17 (today's 43-strike branch) AND a 40.5C/4-24 (today's 41-strike branch). The 43-strike branch is one node deeper, so BFS visits it last; only the 43-strike head gets a summary row. The 41-strike chain is intact in `position_groups` but invisible in the rollup.

The position-graph data itself is correct — `position_groups.rolled_from_group_id` already encodes the tree structure properly. The bug is purely in the summary materializer.

## Fix

Enumerate one summary row per **leaf** (terminal node — a group with no children in the tree, OPEN or CLOSED), not one per root. The `chain` for each summary is the unique root-to-leaf path. Shared upstream history appears in both summaries — that's correct per spec §5.2 since each leaf is an independent continuation of the same source.

## Test

Add a unit test in `tests/unit/test_roll_chain_summary.py` (creating it if needed): one root, two children at the same generation, one further-rolled grandchild on one branch only. Expect 2 summary rows (one per leaf), both pointing at the same root.

## Production data

After landing, the user should hit `/api/reprocess` (or just the rollup stage) to repopulate `roll_chain_summaries`. The 41-strike chains will then surface in the UI.
