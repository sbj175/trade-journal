# OPT-119: Research: Futures transaction handling on import

Research how futures and futures options transactions from Tastytrade would interact with our import/processing pipeline. Determine whether we need to filter them out during sync or if they'll cause issues with strategy detection, P&L calculation, or position tracking.

## Comments

### 2026-02-25 — Steve Johnson

## Research Findings: Futures & Futures Options in Tastytrade API/SDK

### 1. InstrumentType Enum (tastytrade SDK v12)

The `InstrumentType` enum in `tastytrade/order.py` defines all instrument types:

```python
class InstrumentType(str, Enum):
    BOND = "Bond"
    CRYPTOCURRENCY = "Cryptocurrency"
    CURRENCY_PAIR = "Currency Pair"
    EQUITY = "Equity"
    EQUITY_OFFERING = "Equity Offering"
    EQUITY_OPTION = "Equity Option"
    FIXED_INCOME = "Fixed Income Security"
    FUTURE = "Future"
    FUTURE_OPTION = "Future Option"
    INDEX = "Index"
    LIQUIDITY_POOL = "Liquidity Pool"
    UNKNOWN = "Unknown"
    WARRANT = "Warrant"
```

The two futures-related types are **`Future`** and **`Future Option`**. These come through as `instrument_type` on `Transaction` and `CurrentPosition` objects.

---

### 2. OrderAction Differences (Critical for Position Tracking)

The `OrderAction` enum has special values for futures:

```python
class OrderAction(str, Enum):
    BUY_TO_OPEN = "Buy to Open"
    BUY_TO_CLOSE = "Buy to Close"
    SELL_TO_OPEN = "Sell to Open"
    SELL_TO_CLOSE = "Sell to Close"
    BUY = "Buy"       # for futures only
    SELL = "Sell"      # for futures only
```

**Equity/equity options** use `Buy to Open`, `Sell to Close`, etc.
**Futures** use bare `Buy` and `Sell` -- these do NOT indicate whether a position is being opened or closed. The SDK comments explicitly state these are "for futures only."

**Impact on OptionLedger:** Our order processor currently maps actions like this:
- `opening_actions = {'BTO', 'STO', 'BUY', ...}`
- `closing_actions = {'BTC', 'STC', 'SELL', ...}`

This incorrectly treats ALL `Buy` actions as opening and ALL `Sell` actions as closing. For futures, `Buy` could be opening a long OR closing a short, and `Sell` could be opening a short OR closing a long. There's no way to distinguish without tracking net position state.

**Note:** Futures OPTIONS (instrument type `Future Option`) DO use `Buy to Open`/`Sell to Close` etc., just like equity options. The bare `Buy`/`Sell` actions are only for futures contracts themselves.

---

### 3. Symbol Format Differences

#### Equity Symbols
- Stocks: `AAPL`, `SPY`, `MSFT`
- Options: `AAPL  241220C00150000` (OCC format: 6-char padded symbol + YYMMDD + C/P + 8-digit strike)

#### Futures Symbols
- Format: `/{product_code}{month_letter}{year_digit}`
- Examples: `/ESZ5` (E-mini S&P 500, December 2025), `/NQH6` (E-mini Nasdaq, March 2026)
- Month codes: F=Jan, G=Feb, H=Mar, J=Apr, K=May, M=Jun, N=Jul, Q=Aug, U=Sep, V=Oct, X=Nov, Z=Dec
- The leading `/` is the key identifier for futures symbols
- In API URLs, `/` must be encoded as `%2F`

#### Futures Options Symbols
- Format: `./{underlying_future} {option_root} {YYMMDD}{C/P}{strike}`
- Example: `./ESU3 E1DQ3 230803P3860`
- Starts with `./` prefix
- Contains spaces separating the parts
- In API URLs, needs encoding: `.%2FESU3%20E1DQ3%20230803P3860`

**Impact:** Our option symbol parser in `create_position_from_transaction()` assumes OCC format (`symbol.split()` with `parts[1]` being the option details). Futures options have a completely different 3-part format that would break this parsing.

### 4. Contract Multipliers (NOT Always 100)

Our code currently hardcodes: `multiplier = 100 if 'OPTION' in instrument_type else 1`

This is wrong for futures. Each futures product has its own `notional_multiplier`:

| Product | Symbol | Multiplier | Tick Size | Tick Value | Settlement |
|---------|--------|-----------|-----------|------------|------------|
| E-mini S&P 500 | /ES | $50 | 0.25 | $12.50 | Cash |
| Micro E-mini S&P | /MES | $5 | 0.25 | $1.25 | Cash |
| E-mini Nasdaq 100 | /NQ | $20 | 0.25 | $5.00 | Cash |
| Micro E-mini Nasdaq | /MNQ | $2 | 0.25 | $0.50 | Cash |
| E-mini Dow | /YM | $5 | 1 | $5.00 | Cash |
| E-mini Russell 2000 | /RTY | $50 | 0.1 | $5.00 | Cash |
| Crude Oil | /CL | $1,000 | 0.01 | $10.00 | Physical |
| Natural Gas | /NG | $10,000 | 0.001 | $10.00 | Physical |
| Gold | /GC | $100 | 0.10 | $10.00 | Physical |
| Silver | /SI | $5,000 | 0.005 | $25.00 | Physical |
| Copper | /HG | $25,000 | 0.0005 | $12.50 | Physical |
| 30-Year T-Bond | /ZB | $1,000 | 1/32 | $31.25 | Physical |
| 10-Year Note | /ZN | $1,000 | 1/64 | $15.625 | Physical |
| VIX | /VX | $1,000 | 0.05 | $50.00 | Cash |
| Euro FX | /6E | $125,000 | 0.00005 | $6.25 | Physical |

**The multiplier is available from the API** via `FutureProduct.notional_multiplier` and also from `CurrentPosition.multiplier` for open positions. The `Transaction` object does NOT include a multiplier field -- it would need to be looked up from the `FutureProduct` or `Future` instrument data.

For **futures options**, the `FutureOption` class has a `multiplier: Decimal` field that is specific to each contract.

### 5. SDK Classes for Futures Data

The SDK provides these key classes in `tastytrade/instruments.py`:

- **`Future`**: Tradeable futures contract with `notional_multiplier`, `tick_size`, `display_factor`, `expiration_date`, `product_code`, `exchange`
- **`FutureProduct`**: Non-tradeable product family info with `notional_multiplier`, `cash_settled`, `listed_months`, `active_months`, `roll` info
- **`FutureOption`**: Tradeable futures option with `multiplier`, `strike_price`, `option_type`, `settlement_type`, `underlying_symbol`, `strike_factor`, `display_factor`
- **`FutureOptionProduct`**: Product family info for futures options with `cash_settled`, `clearing_price_multiplier`, `settlement_delay_days`

### 6. Transaction Object Structure

The `Transaction` dataclass in `tastytrade/account.py` is shared for ALL instrument types. Key fields:

```python
class Transaction(TastytradeData):
    id: int
    account_number: str
    transaction_type: str        # "Trade", "Receive Deliver", etc.
    transaction_sub_type: str    # Sub-classification
    symbol: str | None           # "/ESZ5" for futures, "./ESZ5 ..." for fut opts
    instrument_type: InstrumentType | None  # "Future" or "Future Option"
    underlying_symbol: str | None  # "/ES" for futures
    action: OrderAction | None   # "Buy"/"Sell" for futures, "Buy to Open" etc for fut opts
    quantity: Decimal | None
    price: Decimal | None
    value: Decimal | None
    commission: Decimal | None
    order_id: int | None
    # ... other fields
```

**No multiplier field on Transaction.** The `value` field includes the total dollar value of the transaction (already multiplied), but `price` is per-unit before multiplier.

**The `multiplier` field IS available** on position objects, which means positions will have accurate multiplier info.

### 8. Market Data Differences

The `get_market_data_by_type()` function requires separating futures and futures options into their own parameter lists:

```python
get_market_data_by_type(
    session,
    equities=['AAPL'],            # Equity symbols
    options=['AAPL  250131C00200000'],  # Equity options
    futures=['/ESZ5'],            # Futures (separate param!)
    future_options=['./ESZ5 ...'], # Futures options (separate param!)
)
```

Our WebSocket quote system would need to classify symbols by instrument type before requesting quotes.

### 9. Settlement and Expiration Differences

- **Equity options**: Expire on Fridays (standard) or throughout the week (0DTE/weeklies). American-style exercise. Physical settlement (shares delivered).
- **Futures**: Expire on product-specific dates. Can be cash-settled (index futures like /ES) or physically-settled (commodities like /CL).
- **Futures options**: Expire into futures positions (not shares). Can be cash-settled or physically-settled depending on the product. Some have weekly expirations.
- **First notice date**: For physically-settled futures, the `first_notice_date` is when the holder may be required to take delivery. This doesn't apply to cash-settled contracts.

### 10. Areas of OptionLedger That Would Break

1. **`order_models.py:1017`** -- `multiplier = 100 if 'OPTION' in instrument_type else 1` -- wrong for both futures (varying multipliers) and futures options (not always 100)
2. **`order_models.py:946`** -- Option symbol parsing assumes OCC format; futures option symbols have completely different format
3. **`order_models.py:836-843`** -- Order type detection treats `Buy`/`Sell` as opening/closing; for futures these are ambiguous
4. **`tastytrade_client.py:200`** -- Position multiplier calculation hardcodes 100 for options, 1 for non-options
5. **`tastytrade_client.py:213-261`** -- Cost basis and market value calculations assume either stock (multiplier 1) or option (multiplier 100)
6. **Strategy detection** -- `strategy_detector.py` and `order_processor.py` have no futures-aware logic
7. **WebSocket quotes** -- Would need to route futures symbols to the correct market data endpoint
8. **P&L calculations** -- All multiplier assumptions are equity-centric

### Recommended Next Steps

Given the scope of changes needed, the simplest short-term approach would be to **filter out futures and futures options during sync** by checking `instrument_type` against `"Future"` and `"Future Option"`. This prevents corrupted data while full futures support is designed.

For full support, the major work items would be:
1. Multiplier lookup system (cache `FutureProduct.notional_multiplier` values)
2. Futures symbol parser (separate from OCC parser)
3. Position-state-aware action classification for bare `Buy`/`Sell`
4. Market data routing by instrument type
5. Updated P&L calculations using actual multipliers

## Research Findings

### The short answer: futures are NOT like equities/equity options — they'd need significant work to support properly. For now, we should filter them out on import.

### Key Differences from Equities

**1. Multipliers are product-specific (not uniform)**
- Equity options: always 100x
- `/ES` (S&P E-mini): 50x
- `/NQ` (Nasdaq E-mini): 20x
- `/CL` (Crude Oil): 1,000x
- `/NG` (Natural Gas): 10,000x
- Every futures product has its own multiplier — there's no universal constant

**2. Symbol format is completely different**
- Equity option: `AAPL  241220C00150000` (OCC format)
- Future: `/ESZ5` (slash + product + month code + year)
- Future option: `./ESH5 EW4G5 250228C5950` (three-part format)
- Our OCC symbol parser would fail on these

**3. Buy/Sell actions are ambiguous for futures**
- Equity options use `Buy to Open`, `Sell to Close`, etc. (intent is explicit)
- Futures use bare `Buy` and `Sell` — you'd need position state tracking to know if it's opening or closing
- Futures OPTIONS do use the full `Buy to Open`/`Sell to Close` actions (same as equity options)

**4. Quote routing requires instrument type classification**
- Tastytrade API requires symbols separated by type: `equities=`, `options=`, `futures=`, `future_options=`
- Our WebSocket quote system only classifies symbols as "equity" or "option"

### Current State in Our Codebase

**Good news:** The OrderProcessor already filters out non-equity transactions (line 260 of `order_processor.py`), so futures data from a Tastytrade account with futures trades would be silently dropped during chain processing. No crashes, but no data either.

**Bad news:** Several other paths don't have this filter:
- `get_positions()` would fetch futures positions and apply wrong multipliers
- `_classify_symbols()` would misroute futures symbols for quotes
- Sync reconciliation has explicit `EQUITY`/`EQUITY_OPTION` filters but `get_positions()` doesn't

**Hardcoded `* 100` multiplier in 30+ locations** across the codebase (P&L calculator, lot manager, chain service, report service, frontend JS). All would produce wrong results for futures.

### Recommendation

**Immediate action:** Add an explicit instrument type filter at the transaction fetch layer (`get_transactions()` and `get_positions()` in `tastytrade_client.py`) to skip `Future` and `Future Option` instrument types. This is safer than relying on downstream filters.

**Future support (if ever):** Would require:
1. `multiplier` column on Position/PositionLot tables
2. Futures symbol parser
3. Replace 30+ hardcoded `* 100` sites with dynamic multiplier lookups
4. Extend quote classification for futures symbols
5. Position-state-aware action classification for bare Buy/Sell
