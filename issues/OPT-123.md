# OPT-123: Cross-order Strategy Engine: position-snapshot recognition

## Summary

Wire the Strategy Engine (`src/pipeline/strategy_engine/`) into the pipeline so it runs on every position change and updates group strategy labels dynamically.

The engine itself is built (`recognizer.py`, pattern modules, `constants.py`, `types.py`, `adapters.py`) and is called from `group_manager.py`. But several planned features from the OPT-121 design were not implemented:

## Remaining work

### 1\. Strategy history on groups

The OPT-121 design called for a `strategy_history` JSON column on `position_groups` to track how the strategy label evolves over time:

```json
[
  {"strategy": "Bull Put Spread", "from_date": "2025-01-15", "to_date": "2025-01-20"},
  {"strategy": "Iron Condor", "from_date": "2025-01-20", "to_date": "2025-03-21"}
]
```

Currently only `strategy_label` (single text field) exists. This history is needed for the Reports page to attribute P&L to the correct strategy at the time it was realized.

### 2\. User override flag

Add `user_override` boolean column to `position_groups`. When a user manually sets a strategy label on the Ledger page, flag it so the Strategy Engine doesn't overwrite it on reprocessing.

### 3\. Wheel progression detection

The Wheel is an event sequence (Short Put → Assignment → Covered Call → Called Away → repeat), not a position pattern. Detect it by examining chain history:

* If chain has Short Put → Assignment → Short Call on assigned stock → label as "Wheel" with phase indicator
* Requires `strategy_engine/progressions.py` module

## Database changes

* Add `position_groups.strategy_history` (JSON)
* Add `position_groups.user_override` (boolean, default false)
* Alembic migration required

## Depends on

Spun out of [OPT-121](https://linear.app/optionedge/issue/OPT-121).

## Verification Complete: Cross-Order Strategy Detection Works

Verified end-to-end that the strategy engine correctly detects cross-order strategy evolution through the pipeline.

### Integration Test (`tests/integration/test_cross_order_strategy.py`)

5 test scenarios, all passing:

1. **Bull Put Spread** — single order (short $50P + long $45P) → labeled "Bull Put Spread"
2. **Iron Condor** — add Bear Call Spread (short $55C + long $60C, same expiration) → group merges via Rule 2 (same account/underlying/expiration) → labeled "Iron Condor"
3. **Back to spread** — close the call wing → label updates to "Bull Put Spread" (strategy engine only sees open lots)
4. **Lot count progression** — verifies 2 → 4 → 4 (2 open + 2 closed) lot counts through the evolution
5. **Fully closed** — when all legs closed from scratch, groups split into per-wing groups (Bull Put Spread + Bear Call Spread) because the group manager won't merge into already-closed groups. This is expected behavior; strategy_history (future work) will preserve the Iron Condor label.

### Demo Script (`scripts/demo_cross_order.py`)

Interactive script for visual verification on the Ledger page:
```
python scripts/demo_cross_order.py inject     # → Bull Put Spread
python scripts/demo_cross_order.py add-wing   # → Iron Condor
python scripts/demo_cross_order.py cleanup    # → remove ZTEST data
```

### Key Finding

The cross-order merging works via **Rule 2** in `assign_lots_to_groups()`: lots with the same `(account, underlying, expiration)` merge into existing open groups. This means a Bull Put Spread and Bear Call Spread opened as separate orders but sharing the same expiration automatically combine into an Iron Condor group.
