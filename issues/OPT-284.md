# OPT-284: OPT-284: Migrate roll lineage from group-level to lot-level (epic)

## Why

Today's roll lineage lives at the group level via `position_groups.rolled_from_group_id`. That data structure permits **branching** — one source group can have multiple children — which makes the math non-additive: a parent's `realized_pnl` ends up counted in every child chain's cumulative total, so summing chain totals double-counts the trunk. Strict additivity (`total portfolio P&L = sum of chain P&Ls`) is incompatible with branching at whatever level the chain is defined.

The structural fix is to move chain identity to the **lot** level, where every new lot has at most one parent lot by construction. Lot chains are strictly linear; chain math is trivially additive; position-level lineage becomes a *derived view* of the lot graph. Multiple subtle bugs we've been chasing dissolve as not-applicable:

* **IBIT 42C** (the immediate motivator): a "roll + add" ticket attached the add-3 to the closed-5's chain. Under lot-level lineage, the add lots have no parent lot → fresh chain, automatically.
* [OPT-282](https://linear.app/optionedge/issue/OPT-282/opt-282-roll-chain-summary-loses-branch-heads-one-summary-per-leaf-not) **per-leaf summary workaround**: branching tree → multiple leaves per root → workaround. Under lot-level lineage, branching is impossible → workaround irrelevant.
* **Partition-at-branch question** (FIFO vs proportional vs weighted): only matters because group chains can branch. Under lot-level lineage, the partition is structural (each lot has one parent) and only surfaces when the trader's intent is genuinely ambiguous (e.g., 5→3+2 same-shape split — addressed by manual link/split, spec §7).

User-facing surface (Ledger card, chain modal, "rolls" counter on a group) stays unchanged in the common case where all legs of a position roll together — the position-level lineage is a clean derivation of synchronized lot lineage.

## Phases

Each phase is independently shippable and testable.

### Phase 1 — schema + backfill (additive, reversible)

* Add `position_lots.parent_lot_id` (nullable FK to `position_lots.id`).
* Backfill from existing `rolled_from_group_id` where unambiguous: when a target group has exactly one source group, pair lots by (option_type, direction, quantity) and set `parent_lot_id`. Leave NULL where ambiguous.
* No detection logic changes yet. `rolled_from_group_id` remains authoritative.
* **Acceptance**: schema migration applies cleanly, backfill is non-destructive, suite green.

### Phase 2 — detection moves to lot level (semantics change)

* New pipeline stage (likely between current Stage 4 and Stage 6): pair same-day compatible closes and opens, set `parent_lot_id`. Quantity-aware (spec §0.2). Source: spec §1 + §6 reframed lot-centric.
* Stage 3 (`roll_splitter`) stops bookkeeping `chain_id`. (May still split orders, but no chain propagation.)
* Stage 6 (`group_manager`) drops `_detect_roll_links` entirely; just routes lots into groups by `(account, underlying, expiration)`.
* `position_groups.rolled_from_group_id` becomes a *derived* view: a group has a single rolled-from group iff every lot in the group has a parent_lot_id pointing into the same source group AND that source group is fully closed. Otherwise NULL — UI may surface partial/multi-source cases as adjustments rather than rolls.
* **Acceptance**: IBIT 42C case correctly classified as a fresh chain; existing roll detection tests still pass after rewriting them to the new model; user verifies a sampling of real positions on production.

### Phase 3 — summaries + cleanup

* `roll_chain_summaries` reschemas to `(root_lot_id, current_lot_id)`. Or, depending on UI needs, derive position-chain summaries on demand from lot lineage instead of materializing.
* Drop the [OPT-282](https://linear.app/optionedge/issue/OPT-282/opt-282-roll-chain-summary-loses-branch-heads-one-summary-per-leaf-not) per-leaf workaround in `populate_roll_chain_summaries` (no longer needed — lot chains don't branch).
* Drop `position_groups.rolled_from_group_id` storage if Phase 2 made it derived. Or keep it as a read-only cache, refreshed during the rollup stage.
* Spec doc rewrite: `docs/roll-detection-spec.md` reframed lot-centric. §1 becomes lot-pairing, §5 (multi-roll dynamics) collapses (branching impossible), §6 (active-at-close filter) goes away.

### Phase 4 — UI verification

* Walk through Ledger card / chain modal / rolls counter / Reports tab for sampled positions:
  * All-leg synchronized roll (common case): UI unchanged.
  * Roll + add (IBIT case): rolled position has correct chain length; add appears as a new position with no chain.
  * Partial-leg roll (puts only): position stays in same group; chain doesn't advance.
  * Manual link/split (spec §7): operates on lot pairs now; UX revisit.
* Decide whether the "rolls" badge on a group shows max-leg-depth, per-leg, or count-of-management-events. (Open question — defer until Phase 4.)

## Out of scope (open questions to revisit)

* Whether to expose any lot-level chain UX directly. Default answer: no, lot lineage is plumbing only.
* "Opportunities for further improvement" the user flagged during the design discussion — leave the door open but don't pre-scope.
* Manual link/split (spec §7) granularity under the new model.

### 2026-04-28 — Steve Johnson

**Phase 3a shipped on `main` (commit `7e77c1e`).**

`populate_roll_chain_summaries` now computes cumulative metrics by walking lot-level lineage instead of summing across groups in the chain. The math invariant — *total portfolio realized P&L = sum of chain `cumulative_realized_pnl` rows* — is now structural.

What changed
- For each summary row, walk `parent_lot_id` back from the leaf-group's lots to enumerate the lots in this chain's lineage.
- Sum `realized_pnl` across just those lots. Each lot has at most one parent (spec §0.2), so each lot's P&L belongs to exactly one chain.
- `cumulative_premium` / `net_premium` similarly: ancestor lots' realized P&L + leaf-group lots' open premium.

The behavior changes only at branch points where group-level summing previously over-credited the shared trunk. For the common all-legs-sync-rolled case the lot walk reaches every lot in every group in the chain, so cumulative figures are unchanged.

**Tests added** (`TestAdditivityAcrossSiblingChains`, +2):
- 5-into-3+2 partition test: B credits 300 (its 3 ancestors), C credits 200 (its 2). Sum = 500 = A's full P&L. Verified the test fails pre-fix at 500/500/sum=1000 by stashing the production change.
- Unpaired leaf lot (the IBIT 42C add-on shape, `parent_lot_id=NULL`): excluded from cumulative_realized_pnl since it has no upstream lineage.

Suite: 220 → 222.

User action: `/api/reprocess` once for the new cumulative numbers to flow into roll_chain_summaries (the rollup gets rebuilt every run, so this is just timing).

**Phase 3 remaining**: spec doc rewrite (lot-centric reframe of `docs/roll-detection-spec.md` §1, §5, §6). Doc-only; tracked as Phase 3b.

### 2026-05-01 — Steve Johnson

**Phase 3c shipped on `main` (commit `44fadb8`).**

Per-lot chain attribution via single rule: *every contract belongs to the chain its lineage ends up in.* Walk forward via `parent_lot_id` (each lot has at most one child per spec §0.2) until a terminal lot; the terminal lot's group identifies the chain. Orphans (terminal lands in a non-leaf group) tie-break to the chain whose leaf opened most recently.

Both invariants hold simultaneously for the first time:
- **Per-chain math matches the chain modal.** User's IBIT 41-strike chain now reports `cumulative_realized_pnl = -$4,652` exactly matching the modal's "Chain Realized" total.
- **Portfolio math is additive.** At a branching source group (rare 5x→3x+2x same-shape partition), each source contract follows its single child; sibling chains never share a contract.

Implementation
- New `build_chain_attribution()` in `src/pipeline/lot_lineage.py` returns `(chains_by_leaf, lot_to_leaf)`. Single helper used by both rollup and modal.
- `populate_roll_chain_summaries` (rollup) sums realized + premium over each chain's attributed lots.
- `/api/ledger/group-roll-chain/{group_id}` (modal endpoint) filters per-row realized and premium to attributed lots, so per-row totals add to the chain summary's total in branching scenarios.

Tests
- Replaced the lot-walk-shaped test (`test_unpaired_leaf_lots_excluded_from_lineage`) with two scenario tests:
  - `test_chain_group_with_mixed_parent_lots_includes_all` — the user's 41-strike shape (chain root with one parent-in-chain lot + one orphan adjustment lot, both attributed to the same chain).
  - `test_lot_in_chain_does_not_pull_in_unrelated_chain` — cross-chain `parent_lot_id` link does NOT pull the unrelated chain's totals into this chain.
- Suite: 222 → 223.

Production: live `roll_chain_summaries` rebuilt against the user's data immediately after merge (90 rows). The 41-strike row will now read −$4,652 in the UI matching the modal exactly.
