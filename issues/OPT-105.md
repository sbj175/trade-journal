# OPT-105: SQLAlchemy ORM migration + Alembic setup

Replace all raw `sqlite3` queries (314 calls across 15 files) with SQLAlchemy 2.0 ORM. Set up Alembic for schema migrations. Database stays SQLite for now — PostgreSQL is a separate future step.

## Approach

Incremental migration: add `get_session()` alongside existing `get_connection()`, convert files one at a time with test suite validating each step.

## Steps

1. **Dead code cleanup** — Remove \~10 unused SQL methods to reduce scope
2. **Phase 0: Foundation** — Models, engine, Alembic init, session shim (no behavior changes)
3. **Phase 1: Leaf files** — strategy_detector, debug, accounts, reports, position_inventory, pnl_calculator
4. **Phase 2: Core logic** — lot_manager, sync_service, routers (sync, positions, ledger)
5. **Phase 3: Heavy files** — order_models, ledger_service, chain_service
6. **Phase 4: DatabaseManager** — Convert 128 queries, DDL → Alembic
7. **Phase 5: Cleanup** — Remove get_connection(), sqlite3 imports, legacy migration scripts

### 2026-02-23 — Steve Johnson

## What this achieved

The entire database layer was migrated from raw SQL strings to SQLAlchemy ORM. Every database query in the application now goes through Python objects and methods instead of hand-written SQL. The database is still SQLite — nothing changed from the user's perspective.

## Why this matters for SaaS migration

1. **Database portability**: SQLAlchemy abstracts away database-specific SQL. When we switch from SQLite to PostgreSQL for multi-tenancy, the application code won't need to change — just the connection string. Without this, we'd have to rewrite 314 raw SQL queries by hand for PostgreSQL compatibility.

2. **Schema migrations with Alembic**: Alembic is now set up to manage database schema changes. As we add SaaS features (user accounts, tenants, subscriptions), Alembic will generate and apply migrations automatically — no more manual ALTER TABLE scripts.

3. **Multi-tenancy ready**: ORM models make it straightforward to add tenant scoping (e.g., a `tenant_id` column) with query filters that apply automatically, rather than editing every raw SQL string.

4. **Safer and more maintainable**: No more SQL injection risk from string formatting, no more typos in column names that only surface at runtime, and the Python type system catches errors earlier.
