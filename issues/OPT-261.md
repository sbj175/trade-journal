# OPT-261: Position groups don't reset when all lots close before new lots open

## Problem

`group_manager.py` uses `(account, underlying, expiration)` as the routing key for option lots (line 77). Once a position group exists for that key, any new lots with a matching key join the same group — even if all previous lots in that group have already closed and the new lots belong to a chronologically distinct trade idea.

This means two unrelated trades with the same underlying and expiration get merged into one position group, and the strategy recognizer runs on the union of their legs. That produces phantom strategies (e.g., a leg from trade A paired with a leg from trade B via geometry alone).

## Repro

* Account: Roth IRA ([sbj175@gmail.com](<mailto:sbj175@gmail.com>))
* Symbol: SPY, expiration 2025-07-18
* Trade A: Bull Call Spread 601C/606C, opened 2025-05-20, closed 2025-06-02
* Trade B: 2× Put Butterfly 555P/565P/570P, opened 2025-06-02 (same day trade A closed), closed 2025-06-17
* Observed: three groups, one of which is `Long Strangle (2×570P + 1×601C)` with span 5/20–6/17 — legs from both trades mashed together
* Expected: two distinct groups (one per trade), each with its own date span

## Where to fix

* `src/pipeline/group_manager.py` — when resolving the group key for a new option lot, check `_is_group_open(group_key)` (already exists at line 86). If the existing group has zero open lots across all its members, route the new lot to a **new** group key instead of re-using the closed one.
* Equity routing at line 79 may need the same treatment.
* Chain-aware routing at line 81 (rolls via `chain_id → group_key`) should continue to bypass the reset, since rolled positions intentionally extend the original chain.

## Acceptance

* Two trades on the same `(account, underlying, expiration)` separated by full flat-state in between produce two separate `PositionGroup` rows
* Rolls still extend their original chain (chain-routed lots don't get a new group)
* Strategy recognition runs per group, not across chronologically distinct trades

## Related

Paired with the missing-butterfly-pattern bug (separate issue). Fixing both is required to clean up the real-world case above.

## Comments

### 2026-04-26 — Steve Johnson

Resolved by the OPT-270 / OPT-274 / OPT-275 work in `group_manager.py`.

Rule 1 routing now requires `_is_group_open(candidate_group)` before merging — so when Trade A's group fully closes (all lots have remaining_quantity=0), a subsequent lot at the same `(account, underlying, expiration)` falls through to Rule 3 and creates a new group instead.

Verified on the SPY 2025-07-18 repro from the issue:
- gid 8218: Bull Call Spread 601/606 (5/20 → 6/2)
- gid 8236: Put Broken Wing 555/565/570 (6/2 → 6/17)

Two clean groups with independent date ranges. The phantom "Long Strangle (2×570P + 1×601C)" merged group is gone.

The "missing-butterfly-pattern bug" mentioned as paired also appears resolved — the recognizer now returns `Put Broken Wing` for the 1×2×1 ratio shape.
