# OPT-113: Per-user Tastytrade OAuth credential management

Transform Tastytrade credential management from a global singleton to per-user: each authenticated user stores their own encrypted OAuth credentials in the database and gets their own cached connection.

Key changes:

* Credential encryption module (Fernet)
* UserCredential database model + migration
* Per-user connection pool in ConnectionManager
* `get_tastytrade_client` FastAPI dependency
* All routers updated to use new dependency
* Frontend: delete credentials support
* Auth-disabled path unchanged (backward compatible)

### 2026-02-24 — Steve Johnson

### Why this matters for SaaS migration

Before OPT-113, there was a fundamental blocker: even though Supabase Auth (OPT-110) gave each user their own tenant-scoped data, **every user shared the same Tastytrade connection** from the server's `.env` file. This meant all users saw the same brokerage accounts and synced the same trades — defeating the purpose of multi-tenancy.

Now each user brings their own Tastytrade credentials, stored encrypted in the database, and gets their own cached connection. This completes the per-user isolation story:

- **Auth (OPT-110)** isolates *data* — each user only sees their own trades, positions, and chains
- **Credentials (OPT-113)** isolates *brokerage access* — each user connects to their own Tastytrade account

Together these two pieces make the app genuinely multi-user: User A and User B can sign up, enter their own Tastytrade credentials, sync independently, and never see each other's data. The single-user local mode is fully preserved when auth is disabled.

### Testing Instructions

#### Prerequisites

Apply the new migration first:

```bash
# SQLite
venv/bin/alembic upgrade head

# PostgreSQL (if using)
DATABASE_URL=postgresql://optionledger:optionledger@localhost:5432/optionledger venv/bin/alembic upgrade head
```

---

#### Test 1: Auth Disabled (backward compatibility)

Make sure no Supabase env vars are set in `.env`, then start the app normally.

1. Start the app: `./start.sh`
2. Verify startup log shows `"OAuth credentials found, connecting to Tastytrade..."` (not the multi-user message)
3. Visit `/settings` → Connection tab — should show Connected with your accounts
4. Sync on Positions or Chains page — should work exactly as before
5. The `user_credentials` table should remain empty (credentials still come from `.env`)

---

#### Test 2: Auth Enabled — Fresh User, No Credentials

Set your Supabase env vars in `.env` and **remove** (or comment out) `TASTYTRADE_PROVIDER_SECRET` and `TASTYTRADE_REFRESH_TOKEN`:

```
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_ANON_KEY=eyJ...
SUPABASE_JWT_SECRET=your-secret
# TASTYTRADE_PROVIDER_SECRET=...
# TASTYTRADE_REFRESH_TOKEN=...
```

Also add an encryption key (or let it auto-generate):
```
CREDENTIAL_ENCRYPTION_KEY=<paste the key from startup log, or generate one with python>
```

1. Start the app — log should say `"Multi-user mode: per-user Tastytrade connections on demand"` (no auto-connect)
2. Log in via `/login`
3. Visit `/settings` → Connection tab — should show **"Not Configured"** (yellow badge)
4. Try syncing on Positions page — should get a 503 error ("Tastytrade not connected")

---

#### Test 3: Auth Enabled — Save Credentials & Connect

Continuing from Test 2:

1. On `/settings` → Connection tab, enter your Client Secret and Refresh Token
2. Click **"Save & Connect"**
3. Should show **"Connected"** (green badge) with your accounts listed
4. The "Remove Credentials" button should now appear
5. Go to Positions page → click Sync → should work, positions populate
6. Go to Chains page → click Sync → transactions import, chains build

Verify in the database that credentials are encrypted:
```bash
venv/bin/python -c "
from src.database.engine import init_engine, get_session
init_engine(None)
from src.database.models import UserCredential
with get_session() as s:
    for row in s.query(UserCredential).all():
        print(f'user={row.user_id[:8]}... provider={row.provider}')
        print(f'  secret={row.encrypted_provider_secret[:40]}...')
        print(f'  token={row.encrypted_refresh_token[:40]}...')
"
```

The values should be Fernet-encrypted blobs, not plaintext.

1. After Test 3, navigate between pages (Positions → Chains → back)
2. Each page triggers API calls that need the Tastytrade client
3. Check the logs — you should see `"Cached new connection for user ..."` only once, then subsequent requests reuse it (no repeated auth messages)

2. Confirm the dialog
3. Status should flip to **"Not Configured"**

5. Re-enter credentials and Save & Connect to restore

#### Test 7: Multi-User Isolation (if you have two Supabase accounts)

1. Log in as User A → save TT credentials → sync
2. Log out → log in as User B → Settings should show "Not Configured" (User B has no credentials yet)
3. User B saves their own credentials → sync → sees only their own accounts/data
4. Switch back to User A — their data is still intact, independent of User B

---

#### What to Watch For

- **Startup log**: "Multi-user mode" vs "OAuth credentials found" — confirms the right path
- **No plaintext credentials in DB** — always encrypted Fernet tokens
- **503 errors only when expected** — no credentials configured, or bad credentials
- **`CREDENTIAL_ENCRYPTION_KEY` warning** — if you didn't set it in `.env`, the startup log will print a generated key; copy it to `.env` before restarting or you'll lose access to previously encrypted credentials
