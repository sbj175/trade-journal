# OPT-294: Partial sync drops Rolled tag on existing chains (full reprocess restores it)

After importing new close transactions via the regular sync codepath (incremental / `affected_underlyings`), the **Rolled** tag on an existing covered-call chain disappears. Running `/api/reprocess` (full reprocess, no `affected_underlyings`) restores the tag immediately with the same underlying data.

The two codepaths therefore disagree on at least one of:

- `position_lots.parent_lot_id` (lineage — produced by `detect_lot_lineage`)
- `position_groups.rolled_from_group_id` (derivation — produced by `derive_rolled_from_group_id`)
- `position_group_lots` membership (group routing — produced by `GroupPersister.process_groups`)

The Rolled tag in the UI is derived from `rolled_from_group_id`, so the visible symptom is downstream of whichever of the three actually diverges.

## Reproduction

1. Have an existing rolled covered-call chain in the DB (e.g. IBIT covered calls in the Trad IRA, currently sitting on a 41.5 short call that was rolled from a lower strike). The 41.5 group has `rolled_from_group_id` set to the prior (closed) source group, and the Ledger UI shows the **Rolled** tag.
2. Import a new batch of transactions that closes (or partially closes) one of the IBIT lots — e.g. the user's regular sync this morning that brought in fresh IBIT close transactions.
3. Reload the Ledger. The Rolled tag on the IBIT 41.5 group is gone.
4. Trigger `POST /api/reprocess` (full reprocess). Reload. The tag is back.

The bug is **codepath-specific**, not data-specific — same `raw_transactions` table contents, different result depending on whether `affected_underlyings` is passed to `orchestrator.reprocess()`.

## Diagnosis (initial — to be confirmed by diff harness)

`src/pipeline/orchestrator.py:65-198` accepts `affected_underlyings: Optional[Set[str]] = None`.

**Full reprocess** (`affected_underlyings=None`):
- `lot_manager.clear_all_lots()` — wipes every lot
- `_clear_groups()` — wipes every group, group-lot link, tag, note, roll chain summary
- Rebuilds everything from raw transactions in one pass

**Partial sync** (`affected_underlyings={'IBIT'}`):
- `lot_manager.clear_all_lots(underlyings={'IBIT'})` — wipes only IBIT lots; **groups are not cleared**
- Orders filtered to IBIT only before `process_lots`
- `detect_lot_lineage` runs over the whole DB (lot_lineage.py:68-115 — global reset + global re-pair)
- `GroupPersister.process_groups(account_number=…)` runs in incremental mode — adapts existing non-IBIT groups instead of rebuilding from zero
- `derive_rolled_from_group_id` walks every group

Two candidate root causes to verify, ranked by likelihood:

1. **`GroupPersister.process_groups()` incremental routing differs from full-rebuild routing.** When the existing 41.5 group is already persisted, the partial-sync routing may attach the new closes (or any newly-rebuilt IBIT open lots) to the wrong `position_group`, breaking the "all lots in the target have parent_lot_id pointing to lots in a single source group" precondition that `derive_rolled_from_group_id` requires (lot_lineage.py:172-243).

2. **Source group's `is_closed` flag flips when new IBIT closes restructure lot membership.** `derive_rolled_from_group_id` requires the source group to be **fully CLOSED**. If the new closes cause the source group to gain or retain an open lot, the precondition fails and the Rolled tag is cleared even though `parent_lot_id` is correct.

The earlier hypothesis that "`detect_lot_lineage` resets `parent_lot_id` globally but only re-pairs within the current batch's days" does **not** hold up on closer reading — `closes_bucket` is built from every `LotClosing` row in the DB (lot_lineage.py:76-84), not just rows touched by the current batch, so `days_of_interest` spans full history regardless of `affected_underlyings`.

## Plan

1. **Snapshot** the current broken state for the IBIT Trad IRA chain — capture `position_lots`, `position_group_lots`, `position_groups.rolled_from_group_id`, and the relevant `lot_closings` rows.
2. **Run a full reprocess inside a rolled-back transaction** (or against a Postgres `pg_dump` snapshot) and diff the two states row-by-row. Whichever table diverges is the smoking gun.
3. Trace the divergent stage in code.
4. Write a **regression test** that loads a two-phase fixture (sync #1 opens + rolls the chain, sync #2 lands the new closes) and asserts partial-sync ≡ full-reprocess for `rolled_from_group_id` on every affected group.
5. Fix.
6. Re-run the IBIT case end-to-end (without full reprocess) and confirm the Rolled tag returns.

## Out of scope

- Restructuring `detect_lot_lineage` / `derive_rolled_from_group_id` into a single pass — OPT-289 covers spec-doc + fixture follow-ups for the roll model.
- Same-day legged-out-and-back-in rolls — OPT-280.

### 2026-05-18 — snapshot pinpoints the divergence

Snapshot of the broken state (`scripts/opt294_snapshot.py`, output saved to `scripts/opt294_state_partial.json`) revealed a structurally impossible group in account **5WZ26959**:

- **Group `15cc835f`** — opened 2026-04-24, `status=OPEN`, `closing_date=None`, **`rolled_from_group_id=NULL`** (← missing tag) — contains:
  - lot **36234**: strike **41.0**, `status=CLOSED` — the leg that was rolled out of on 2026-05-01
  - lot **36241**: strike **41.5**, `status=OPEN`, `parent_lot_id=36234` — the post-roll leg, entry_date 2026-05-01

A single position group cannot span two strikes from before and after a roll. The post-roll 41.5 lot should live in its **own** group, linked by `rolled_from` to the (now fully-CLOSED) source 41.0 group.

For comparison, the same chain pattern in **5WZ28644** (which did NOT have this bug) produced the correct shape:
- Group `3ec01048` — closed 2026-05-01, contains lot 36233 (41.0, CLOSED)
- Group `d45c2601` — opened 2026-05-01, OPEN, **`rolled_from=3ec01048`** ✓, contains lot 36238 (41.5, OPEN, `parent_lot_id=36233`)

The asymmetry between the two accounts is the tell: 5WZ26959 had a pre-existing persisted group `15cc835f` from a prior sync, and the new 41.5 open lot was **merged into it** rather than placed in a new group. 5WZ28644's groups happened to be created fresh during today's sync, so the routing produced the correct shape.

Therefore the bug is in **`GroupPersister.process_groups()` incremental routing**, not in `detect_lot_lineage` (`parent_lot_id` is correctly set on lot 36241) and not in `derive_rolled_from_group_id` (it correctly returns NULL given that the parent lot is in the *same* group as the target). The downstream symptom — missing Rolled tag — is a faithful projection of the upstream group-membership error.

Hypothesis for the routing bug: when assigning a newly-built lot to a group in incremental mode, the router considers the existing same-expiration sibling group as a merge candidate and fails to check whether the candidate is the source group of a roll (i.e. whether `target_lot.parent_lot_id` points to a lot already in the candidate group). Per OPT-287 the orchestrator runs `detect_lot_lineage` **before** group routing precisely so routing can consult `parent_lot_id` — but that guard appears not to fire in the incremental codepath.

Next step: read `src/pipeline/group_manager.py` to find the exact merge decision and fix.

### 2026-05-18 — root cause + fix

**Root cause confirmed in two places that disagree about an invariant:**

- `src/models/lot_manager.py:546-549` documents that `clear_all_lots(underlyings=…)` *intentionally* preserves `PositionGroupLot` rows because `transaction_id` is stable across rebuilds. The comment claims stale links are cleaned up downstream.
- `src/pipeline/group_manager.py:670` does the opposite of cleanup — it filters routing to only lots whose `transaction_id` is **not** already linked to a group: `new_lots = [lot for lot in all_lots if lot.transaction_id not in txn_to_group]`. So once a link exists, the OPT-287/288 `parent_lot_id` routing guards (lines 104-173) **never re-evaluate it**.

A bad link persisted from any earlier sync (a routing bug, an out-of-order import, etc.) is sticky and survives every subsequent partial sync. Full reprocess works only because `_clear_groups` in `orchestrator.py:34-52` wipes `position_group_lots` entirely, so every lot is "new" and re-routes through the correct logic.

**Fix (`src/pipeline/group_manager.py` Phase 1.5):** at the start of `process_groups()`, scan every existing link and delete any whose child lot's `parent_lot_id` points to another lot **in the same group** at a different strike / expiration / option_type. That structural mismatch is a logical impossibility for a covered-call roll chain. Phase 3 then re-routes the orphaned lot freshly — picking up the OPT-288 `rolled_from_group_key` lineage at mint time so the new group's `rolled_from_group_id` is correct without needing `derive_rolled_from_group_id` to back-fill.

**Regression test:** `tests/unit/test_group_manager.py::TestGroupPersister::test_self_heal_splits_roll_boundary_violation` constructs the broken shape directly (parent CLOSED + child OPEN, parent_lot_id linked, both in one group at different strikes) and asserts the persister splits them with rolled_from set.
