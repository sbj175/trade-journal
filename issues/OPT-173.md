# OPT-173: Sync/reprocessing undoes user group merges

## Bug

When a user merges groups via the Ledger suggestion system ([OPT-170](https://linear.app/optionedge/issue/OPT-170/ledger-group-merge-suggestions-detect-potential-multi-group-strategies)), the merge is lost on the next sync because `process_groups()` rebuilds all groups from scratch based on chain relationships.

## Root Cause

`GroupPersister.process_groups()` in `src/pipeline/group_manager.py` depends on chains as an intermediate step:

```
transactions → lots → chains → groups
```

1. `chain_graph.py` builds Chain objects from lots/lot_closings (union-find)
2. `assign_lots_to_groups()` derives grouping from those chains
3. On every sync, groups are rebuilt from scratch based on chain structure
4. User-initiated merges (which span multiple chains) get split apart
5. Orphaned groups from the old merge get deleted (lines 481-496)

## The Real Problem

This is fundamentally a chain dependency that should be eliminated. Lots already contain everything needed to build groups directly — `order_id`, `closing_order_id`, underlying, account, strike, option type. The chain step is just connecting related lots via their closing relationships, which could be done directly from lot data.

## Desired Approach

Remove the chain dependency from group building:

* Build groups directly from lots (lot relationships via `order_id`/`closing_order_id`)
* Make group processing incremental — only handle new/changed lots, leave existing groups untouched
* This naturally fixes the merge problem since existing groups aren't torn down and rebuilt

This is part of the broader effort to eliminate chain dependencies (see [OPT-169](https://linear.app/optionedge/issue/OPT-169/migrate-reports-page-from-order-chains-to-position-groups-lots) which removed chains from Reports).
