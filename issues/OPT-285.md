# OPT-285: OPT-285: Same-shape opens at same expiration merge into one group

## Symptom

User report (2026-04-30): in a Traditional IRA on Mar 6 the user sold 80 IBIT 39C/Mar-9 and bought them back the same day. The Ledger shows TWO groups instead of one — 11 contracts in one and 69 in the other — even though it's a single logical position. The broker multi-filled the order at two prices ($0.47 / $0.48), creating two position lots; the routing layer then refuses to merge them.

## Why this happens

`_route_lot_to_group` Rule 1 calls `_is_duplicate_open_lot`, which returns True if the existing group already has a same-`(option_type, strike, direction)` open lot. When True, Rule 1 falls through and the second lot mints a new group. The guard was added to enforce spec §4's old rule "two simultaneous similar positions on the same expiration are two groups," but the rule conflicts with spec §3 (adjustments — sizing up, multi-fill — are the same logical position).

TastyTrade itself collapses same-shape lots into one position regardless of fill prices or order count, which is what the user expects.

## Fix

* Drop the `_is_duplicate_open_lot` guard from Rule 1. Same-shape lots at the same `(account, underlying, expiration)` always merge into one group.
* Genuinely parallel positions (two distinct trade ideas at the same strike/expiration) are exotic and out of scope for auto-detection — the user can express them via manual link/split (§6) when that lands.
* Spec §4 amended to match: "Same-shape lots at the same (account, underlying, expiration) merge into one group" (the inverse of what it used to say).
* Updated tests:
  * `test_route_lot.py::test_structural_duplicate_does_not_merge` flipped → `test_same_shape_lots_merge_into_one_group`
  * `test_orchestrator.py::test_separate_opens_at_same_strike_and_expiration_create_separate_groups` flipped → `..._merge_into_one_group`
* The lot-lineage chain math is unaffected (lots still have distinct chain_ids and parent_lot_ids); the merge happens only at the group view layer.

## Production data

User should `/api/reprocess` to re-bucket existing same-shape multi-fill cases into single groups. The IBIT 11/69 split will collapse to a single group of 80 contracts. Other same-shape splits across the user's history will likewise consolidate.
