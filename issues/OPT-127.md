# OPT-127: Fix business-key PKs across 8 tables for multi-tenant support

Follow-up to OPT-126. Eight tables use external/business-key primary keys without user_id in the uniqueness constraint. This causes two users sharing the same Tastytrade account to collide on inserts — `on_conflict_do_nothing` silently drops all data for the second user.\\n\\nTables affected: RawTransaction, Order, OrderChain, OrderChainCache, PositionGroup, PositionGroupLot, OrderComment, PositionNote.\\n\\nFix: add surrogate integer PK to each, composite UniqueConstraint(business_key, user_id), drop cross-table FKs (all tables are independently user_id-scoped), update all on_conflict index_elements.

## Comments

### 2026-02-26 — Steve Johnson

Implemented and merged to main. Migration `fix_business_key_pks_007` tested on both SQLite and PostgreSQL.

**Root cause**: Nine tables used external business-key primary keys (e.g., Tastytrade transaction ID, order ID) without `user_id` in the uniqueness constraint. When user B synced the same Tastytrade account as user A, `on_conflict_do_nothing` silently dropped all inserts because user A's rows already existed with the same PK.

**Fix pattern applied to all 9 tables**:
1. Added surrogate integer PK (`id` / `row_id`)
2. Changed business key to `UniqueConstraint(business_key, user_id)`
3. Updated all `on_conflict` calls to use composite `index_elements` including `user_id`
4. Dropped cross-table FK constraints (all tables independently user_id-scoped)
5. Fixed `session.get()` calls that assumed business-key PKs

**Tables changed**: `raw_transactions`, `orders`, `order_chains`, `order_chain_cache`, `order_chain_members`, `order_positions`, `position_lots`, `position_groups`, `position_group_lots`, `order_comments`, `position_notes`

226 tests pass.

### 2026-02-26 — Steve Johnson

**Follow-up fix: scoped all ORM `.delete()` calls to current `user_id`**

The PK fix alone didn't fully solve multi-tenant isolation. The tenant filter in `do_orm_execute` only scopes SELECT queries — DELETE operations bypass it entirely. This caused `order_chain_cache` (and other tables) to only retain data from the last user who synced.

Fixed 19 `.delete()` calls across 9 files to include explicit `user_id` filtering:
- `chain_service.py` — cache/member/chain deletes in all 3 branches
- `lot_manager.py` — `clear_all_lots()` both branches
- `order_models.py` — order position delete, chain member delete, reprocess bulk deletes
- `group_manager.py` — group-lot stale links and orphan group cleanup
- `ledger_service.py` — `_refresh_group_status` group delete
- `ledger.py` — group-lot reassignment delete
- `db_manager.py` — position save delete, sync metadata reset, strategy target reset, comment/note deletes
- `settings.py` / `tastytrade_oauth.py` — credential deletes

All 226 tests pass. Merged to main.
