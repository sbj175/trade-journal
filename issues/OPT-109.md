# OPT-109: Add user_id to all tables + tenant scoping (Phase 1)

## Phase 1 of SaaS conversion: Multi-tenant data isolation

Add `user_id` column to every data table with automatic query filtering so each user only sees their own data. Uses a default user until authentication is implemented in a later phase.

### Scope

* **User model** + default user seeding
* `user_id` column on 19 data tables (nullable, indexed, FK to [users.id](<http://users.id>))
* **SQLAlchemy event-based tenant filtering** (`do_orm_execute` + `before_flush`)
* `user_id` in all `dialect_insert()` call sites (\~30 across 7 files)
* **Updated unique constraints** (SyncMetadata, StrategyTarget, PositionsInventory)
* **Alembic migration** for schema changes
* **Updated SQLite-to-PG migration script**
* **Tenant isolation tests**

### 2026-02-24 — Steve Johnson

## What was done (plain English)

Every table that stores user data now has a `user_id` column that tags each row with the user who owns it. A new `users` table was created to hold user records, and a "default user" is automatically created so all existing data continues to work exactly as before.

The key innovation is **automatic tenant filtering** — we wired up two SQLAlchemy hooks that run behind the scenes:

1. **Every database read** automatically adds "…and user_id = current user" to the query. No developer has to remember to filter — it just happens.
2. **Every database write** automatically stamps the current user's ID onto new records. Again, no manual work needed.

This means the app behaves identically to before for a single user, but the infrastructure is now in place to support multiple users seeing only their own data.

## Why this matters for SaaS

This is the foundational layer that makes multi-user possible. Without it, every user would see everyone else's trades, positions, and settings — obviously a non-starter for a paid product.

**What this enables for the next phases:**
- **Authentication (next step):** Once we add login/JWT, the "current user" will come from the auth token instead of the hardcoded default. The tenant filtering will immediately start isolating data per logged-in user — no additional database changes needed.
- **User registration & onboarding:** The `users` table is ready to store email, display name, and active status for real users.
- **Per-user settings:** Strategy targets, sync metadata, and position notes are already scoped per user, so each customer gets their own configuration.
- **Stripe integration:** When we add subscriptions, we can tie billing status to the `users` table and gate access accordingly.

**What didn't change:** The app still runs locally with no login required. All existing data is owned by the default user. Nothing breaks — 82/82 tests pass.
