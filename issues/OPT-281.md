# OPT-281: Formalize roll-detection spec + scenario test matrix (epic)

## Why

Roll detection accumulated four edge cases in a single session — each found by the user spotting real-world data that didn't match their mental model. Without a written spec, every fix has been incremental whack-a-mole, and we keep trading one edge for another:

| Issue | Edge case |
| -- | -- |
| [OPT-261](https://linear.app/optionedge/issue/OPT-261/position-groups-dont-reset-when-all-lots-close-before-new-lots-open) | Trade A closed → Trade B opened same day with **different chain & different signature** → must NOT link (phantom-strangle prevention) |
| [OPT-279](https://linear.app/optionedge/issue/OPT-279/same-expiration-rolls-create-new-groups-instead-of-staying-in-one) | Same-exp roll within ONE ROLLING order → must stay in **one group** (not two linked) |
| [OPT-280](https://linear.app/optionedge/issue/OPT-280/roll-detection-misses-same-day-legged-out-and-back-in-rolls-chain-only) | Same-day legged-out-and-back-in via separate orders → must link via **signature match** when no chain overlap |
| [OPT-280](https://linear.app/optionedge/issue/OPT-280/roll-detection-misses-same-day-legged-out-and-back-in-rolls-chain-only) follow-up | Source group's signature must filter to **lots active at the closing event**, ignoring rolled-out mid-life legs |

Each leaf is correct in isolation, but the tree they form has never been written down. The roll-chain modal's "learn more" content is the user-facing summary; the actual implementation lives in `_detect_roll_links` and the routing rules. Today they're in sync — but only because we keep noticing when they drift.

## Goal

A canonical, written spec for roll detection that:

* The implementation references (the code's docstring points to the spec)
* The "learn more" UI text is generated from (or carefully kept consistent with)
* New edge cases are added to as YES/NO branches with concrete examples
* A scenario test matrix iterates so adding a row is the only thing required to cover a new case

## Sub-bullets

### 1\. `docs/roll-detection-spec.md`

A decision-tree document with:

* The four primary inputs: same account, same underlying, close.day == open.day, presence of chain overlap
* The structural inputs: leg-shape signature (multiset of (option_type, sign)), strikes, expiration
* A YES/NO branch per leaf with a one-paragraph rationale and a named example scenario
* A "non-rolls" section enumerating cases that look like rolls but aren't (phantom-strangle, parallel-position-not-chain, etc.)

### 2\. Canonical fixtures library entries

`tests/fixtures/roll_*.py`, one module per scenario in the spec. Each module exposes `.transactions()` returning raw_tx dicts and `.expected()` returning the expected post-pipeline group structure (link / no link / merged into one group). Examples:

* `roll_same_exp_in_one_order.py` ([OPT-279](https://linear.app/optionedge/issue/OPT-279/same-expiration-rolls-create-new-groups-instead-of-staying-in-one) case)
* `roll_legged_out_and_back_in.py` ([OPT-280](https://linear.app/optionedge/issue/OPT-280/roll-detection-misses-same-day-legged-out-and-back-in-rolls-chain-only) USO case)
* `roll_with_mid_life_adjustment.py` (active-at-close filter case)
* `roll_parallel_ladder.py` (IBIT case — chain match)
* `non_roll_phantom_strangle.py` ([OPT-261](https://linear.app/optionedge/issue/OPT-261/position-groups-dont-reset-when-all-lots-close-before-new-lots-open) case)
* `non_roll_different_account.py`
* `non_roll_different_underlying.py`
* `non_roll_dates_differ.py`

### 3\. `tests/integration/test_roll_detection_spec.py`

A parameterized test that iterates the spec table:

```python
@pytest.mark.parametrize("fixture_module", ALL_ROLL_FIXTURES)
def test_roll_spec(db, lot_manager, fixture_module):
    txs = fixture_module.transactions()
    expected = fixture_module.expected()
    reprocess(db, lot_manager, txs)
    assert _canonical_view(db) == expected
```

Adding a new spec branch becomes: write the fixture module, add it to `ALL_ROLL_FIXTURES`. No new test class needed.

### 4\. UI sync

The roll-chain modal's "learn more" content becomes a concise summary derived from the spec. When the spec changes, the modal copy is updated in the same PR.

### 5\. Discipline carry-forward

* New roll-detection bugs must add a fixture module AND a spec entry (not just a one-off test)
* The decision tree is the source of truth — changes to `_detect_roll_links` reference the spec section being changed
* "Where does this case fit in the spec?" is the first question for any new edge
