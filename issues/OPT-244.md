# OPT-244: Roll not detected when additional leg opened alongside roll

## Bug

When rolling a covered call and simultaneously opening an additional call at a different strike/expiration, the roll is not detected. Instead, the strategy detector classifies the entire order as a Diagonal Spread.

## Example (real trade, IBIT, 2026-04-10)

**Transactions:**

1. Close 39-strike CC (today's expiration)
2. Open 40-strike CC (14 DTE) — this is the roll
3. Open 42-strike CC (7 DTE) — additional income, separate position

**Expected behavior:**

* Close 39 + Open 40 → detected as a **roll**, continuing the existing chain
* Open 42 → new position group (separate covered call)

**Actual behavior:**

* All three legs grouped into one order
* Strategy detector classifies as Diagonal Spread
* No roll detected, chain broken

## Root Cause (suspected)

The order assembler (`src/pipeline/order_assembler.py`) bundles all three transactions into a single order because they share the same underlying, account, and day. The strategy detector then sees two opens at different strikes/expirations and labels it a Diagonal Spread. Roll detection never gets the chance to match the close-and-reopen pattern.

## Investigation needed

* How does the order assembler decide what goes into a single order vs separate orders?
* Should the pipeline split close+open pairs for roll detection before strategy classification?
* Are there other edge cases where adding legs alongside a roll breaks detection?

## Files to investigate

* `src/pipeline/order_assembler.py` — transaction grouping logic
* `src/models/order_processor.py` — order chain processing
* `src/pipeline/strategy_engine/recognizer.py` — strategy classification
* `src/pipeline/group_manager.py` — roll detection and chain linking
