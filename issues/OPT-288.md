# OPT-288: Consolidate group routing and roll-lineage derivation into one stage

## Why now

OPT-287 fixed the immediate IBIT case by teaching Rule 1 in `_route_lot_to_group` to consult `parent_lot_id`. That fix is correct and low-risk, but it's the latest in a long line of point-fixes (OPT-279 revert, OPT-281, OPT-284 Phases 1–3c, OPT-285, OPT-285 follow-up, OPT-286 fee display, now OPT-287). Each fix has been narrowly correct; the underlying architecture keeps generating new edge cases because **roll-related decisions are spread across multiple pipeline stages that communicate by DB state, not by a single source of truth.**

## The problem

The roll lifecycle currently touches at least six stages:

1. `order_assembler.py` — group raw transactions into Order objects
2. `roll_splitter.py` — split orders that mix opens and closes
3. `lot_manager.py` / `position_ledger.py` — open/close lots, set `parent_lot_id` (OPT-284's lineage source of truth)
4. `group_manager.py` `_route_lot_to_group` — decide which group the lot joins
5. `strategy_engine/` — label the group
6. `lot_lineage.py` `derive_rolled_from_group_id` — back-fill `rolled_from_group_id` on groups from `parent_lot_id`

Stage 3 produces the canonical signal (`parent_lot_id`). Stage 4 makes group-assignment decisions without consulting that signal directly. Stage 6 tries to back-fill the link but legitimately refuses when stage 4 produced a "mixed-source" group. Each stage is locally correct; the system as a whole can be globally wrong (today's IBIT case, OPT-287 symptom).

## Goal

Make `parent_lot_id` the **single source of truth** for both group assignment AND `rolled_from_group_id`, computed in one pass. The outcome of "which group does this lot belong to" and "what is this group's predecessor" should be derived together, not independently.

## Possible shapes

Not pre-committing to a design — these are starting points:

1. **Merge stages 4 and 6.** Walk lots in chronological order. For each lot:
   - If `parent_lot_id` is unset → fall through to today's routing rules (chain / aue / au).
   - If `parent_lot_id` is set AND parent's group is fully CLOSED → mint a new group, record `rolled_from = parent's group`.
   - If `parent_lot_id` is set AND parent's group is still open → adjustment, route into parent's group.
   Both the group assignment AND the lineage link are decided in this one pass.

2. **Lot-first, group-later.** Build the full lot-lineage graph first (already exists in `lot_lineage.py:detect_lot_lineage`). Then derive groups as connected components of the lineage, partitioned by status (closed vs open) and labeled by strategy. Routing rules become emergent properties of the lineage graph rather than ad-hoc heuristics layered over it.

3. **Stage 4 reads the parent map.** Less ambitious: keep stages separate but give stage 4 read access to the lot-lineage map so its decisions are consistent with what stage 6 will do. Lower-risk middle ground.

## Acceptance criteria

- All existing roll-detection tests pass (OPT-278 / OPT-281 fixture matrix, OPT-287 new fixture, OPT-285 multi-fill cases).
- After the refactor, deleting `derive_rolled_from_group_id` as a separate stage is possible (its outputs come "for free" from the unified pass).
- A documented invariant: `parent_lot_id` determines group membership AND `rolled_from_group_id`; no other code path can mutate either without going through this stage.

## Risks

- The current spaghetti has bedded-in semantics (e.g. OPT-285's same-shape multi-fill merge, OPT-270's invariant about stale closed groups not absorbing new opens, the "fully-closed source falls through" comment in Rule 0). All of these were correct fixes for real bugs. The refactor must preserve every one of those behaviors — that means expanding the test matrix, not contracting it.
- Touching the pipeline orchestrator + 1–2 stages downstream. Probably worth a feature branch with the test matrix landing first, then the refactor.

## Not blocking

OPT-287 ships independently. This issue is the structural follow-up — schedule when test matrix coverage feels strong enough to refactor confidently.
