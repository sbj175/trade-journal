# OPT-271: Full reprocess is not idempotent — stale duplicate position_groups accumulate

## Status: Canceled — original diagnosis was wrong

The "duplicate" position_groups I observed are **not** stale residue from prior reprocesses. They are consecutive gids created in the same reprocess pass, and they correspond to legitimate **parallel covered-call positions** on the same expiration dates ("Covered Call ladder strategy" per the user). Pre-[OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same), these parallel positions were fused into single groups by the same-expiration grouping rule. The [OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same) fix correctly separates them, which is why the chain summary's BFS walk now reveals the true 12+ roll count instead of the artificially-collapsed 8.

In other words, "8 rolls" was the lie (fusion hiding parallelism); "12 rolls" is closer to the truth — though there may still be an open question about how the chain modal *should* present parallel chains for usability (separate issue if it matters).
