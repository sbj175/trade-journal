# OPT-126: Major multi-tenant issue

Tables accounts and account_balances have only 3 rows globally. These are not being used in a multi-tenant fashion. This is a major security issue.

## Comments

### 2026-02-26 — Steve Johnson

Implemented and merged to main. Changes:

- Added surrogate `id` PK to `accounts` table
- Added `UniqueConstraint("account_number", "user_id")` so each user can have their own row for the same account
- Updated `save_account()` upsert to use composite `index_elements=['account_number', 'user_id']`
- Removed `user_id` from the `set_` dict (it's now part of the conflict key, not updatable)

**Bug 2 — Initial sync scoping:**
- All 8 model deletes in `initial_sync` now filter by `user_id` instead of using unscoped `table.delete()` / `.delete()`
- Models fixed: `OrderChainMember`, `OrderChainCache`, `OrderChain`, `OrderPosition`, `Order`, `Position`, `AccountBalance`, `RawTransaction`

**Cleanup:**
- Dropped FK from `positions.account_number → accounts.account_number` (both tables independently user_id-scoped)
- Removed `Account ↔ Position` ORM relationship
