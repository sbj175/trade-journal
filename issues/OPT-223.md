# OPT-223: Websocket management

If you let raw WebSocket connections run wild in a SaaS like real-time stock quotes, you’ll get crushed pretty quickly—connection count, bandwidth, and upstream data costs all explode. The trick is to treat WebSockets as a scarce resource and actively manage them.

Here’s how this is usually handled in production:

\---

1\. Connection limits + authentication

You don’t just let anyone open infinite sockets.

Require auth (API key / JWT)

Enforce per-user connection limits (e.g., 1–3 sockets per account)

Kill oldest connection if they exceed limits

This alone stops someone from opening 50 tabs and hammering you.

\---

2\. Idle + heartbeat timeouts

Browsers sitting open ≠ active users.

Use ping/pong (heartbeat) every \~20–30 seconds

If no response → terminate connection

Track last meaningful activity (subscriptions, messages)

You can also:

Drop connections after X minutes of inactivity

Force periodic reconnects (e.g., every few hours)

\---

3\. Subscription-based streaming (huge one)

Don’t stream “everything” to everyone.

Instead:

Client subscribes to specific tickers: AAPL, SPY, etc.

Server only pushes what they asked for

Even better:

Deduplicate internally → if 10,000 users want AAPL, you fetch it once, fan it out

This is the core scalability pattern.

\---

4\. Rate limiting + throttling

Even with subscriptions, updates can be insane.

Throttle updates per client (e.g., max 5–10 msgs/sec)

Batch updates (send snapshots every 200–500ms instead of tick-by-tick)

Offer tiers:

Free → delayed or throttled

Paid → real-time / higher frequency

\---

5\. Connection tiering (business model meets engineering)

This is where SaaS control really kicks in:

Free users:

Fewer symbols

Slower updates (e.g., 15 sec delay)

Aggressive disconnect rules

Paid users:

More symbols

Lower latency

Priority connections

You’re aligning cost with revenue.

\---

6\. Backpressure handling

If a client can’t keep up:

Detect slow consumers (buffer growing)

Either:

Drop messages (common for market data)

Or disconnect them

You never want one slow browser tab consuming server memory.

\---

7\. Horizontal scaling + fan-out layer

At scale, you don’t have one WebSocket server.

Typical architecture:

Edge WebSocket servers (stateless)

Pub/Sub layer (Redis, NATS, Kafka)

Market data ingested once → published → fanned out

So:

1 upstream feed → millions of clients efficiently

\---

8\. Force reconnect / session rotation

Long-lived connections are dangerous.

Force reconnect every N hours

Helps:

Load balancing

Deployments

Cleaning up zombie connections

\---

9\. Detect “zombie tabs”

Browsers don’t always close cleanly.

Heartbeats (again) are key

Also track:

Sudden silence

TCP half-open connections

Kill aggressively.

\---

10\. Metering + billing

If you're running a real SaaS:

Track:

Connection minutes

Messages sent

Symbols subscribed

Enforce quotas or bill overages

\---

The big mental model

Think of WebSockets like open file handles + a live data firehose.

You control cost by:

Limiting who connects

Limiting what they get

Limiting how fast they get it

Killing anything that looks idle or unhealthy

\---

Real-world simplification

If you were building this yourself:

1 connection per user

Max 50 symbols

250ms batching

Heartbeat every 30s, drop after 2 misses

Redis pub/sub for fan-out

That alone gets you surprisingly far.
