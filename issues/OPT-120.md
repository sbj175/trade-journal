# OPT-120: Documentation: Data pipeline from raw transactions to position groups

A plain-language walkthrough of how OptionLedger transforms raw Tastytrade transactions into the organized positions and trade groups you see in the app. This serves as a reference for understanding the system's core data flow.

---

## The Big Picture

## Stage 1: Raw Transaction Ingestion

**What it does:** Fetches your trading history from Tastytrade and stores every transaction exactly as-is вҖ” no interpretation, no grouping.

**How it works:**

1. You click Sync вҶ’ the app calls the Tastytrade API for each of your brokerage accounts
2. Each transaction comes with key info: what action was taken (Buy to Open, Sell to Close, etc.), which instrument, how many contracts, at what price, and when
3. Every transaction is saved to the `raw_transactions` table using its Tastytrade ID as the key (duplicates are skipped)
4. Futures and futures options are filtered out at this stage (not supported)

**Think of it like:** Dumping your entire trade confirmation history into a filing cabinet, one slip per fill.

**Database table:** `raw_transactions` вҖ” the single source of truth. Everything else is derived from this.

---

## Stage 2: Order Grouping

**What it does:** Groups individual fills into the logical orders you placed. If your order for 5 contracts filled as 2 + 3, those two fills become one order of 5.

**How it works:**

1. Each transaction is grouped by `(account, underlying symbol, broker order ID)` вҖ” all fills from the same broker order go together
2. Within each group, identical fills (same action, symbol, price) are combined and their quantities summed
3. Each order is classified:
   * **OPENING** вҖ” only "to open" actions (you're entering a new position)
   * **CLOSING** вҖ” only "to close" actions (you're exiting a position), including system events like expiration and assignment
   * **ROLLING** вҖ” both opening AND closing in the same order (you rolled a position)

**Special cases:**

* Expirations, assignments, and exercises have no broker order ID, so synthetic IDs are generated (e.g., `SYSTEM_Expiration_2025-06-20_AAPL...`)
* Corporate actions (ticker changes) are split into separate close and open orders

**Think of it like:** Taking all those individual trade confirmations and stapling together the ones that belong to the same order.

---

## Stage 3: Lot Creation

**What it does:** For every opening transaction, creates a "lot" вҖ” a record of exactly what you bought or sold, when, and at what price.

**How it works:**

* Each opening transaction creates a row in the `position_lots` table
* A lot tracks: the symbol, quantity, entry price, entry date, whether it's long or short, and which chain it belongs to
* The lot starts with `remaining_quantity` equal to the full amount вҖ” this decreases as the position is closed

**Think of it like:** Each lot is like a receipt for a specific batch of contracts you bought or sold. You might have multiple lots for the same symbol if you entered at different times or prices.

---

## Stage 4: FIFO Lot Matching

**What it does:** When you close a position (or it expires/gets assigned), the system figures out *which* lots are being closed using First-In, First-Out вҖ” the oldest lots get closed first.

**How it works:**

1. Find all open lots matching the symbol being closed, ordered oldest first
2. Walk through them one by one:
   * Close as much as possible from the current lot
   * Calculate the realized P&L for that portion (selling price minus buying price, times quantity, times multiplier)
   * Create a `lot_closings` record documenting what was closed, how much, and the P&L
   * If the lot is fully closed, mark it `CLOSED`; if partially, mark it `PARTIAL`
3. Move to the next lot until the entire closing quantity is accounted for

**Example:** You opened 3 AAPL calls on Monday and 2 more on Wednesday. If you close 4 on Friday, FIFO matches all 3 from Monday and 1 from Wednesday. The Wednesday lot still has 1 contract remaining.

**Assignment handling:** When an option is assigned, the option lot is closed, and a new stock lot is automatically created (a "derived lot") at the option's strike price. This links the stock position back to the original option trade.

**Think of it like:** A warehouse shipping system вҖ” when an order comes in to ship 4 items, you ship the oldest 3 first, then 1 from the next batch.

**Database tables:** `position_lots` (updated) and `lot_closings` (created)

---

## Stage 5: Chain Derivation

**What it does:** Links related orders into "chains" вҖ” the complete lifecycle of a trade from opening through any rolls to final closing.

**How it works:**

1. **Opening orders** start a new chain
2. **Closing orders** are matched to existing chains by looking at which chain owns the lots being closed (this info was captured during FIFO matching in Stage 4)
3. **Rolling orders** keep the same chain вҖ” the old position is closed and the new one opens under the same chain ID
4. **Chain merging:** If a closing order happens to close lots from two different chains (rare but possible), those chains are merged into one
5. **Chain status** is determined by checking if any lots still have remaining quantity:
   * All lots closed вҶ’ `CLOSED`
   * Some lots still open вҶ’ `OPEN`
   * Assignment occurred with stock still open вҶ’ `ASSIGNED`

**Strategy detection** runs on each chain to identify complex strategies (Iron Condors, Verticals, Covered Calls, etc.) based on the combination of legs.

**Think of it like:** A chain is the complete story of a trade. "I opened an Iron Condor on Jan 15, rolled the put side on Feb 3, and the whole thing expired on March 21." That's one chain with three orders.

**Database tables:** `order_chains`, `order_chain_members`, `order_chain_cache`

---

## Stage 6: Position Groups

**What it does:** Creates the user-facing trade groupings displayed on the Ledger page. Each group represents a trade (or collection of related lots) that you can see, annotate, and manage.

**How it works:**

1. **Initial seeding:** For each chain that has lots, create one position group linked to that chain. Copy the strategy label, dates, and status.
2. **Incremental seeding:** After each reprocess, any new lots not yet in a group are assigned вҖ” either to an existing group (if their chain matches) or to a new group.
3. **Ungrouped lots** (e.g., stock from ACAT transfers with no chain) are bucketed by account + underlying into "Shares" groups.
4. **Status refresh:** Each group's status is recalculated from its lots вҖ” if any lot has remaining quantity, the group is `OPEN`; otherwise `CLOSED`.

**Key design choice:** Groups link to lots via `transaction_id` (the stable Tastytrade ID), not the internal lot ID. This means groups survive reprocessing even when internal lot IDs change.

**Think of it like:** If chains are the system's internal view of trade lifecycles, groups are the user-facing presentation. They start as one group per chain but can be manually reorganized by the user.

**Database tables:** `position_groups` and `position_group_lots`

---

## Stage 7: Equity Transactions (Separate Pass)

**What it does:** Stock buy/sell transactions are processed in a separate pass after options, because the options chain system only handles options.

**How it works:**

1. Query raw transactions for equity (stock) trades
2. Create lots for buys, close lots (FIFO) for sells
3. **Netting pass:** If assignment created -800 shares and a transfer brought in +800 shares of the same stock, these opposing lots are netted against each other

---

## Stage 8: P&L Calculation

**What it does:** Calculates profit and loss at every level.

* **Per lot closing:** Realized P&L is calculated at the moment of closing (sell price вҲ’ buy price Г— quantity Г— multiplier) and stored in `lot_closings`
* **Per chain:** Sum of all realized P&L from lot closings in that chain, plus unrealized P&L from open lots (current market price vs. entry price)
* **Per group:** Derived from the lots in the group

---

## What Happens During "Reprocess Chains"

This is a full rebuild from raw transactions:

1. Load all raw transactions from the database
2. **Delete** all derived data: lots, closings, position inventory, chains, chain cache
3. Re-run Stages 2вҖ“6 from scratch on the full transaction history
4. Position groups are **preserved** вҖ” they just get re-linked to the new lots via transaction_id

This is safe to run anytime and is useful after software updates that improve the processing logic.

## Database Table Reference

| Table | Purpose | Key Columns |
| -- | -- | -- |
| `raw_transactions` | Verbatim Tastytrade data | id, account_number, symbol, action, quantity, price, executed_at |
| `orders` | Grouped/classified orders | order_id, account_number, underlying, order_type (OPENING/CLOSING/ROLLING) |
| `order_positions` | Legs within each order | order_id, symbol, action, quantity, price, strike, expiration |
| `position_lots` | Individual lot records | transaction_id, symbol, quantity, remaining_quantity, entry_price, status, chain_id |
| `lot_closings` | Each closing event against a lot | lot_id, close_quantity, close_price, realized_pnl, closing_type |
| `order_chains` | Trade lifecycle chains | chain_id, status, strategy, realized_pnl, open_date, close_date |
| `order_chain_members` | Orders belonging to each chain | chain_id, order_id |
| `position_groups` | User-facing trade groups (Ledger) | group_id, account, underlying, strategy_label, status, source_chain_id |
| `position_group_lots` | Lots belonging to each group | group_id, transaction_id |
| `positions_inventory` | Legacy running position tallies | account, symbol, quantity, cost_basis |
