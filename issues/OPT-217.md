# OPT-217: Research: Partial roll strategy label preservation for multi-leg positions

## Problem

When a multi-leg strategy (e.g., Iron Condor) is partially rolled — only some legs closed and reopened at a new expiration — the pipeline loses the original strategy identity.

### Current behavior

1. Open Iron Condor: 520P/515P + 560C/565C, all Mar 21 → correctly labeled "Iron Condor"
2. Roll only the put spread to Apr 17 (510P/505P) → pipeline reprocesses:
   * Group manager splits by expiration → 2 groups
   * Strategy engine re-evaluates each group's open legs → "Bear Call Spread" + "Bull Put Spread"
   * Original "Iron Condor" label is lost

### What works correctly

* **Chain linkage**: All 6 lots share the same `chain_id` — the chain correctly tracks the full position lifecycle
* **Lot tracking**: Closed put spread lots are marked CLOSED, new put spread lots inherit the chain
* **Roll detection**: The roll order (close old + open new) is processed as one order

### What breaks

* **Grouping by expiration**: `assign_lots_to_groups()` groups by `(account, underlying, expiration)`. Mixed expirations after partial roll → 2 groups
* **Snapshot-based strategy detection**: `lots_to_legs()` only includes open lots. Each group has 2 open legs → vertical spread, not iron condor
* **Strategy label re-evaluation**: Labels are recomputed on every reprocess, so there's no memory of the original strategy

### Design considerations

* The position IS no longer a pure iron condor after the partial roll — it's a diagonal (mixed expirations)
* The chain knows all legs belong together, but the group layer doesn't leverage this
* Possible approaches:
  1. **Strategy label preservation**: Lock the label from the original opening and don't re-evaluate
  2. **Original strategy tracking**: Add `strategy_label_original` field to preserve opening intent while `strategy_label` reflects current state
  3. **Cross-expiration grouping**: Allow multi-leg groups to span expirations when they share a chain
  4. **Roll-aware group updates**: When a roll is detected, update the existing group rather than creating a new one
* Each approach has tradeoffs between accuracy (what the position IS now) and history (what the position WAS)

### Test data

Script `test_partial_roll.py` creates this exact scenario in the `test-4@example.com` user account (Individual Margin). Run with `venv/bin/python3 test_partial_roll.py`.
