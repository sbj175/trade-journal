# OPT-114: Positions page only shows 1 of 76 open positions after Initial Sync

After a successful Initial Sync (auth-enabled mode, OAuth flow), only 1 of 76 open positions displays on the Positions page.

**What works:**

* 76 positions saved to `positions` table (32 + 15 + 29 across 3 accounts)
* 316 chains created and saved
* 282 chains reprocessed with strategy detection
* Account selector populated correctly
* Account balances showing correctly

**What's wrong:**

* Only 57 lots seeded into position groups
* Only 1 position (NEM Bull Put Spread) visible on the Positions page
* The page fetches from `/api/open-chains` which queries `PositionGroup`, not raw `positions`

**Root cause hypothesis:**
The lot/group pipeline (`order_processor.process_transactions` â†’ `seed_new_lots_into_groups`) isn't generating lots for most positions. This is likely a pre-existing issue in the lot system, not related to OPT-53/auth changes.

**To investigate:**

* Why do 316 chains only produce 57 lots?
* Are certain position types (stocks, single-leg options) not creating lots?
* Compare lot generation between auth-enabled and auth-disabled paths

## Investigation Findings

The 57 lots / 1 visible position issue has **three contributing causes**, none related to OPT-53:

### 1. Empty string chain_id filter gap

In `order_processor.py` line 512, lots are created with `chain_id=temp_chain_id or ''`. When `temp_chain_id` isn't assigned, the lot gets `chain_id=''` (empty string, not NULL).

In `seed_position_groups()` (ledger_service.py lines 27-36):
- Query fetches lots where `chain_id IS NOT NULL` â€” includes empty strings
- Then filters with `if not chain_id: continue` â€” **skips empty strings**

Result: lots with `chain_id=''` are queried but never seeded into groups. They fall through both the "has chain_id" and "ungrouped lots" (`chain_id IS NULL`) paths.

### 2. Incomplete table cleanup during reprocessing

`lot_manager.clear_all_lots()` deletes `PositionLot` and `LotClosing` but does **not** delete `PositionGroup` or `PositionGroupLot`. Stale group records remain after reprocessing.

### 3. Many chains don't produce opening lots

Of 316 chains, only some have opening transactions that create lots. Chains with only closing/system transactions don't generate opening lots.

### Fix approach
- Treat empty string chain_id the same as NULL in `seed_position_groups()`
- Clear `PositionGroup` and `PositionGroupLot` in the reprocessing step
- Ensure ungrouped lots logic catches lots with `chain_id=''`

Implemented fixes 1 and 2 from the investigation, merged to main.

**Fix 1 â€” Empty-string chain_id gap** (`src/services/ledger_service.py`):
- `seed_position_groups()` now adds `chain_id != ''` to the query filter instead of skipping empty strings in the loop

**Fix 2 â€” Stale groups during reprocessing** (`src/models/lot_manager.py`):

- This prevents stale groups from blocking new group creation after reprocessing

All 77 tests pass. Ready for user testing â€” a reprocess-chains or initial sync should produce significantly more position groups now.

**Remaining investigation** (Fix 3): Some chains with open positions still don't create lots because `order_processor.process_transactions()` only creates lots for chains where it processes opening transactions. This is a deeper issue that may need further work.
