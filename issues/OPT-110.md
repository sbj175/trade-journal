# OPT-110: Implement Supabase Auth integration (Phase 1 auth)

## Supabase Auth Integration

Wire real authentication so multiple users can sign up, log in, and each see only their own data. Builds on OPT-109 (user_id tenant scoping).

### Scope

* Backend JWT validation with PyJWT
* ContextVar-based user_id flow (zero changes to db_manager/get_session call sites)
* FastAPI Depends(get_current_user_id) on all data endpoints
* Frontend auth client (auth.js) with Supabase JS SDK
* Login page with email/password
* Nav bar user display + logout
* Data claim flow for migrating DEFAULT_USER_ID data
* Backward-compatible: auth disabled when SUPABASE_JWT_SECRET not set

### Key Design Decisions

* Supabase UUID = local user_id (no mapping table)
* Auto-provision users on first JWT-authenticated request
* authFetch() wrapper replaces all fetch() calls in frontend

## Why This Matters for SaaS Migration

Before this change, OptionLedger was a single-user local app. The tenant scoping from OPT-109 added `user_id` columns everywhere, but there was no way to actually identify *who* was making a request — every request used the same hardcoded default user.

This change closes that gap. Now, when deployed as a SaaS product:

1. **Users can create accounts and sign in.** Supabase handles email/password auth, email verification, and session management — we don't need to build or maintain any of that ourselves.

2. **Each user's data is fully isolated.** When User A signs in, the JWT's `sub` claim flows through the contextvar into every database query, so they only ever see their own trades, positions, and settings. User B's data is invisible to them, and vice versa.

3. **New users get provisioned automatically.** The first time someone signs in, a User row is created and they start with a clean slate — no manual setup required.

4. **Existing local users can migrate seamlessly.** The data claim flow lets the first real user who signs in "adopt" all the data that was created under the default user ID, so nothing is lost when transitioning from local to cloud.

5. **Local-only deployments are unaffected.** If someone just wants to run OptionLedger on their laptop without Supabase, they simply don't set the env vars and everything works exactly as before. Zero breaking changes.

In short: OPT-109 made the database multi-tenant-ready, and OPT-110 makes the application multi-user-ready. The remaining SaaS pieces (deployment, Stripe billing, OAuth2 Tastytrade onboarding) can now be layered on top of a working auth foundation.

## Testing Instructions

### 1. Create a Supabase project (free tier)

- Go to [supabase.com](https://supabase.com) and create an account/project
- Once created, go to **Settings > API** in the Supabase dashboard
- You'll need three values from there:
  - **Project URL** (e.g., `https://abcdefg.supabase.co`)
  - **anon public key** (the long `eyJ...` string)
  - **JWT Secret** (under "JWT Settings")

### 2. Add env vars to `.env`

```
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_ANON_KEY=eyJ...your-anon-key...
SUPABASE_JWT_SECRET=your-jwt-secret
```

### 3. Run the migration and start the app

```bash
venv/bin/alembic upgrade head
./start.sh
```

### 4. Test the flow

- Visit `http://localhost:8000/positions` — should redirect to `/login`
- **Sign up** with an email/password (Supabase may send a confirmation email depending on your project settings — you can disable email confirmation in Supabase Dashboard > Auth > Providers > Email)
- After sign-in, you'll land on `/positions` with an empty state
- If you had existing data from before, you'll be able to claim it

### 5. Test auth-disabled mode (backward compat)

Comment out or remove the three `SUPABASE_*` vars from `.env`, restart the app — everything should work exactly as before with no login required.

### Tip for faster testing

In Supabase Dashboard > **Authentication > Providers > Email**, toggle off "Confirm email" so sign-ups work instantly without needing to check your inbox.

### 2026-02-24 — Steve Johnson

**Follow-up fixes committed** (`d14fc9c`):

1. **ES256/JWKS support** — Modern Supabase projects sign JWTs with ES256 (not HS256). Validator now auto-detects the algorithm from the JWT header and fetches public keys from the JWKS endpoint. HS256 legacy secret still supported as fallback.

2. **Clock skew leeway** — Added 120s tolerance for `iat`/`exp` checks. WSL2 clock drift was causing "token not yet valid" rejections.

3. **Cross-dialect upserts** — Fixed `on_conflict_do_update` in `set_sync_metadata` and `save_strategy_targets` to use `index_elements` (works on both SQLite and PostgreSQL) instead of `constraint` (PostgreSQL-only).

4. **New user seeding** — Default strategy targets are now seeded when a new authenticated user is provisioned, not just at DB init for DEFAULT_USER_ID.

5. **Frontend auth flow** — `requireAuth()` now returns a never-resolving promise when redirecting to `/login`, preventing spurious 401s from API calls that fire before the redirect completes.

6. **AUTH_ENABLED broadened** — Now triggers on `SUPABASE_URL` (for ES256) in addition to `SUPABASE_JWT_SECRET` (for HS256).
