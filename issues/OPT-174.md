# OPT-174: Empty source group left behind after accepting merge suggestion

## Bug

When merging groups via a Ledger suggestion (e.g., merging Shares + Short Call into a Covered Call group), the source group that had its lots moved out still appears on the Ledger page with zero lots.

## Expected Behavior

After accepting a merge suggestion, the source group(s) that lost all their lots should be automatically deleted.

## How It Should Work

The `POST /api/ledger/move-lots` endpoint calls `_refresh_group_status()` for all affected groups (source + target). That function checks `total_count` of `PositionGroupLot` links and deletes the group if count is 0 (lines 416-424 of `src/services/ledger_service.py`).

## Investigation Pointers

* `_refresh_group_status` queries `PositionGroupLot` without user_id scoping in the `total_count` check — could this cause a mismatch?
* The frontend `acceptSuggestion()` (ledger `App.vue:159`) handles pre-empty groups separately via `DELETE /api/ledger/groups/{gid}`, but groups that become empty after the move rely on `_refresh_group_status` cleanup
* Could be a session/commit timing issue where the lot deletion hasn't flushed before the count query runs

## Files

* `src/routers/ledger.py` — `move_lots()` endpoint (line 408)
* `src/services/ledger_service.py` — `_refresh_group_status()` (line 390)
* `frontend/src/pages/ledger/App.vue` — `acceptSuggestion()` (line 159)"
