# OPT-265: Audit raw_transaction string fields for defensive ingestion normalization

## Background

[OPT-264](https://linear.app/optionedge/issue/OPT-264/both-legs-itm-call-spread-expiration-creates-phantom-equity-lot) uncovered a latent bug: `instrument_type` from the Tastytrade SDK switched from `"EQUITY"` to `"Equity"` (title case), and our case-sensitive downstream filters silently stopped routing settlement-derived equity transactions. The fix normalized `instrument_type` at ingestion in `save_raw_transactions`, matching [OPT-235](https://linear.app/optionedge/issue/OPT-235/pipeline-not-creating-lots-for-sell-to-open-transactions-after-roll)'s earlier `action` normalization.

Both of those fields are now defensively normalized. But we only got there by bug reports, twice. Other string fields from the broker SDK still pass through raw — any of them could be the next silent breakage.

## What to audit

Every string-valued field written by `save_raw_transactions` in `src/database/db_manager.py` (lines \~175-188). The ones currently untouched:

* `transaction_type` — already uppercase by convention but unverified
* `transaction_sub_type` — used in many downstream consumers (assembler filter, lot_manager's `is_exercise` / `is_assignment`). Some consumers already defensively `.upper()`; others may not.
* `symbol` — OCC option symbols have a canonical format, but brokers sometimes return with/without spaces
* `underlying_symbol` — should match symbol parsing
* `description` — free text, no normalization expected

## What to do

For each field:

1. Grep usage across `src/` for any case-sensitive comparisons
2. If used in a filter/dispatch decision, normalize at write time
3. Write a one-shot SQL UPDATE + a small Alembic data migration to fix existing rows

## Acceptance

* All string fields used in dispatch/filter logic are normalized at ingestion (one canonical form)
* No downstream consumer relies on defensive `.upper()` for correctness (belt-and-suspenders is fine, but not load-bearing)
* Existing DB rows are normalized via a migration so reprocessing any user's data is safe across versions

## Why this matters

Tastytrade can change their SDK's string formatting with a point release and silently break our pipeline. Two occurrences (action, instrument_type) is a pattern. The third one will bite us in production on an obscure field we haven't audited.
