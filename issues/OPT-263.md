# OPT-263: Rolled position groups display is confusing — strategy label, Rolls column, visual stages

## Design principle (decided 2026-04-20)

Position rolls are represented two different ways in the data model, and both are intentional:

* **Same-expiration roll** (strike adjustment): all legs stay in one `PositionGroup`; rolled-in lots inherit the original `chain_id`. **This is position management** of a continuing trade.
* **Different-expiration roll** (time extension): a new `PositionGroup` is created and linked to the prior via `rolled_from_group_id`; a `roll_chain_summaries` row tracks the chain. **This is a new trade.**

The UI should match that mental model. Chain-of-groups rendering already works well for different-exp rolls (Rolls column, roll-chain modal). This issue is about making same-exp rolls read cleanly **within one card**, not about restructuring the data.

## Target design (decided 2026-04-21)

Chosen after iterating through 5 mockup variations with the user. Final design is TT-style staged sections with explicit roll transformations:

### Group header (the collapsed row in the Ledger list)

```
▼ SPY  Iron Condor 585/600/615/630 (1)          ROLLS: 1
       CLOSED · May 22 → Jul 15 · Realized -$362 · -62.41%
```

* Strategy label shows **current** strikes only (final composition or currently-open composition), never the union of all historical strikes. For a fully-closed trade, use the strikes as of the last open state before close.

### Expanded card content — chronological staged sections

Most recent event at top (descending). Each section has an event header with type + date + net credit/debit for that event. Section types:

* **Closing** — the event that closed out the whole position (or the final disposition legs if partially expired)
* **Roll** — one section per paired close/open event
* **Opening** — the initial position open

### Roll section — explicit transformation syntax

```
│  Roll                 Jun 27, 2025                 +$155 cr  │
│  ─────────────────────────────────────────────────           │
│   ROLLED  530/545  →  585/600                                │
│   [▸ expand for per-leg details]                             │
```

Rules for the `ROLLED X → Y` summary:

* Strikes on each side sorted ascending
* **Single-type roll** (all puts or all calls): no suffix, e.g. `ROLLED 530/545 → 585/600`
* **Mixed-type roll** (puts + calls in same event): C/P suffix on each strike, e.g. `ROLLED 585P/600P/615C/630C → 590P/605P/610C/625C`

Expanded view shows per-leg details (qty, exp, action, closed price → opened price, fees) as V4-style paired rows.

### Counting rolls — walk-and-balance in execution order (from earlier spec)

**Do NOT use** `order_id` as the bucket. Broker order IDs are unreliable — different brokers bundle multi-leg rolls differently, and traders can execute paired close/open actions in separate orders without a shared ID.

Walk the group's transactions in execution-time order:

1. Maintain pending buckets of closes and opens
2. When closes and opens signature-mirror (matching `(option_type, direction)` counts), emit a roll event and reset
3. Edge cases:
   * Legging into a single roll over 3–4 minutes: accept over-count; visual staging makes the reality clear
   * Partial-balance that never completes: orphan closes = plain close, orphan opens = plain open
   * Deterministic ordering required (stable tiebreaker for identical timestamps)

The walk-and-balance output provides the **pair mapping** needed by the `ROLLED X → Y` renderer: we know which closed leg paired with which opened leg.

## Non-goals

* Do NOT restructure same-exp rolls into a chain-of-groups
* Do NOT count rolls by `order_id`, per-leg, or per-pair
* Do NOT add "rolled up/down" language — strikes speak for themselves
* Do NOT touch the data model

## Scope

UI + API shape only. Likely changes:

* Frontend: `LedgerDesktopRow.vue` (or its expanded panel) — render staged sections, update strategy label and Rolls column
* Backend: `/api/ledger/groups/*` response includes a computed `roll_events` array (walk-and-balance output): `[{ date, credit_debit, pairs: [{ closed_lot, opened_lot }] }]`. Or compute client-side from leg data.

## Acceptance

* A trader can glance at a rolled IC group and immediately understand (a) the current structure, (b) how many times and when it was rolled, (c) which strikes were rolled to which, without reading individual leg dates
* Strategy label matches a legitimate strategy shape (4 strikes for IC, not 6)
* Rolls column uses walk-and-balance counting; same-day multi-adjustments counted correctly
* Each roll event renders as `ROLLED X → Y` summary with expandable per-leg details
* Chronological section headers show date + net credit/debit per event
* Different-exp rolls continue to render as chains of cards (unchanged)

## Comments

### 2026-04-22 — Steve Johnson

Implementation landed on branch `opt-263-rolled-position-groups-display`.

**Backend**
- `src/services/roll_timeline.py` — walk-and-balance algorithm producing `opening` / `roll_events` / `closing` sections + `current_strike_label` and `roll_count`. Sort key `(timestamp, CLOSE-before-OPEN, leg_index, lot_id)` gives deterministic pairing.
- `src/routers/ledger.py` — attaches `roll_timeline`, `roll_count`, `current_strike_label` to every group in `/api/ledger` response. `has_roll_chain` + `rolled_from_group_id` untouched (different-exp modal unchanged).
- `tests/unit/test_roll_timeline.py` — 14 cases: no-roll, single roll, mixed-type C/P roll, fully-closed, orphan closes, orphan opens, identical-timestamp determinism, legged-in over-count per spec, open vs closed `current_strike_label`.

**Frontend**
- `frontend/src/components/LedgerRollTimeline.vue` — new component rendering TT-style staged sections (Closing → Roll → Opening). Roll sections show `ROLLED X → Y` with expand toggle for per-leg pair detail.
- `frontend/src/components/LedgerDesktopRow.vue` — header uses `current_strike_label` (backend); Rolls column shows numeric badge for same-exp count, keeps toggle for different-exp chain; expanded panel delegates to `LedgerRollTimeline`.
- `frontend/src/components/LedgerMobileCard.vue` — parity with desktop (same timeline, compact `Rn` badge).

**Live verification**: ran `/api/ledger` handler against the user's data — 31 groups have detected same-exp rolls, e.g. `ALB Bull Call Spread rolls=1 strikes=192.5/205, roll 2026-01-23: 177.5/190 → 192.5/205`. Strategy labels now reflect the current cohort (2 strikes for a call spread, not 4).

Moving to In Review.
