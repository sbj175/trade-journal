# OPT-287: Roll lost when new leg's expiration matches a pre-existing sibling group's expiration

## Symptom

On 2026-05-15 the user rolled IBIT covered calls forward: BTC 5/15 $43.50 + STO 5/22 $44.00 (two orders, 32x and 80x). The closing legs closed their source groups correctly, but **no new "rolled" group was created**. The new STO legs were silently merged into pre-existing OPEN groups from 5/1 that already held an unrelated 5/22 $41.50 short call — the strike expiring on the same date as the new $44 leg. The Ledger shows no roll lineage; the source groups have no descendant link.

## Lot-level lineage is correct

`parent_lot_id` IS set on both new lots: 30622 → 30621, 30623 → 30619. OPT-284 lot-level detection works.

## Root cause

In `src/pipeline/group_manager.py`, `_route_lot_to_group` Rule 1 (account / underlying / expiration) merges any incoming option lot into an existing group at the same `(account, underlying, expiration)` whenever **any** existing lot in that group is still open — the "sizing-up an active position" heuristic.

That rule never consults `parent_lot_id`. So when an opening lot is the new side of a roll AND another open position happens to share the new lot's expiration, Rule 1 fires and absorbs the new lot into the unrelated sibling group.

Downstream, `derive_rolled_from_group_id` (`src/pipeline/lot_lineage.py`) correctly refuses to set `rolled_from` on a mixed-source group (one lot with a parent, one without), so the roll link is dropped entirely.

Secondary issue: the `Lot` dataclass in `src/models/lot_manager.py` does not carry `parent_lot_id` — even though the DB column exists. Until that's threaded through, the routing layer literally cannot consult the canonical signal.

## Targeted fix

1. Add `parent_lot_id: Optional[int] = None` to the `Lot` dataclass.
2. Populate it wherever `Lot` is constructed from a `PositionLot` row.
3. In `_route_lot_to_group` Rule 1, add a guard: if the incoming lot has `parent_lot_id` set AND the parent's lot is in a candidate sibling group whose lots are ALL closed (i.e., would qualify as a "rolled from" source), still merge. But if the parent is in a *different* group (one this lot would not normally inherit from), refuse the merge — fall through and let the caller mint a new group key.

The simpler, equivalent rule: **if `parent_lot_id` is set and the parent is not among the existing group's lots, skip Rule 1's "any open lot" merge**. The "sizing up" heuristic only makes sense when sizing up the same position; if the new lot's parent is somewhere else, it's structurally a roll continuation, not an adjustment.

After the routing fix, `derive_rolled_from_group_id` will see a clean target group (all lots have parents, all parents are in the same closed source) and set `rolled_from_group_id` correctly.

## Test scenario

Add a fixture in the OPT-278 / OPT-281 test matrix:

> Account holds two short calls on the same underlying at *different* expirations (e.g. 5/15 $43.50 and 5/22 $41.50). User rolls the nearer-expiration call forward to a strike matching the farther expiration date (5/15 $43.50 → 5/22 $44.00). Expected: closed source group closes; new group created for the rolled-in lot with `rolled_from_group_id` set to the closed source; the pre-existing 5/22 $41.50 group is untouched.
