# OPT-262: Roll detection misses puts-only roll on multi-leg Iron Condor

## Problem

When a single order contains both closing transactions for existing open lots AND opening transactions for new positions on the same underlying + expiration, the pipeline should recognize that as a **roll** and link the new lots to the old via `derived_from_lot_id` + `derivation_type = ROLL`. For a real user trade it didn't — the pipeline treated the closes as `MANUAL` and the opens as fresh unrelated positions.

Without the derivation link, `GroupManager` passes all historical lots to the strategy recognizer as one blob. The partition algorithm then pairs legs that never coexisted — producing nonsense strategies like `Bear Put Spread 545/585` (legs born on different days) and `Jade Lizard` (a 6/27 short put paired with a 5/22 call spread).

## Repro

* User: [sbj175@gmail.com](<mailto:sbj175@gmail.com>), Traditional IRA
* Symbol: SPY, expiration 2025-07-18
* Trade history:
  * 2025-05-22: Iron Condor opened 530P+ / 545P- / 615C- / 630C+ (+4.14 credit)
  * 2025-06-27: **Puts-only roll** in ONE order (392170105) — STC 530P / BTC 545P / BTO 585P / STO 600P (+1.55 credit)
  * 2025-07-15: Closed 585P/600P/615C/630C (-9.31 debit)
* Observed: fragmented into three groups — `Long Put 530` (5/22→6/27), `Bear Put Spread 545/585` (5/22→7/15), `Jade Lizard 600/615/630` (7/15→7/15)
* Expected: one position group labeled `Iron Condor`, spanning 5/22→7/15, with the 6/27 leg swap reflected as a roll within the chain

## Root cause

`src/pipeline/roll_splitter.py:56-79` assumes a roll has exactly ONE closing leg. The splitter takes `closing_txns[0]` as the anchor (line 56), picks the closest-strike opening leg of the same option_type as its roll target, then dumps **everything else** into a new `{order_id}_split` OPENING order (lines 87-89). For a put-spread roll with 2 closes + 2 opens, that orphans the second close/open pair onto a brand-new chain.

Evidence from DB:

* Lots 9107 (585P long) and 9108 (600P short) have `derived_from_lot_id = NULL`, `derivation_type = NULL`

**No warning or assertion** fires when multiple closing symbols exist. Silent fall-through.

## Where to fix

`src/pipeline/roll_splitter.py:41-79` — generalize the splitter. Two options:

* **Relax**: group `closing_txns` by `(option_type, direction)` and pair each closing group with its own closest-strike open of the same direction. STC (long-close) pairs with BTO (long-open); BTC (short-close) pairs with STO (short-open).
* **Generalize**: pair by direction first, then by option_type, ignoring strike distance when there's exactly one open matching direction+type.

Also:

* `src/pipeline/position_ledger.py:135-141` — chain inheritance uses `next(iter(affected_chains))`; needs to tolerate multi-pair rolls so each new lot inherits from its paired close.
* Consider an assertion/warning when `roll_splitter` orphans transactions, so silent failures surface in logs.
* **Defense in depth**: `GroupManager` should scope strategy recognition to open-at-timestamp snapshots regardless — so a future detection failure can't reproduce this cross-time poisoning.

## Acceptance

* The SPY 5/22 IC + 6/27 roll case produces **one** position group labeled `Iron Condor` with the roll recorded in the chain
* Generic: any single order that closes N legs of an existing open strategy and opens N legs on the same underlying + expiration is detected as a roll and paired by direction
* New lots carry `derived_from_lot_id` → their paired closed lot + `derivation_type = ROLL`
* `order_chains` and `roll_chain_summaries` rows are populated
