## Summary

`roll_chain_summaries.cumulative_realized_pnl` and `cumulative_premium` can become stale during incremental sync. A full reprocess heals the values, but normal `/api/sync` runs (which process only affected_underlyings) leave at least one chain row with wrong totals.

## Repro evidence

Validated against `raw_transactions` for sbj175@gmail.com on 2026-05-16. Four IBIT roll chains were inspected:

| Chain | Account | Modal Realized | RCS row Realized | OK? |
|---|---|---|---|---|
| 3844 (CC 44, 5 rungs) | Traditional IRA | −3,280 | −3,280 | ✓ |
| 3880 (CC 41.5, 8 rungs) | Traditional IRA | −14,791 | −14,791 | ✓ |
| 3845 (CC 44, 5 rungs) | Roth IRA | −1,056 | −1,056 | ✓ |
| **3879 (CC 41.5, 7 rungs)** | **Roth IRA** | **−6,092** | **−3,932** | **✗** |

The modal recomputes from per-rung data and matches raw transactions to the penny. The row caption on the Options list pulls from `roll_chain_summaries` and shows the wrong value for chain 3879 only.

## Diff after full reprocess

| Field | Before (stale) | After reprocess | Δ |
|---|---|---|---|
| cumulative_premium | 8,228 | 6,068 | −2,160 |
| cumulative_realized_pnl | −3,932 | −6,092 | −2,160 |

**Identical $2,160 delta on both metrics.** Per `src/pipeline/roll_chain_summary.py`, both fields include "closings on ancestor lots" — so exactly $2,160 of ancestor-lot closings were either missing or misattributed at the time the stale row was written. Chain length and roll_count were unchanged (7 / 6), so the chain shape was right — only the lot/closing attribution under it was wrong.

## Hypothesis

`populate_roll_chain_summaries` is correct given correct inputs (proven by reprocess healing the row). The bug is upstream — almost certainly in `detect_lot_lineage` (Stage 4b) which sets `position_lots.parent_lot_id`. Under incremental sync, this stage only sees lots for `affected_underlyings`. If lineage decisions depend on lots outside that scope, the pairing comes out differently than a full-portfolio run.

Secondary suspects:
- `populate_pnl_events` (Stage 6)
- `derive_rolled_from_group_id` (Stage 5c)

Stage 7 itself runs unconditionally on every sync and rebuilds from scratch, so it isn't being skipped.

### 2026-05-16 — Claude (root cause)

**Root cause found and reproduced. Not an incremental-sync bug — a non-deterministic order-dependent bug in `detect_lot_lineage`.**

Repro: deleting 2026-05-01 + 2026-05-15 IBIT rows from Roth (5WZ28644), then running full reprocess, surfaced the same defect (Δ +2,160 contamination of chain D). Restoring those rows + reprocessing again *also* reproduced the original buggy value −3,932 — i.e., the same input set produces different `cumulative_realized_pnl` values across runs.

**The mechanism:**

1. On 2026-02-27, Roth IBIT has exactly one MANUAL close (`260227C40` qty 20 @ $0.01 — a dust buyback of an OTM expiring leg) and two SELL_TO_OPEN of `260306C39` qty 20 each.
2. `src/pipeline/lot_lineage.py::detect_lot_lineage` pairs the close with the "closest strike" open. Both opens are distance 1.0 from strike 40, so the tie breaks on `lot_id`. The winning open's `parent_lot_id` is set to the dust close's lot.
3. That dust close itself has a parent chain reaching back: Feb 20 close → Feb 13 close → Feb 11 STO.
4. In `src/pipeline/lot_lineage.py::build_chain_attribution`, lot `33964` (Feb 11, realized $360) and lot `33970` (Feb 13, realized $1,800) hit the *branching* branch (their groups appear in multiple chain-leaves). Walking forward via `child_of`, the terminal lot lands at chain D's May 1 leaf, so they get attributed to chain D. **$360 + $1,800 = $2,160** — the exact contamination.

**Why non-deterministic:** `lot_id` is a Postgres autoincrement serial assigned in INSERT order during reprocess. The two Feb 27 opens are inserted in some order, getting IDs `X` and `X+1`. The closest-strike tiebreak then picks one or the other. On some orderings the resulting `parent_lot_id` graph doesn't propagate the contamination; on others it does. Identical input, different lot ID assignment → different attribution.

**Where to fix (best → worst, IMO):**

- **`detect_lot_lineage`**: don't treat trivial dust closes ($0.01 manual buybacks of about-to-expire OTM legs) as roll predecessors. Heuristics: skip MANUAL closings where `abs(price) < $0.05` AND the lot's strike is far OTM relative to underlying at close time. A roll has economic substance; closing dust does not.
- **`detect_lot_lineage`**: require quantity match (close qty == open qty) for pairing. Mismatched quantities are suspicious for a "roll."
- **`build_chain_attribution` branching tiebreak**: when walking forward lands in a chain whose root group's `opening_date` is *after* the lot's `entry_date`, that's wrong attribution — the lot existed before the chain did. Reject the attribution.

I'd start with the third fix because it's the most defensive (catches this and any future similar lineage glitches) and the cheapest test, then tackle the lineage heuristics for the deeper fix.

### 2026-05-16 — Claude (fix)

Implemented the defense-in-depth fix in `src/pipeline/lot_lineage.py::build_chain_attribution`. In the branching case, the forward walk's `terminal_gid` is now required to be in `candidate_leaves` (the chains containing the lot's source group). When the walk lands outside that set — which happens when an upstream `parent_lot_id` spuriously bridges unrelated chains — attribution falls through to the orphan tiebreak (constrained to legitimate candidates).

Regression test added: `tests/unit/test_lot_lineage.py::TestBuildChainAttribution::test_opt290_forward_walk_into_unrelated_chain_falls_back_to_orphan_tiebreak`.
