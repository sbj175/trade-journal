# OPT-278: Test depth — fixture library, golden snapshots, property-based (epic)

## Why

[OPT-272](https://linear.app/optionedge/issue/OPT-272/test-strategy-coverage-growth-epic) closed out the immediate test-strategy investments (CI on every push, plain-English docstrings on every test, recognition gaps backfilled, routing and chain-pairing helpers extracted with direct unit tests, stale-test cleanup). The bigger architectural ambitions in that epic's original description never landed — captured here as their own focused work.

The single-best motivator: today's session surfaced five separate pipeline bugs ([OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same), [OPT-273](https://linear.app/optionedge/issue/OPT-273/same-direction-multi-leg-roll-collapses-parallel-chains-in-position), [OPT-274](https://linear.app/optionedge/issue/OPT-274/tighten-rule-1-group-routing-require-chain-overlap-with-open-lots), [OPT-275](https://linear.app/optionedge/issue/OPT-275/multi-fill-same-order-opens-split-into-separate-groups-when-later), [OPT-276](https://linear.app/optionedge/issue/OPT-276/iron-condor-strike-label-drops-legs-that-closed-on-earlier-days)), four of which would have shown up as a diff against a golden snapshot of the user's real 9-roll IBIT covered-call chain. Building one good fixture and one good snapshot test would have caught most of them automatically.

## Sub-bullets

### 1\. Fixture library at `tests/fixtures/`

Centralized, named, reusable transaction sequences. Each captures a real-world (or carefully-constructed) scenario as a Python module returning a list of raw_transaction dicts. Replaces inline `make_option_transaction` boilerplate scattered across tests. Examples:

* `iron_condor_two_order.py` — 4 legs across 2 orders
* `parallel_ladder_3_rungs.py` — 3 separate opens + multi-leg roll
* `multi_fill_same_order_roll.py` — 1 open + split-fill roll (24+8)
* `wing_by_wing_close.py` — Iron Condor closed in 2 orders days apart
* `assignment_then_share_close.py` — short put assignment + later sell

Each bug we fix in the future should drop a fixture here so the regression test reads cleanly.

### 2\. Tier 4 end-to-end golden snapshots

For each canonical fixture, compute the post-`reprocess()` state (position_groups + lots + closings + rolled_from chain + strategy_labels + roll_chain_summaries), strip volatile fields (UUIDs, generated timestamps), and save the output as a "golden" reference file. A pipeline test then runs reprocess on the input and asserts equality against the golden.

**The IBIT 9-roll covered-call chain is the canonical first one** — it exercises every pipeline layer in a single test (chain inheritance across 9 generations, parallel-rung routing, multi-fill same-order handling, rolled_from chain walk, summary computation, recognizer label stability).

Maintenance cost: code changes that legitimately alter output require regenerating the golden file. That's a feature — forces you to stop and confirm "yes, this diff is intentional" before merging. Risk of rubber-stamp updates is mitigated by reviewing golden-file diffs in PRs.

### 3\. Tier 3 snapshot tests for the recognizer

Smaller, finer-grained snapshots: feed canonical leg arrangements (verticals, condors, ladders, jade lizards, ZEBRAs, etc.) to the recognizer and snapshot the StrategyResult. Allowed to drift — when the recognizer's heuristics evolve, expected outputs are regenerated and reviewed.

### 4\. Direct tests for the still-untested layers

* `src/pipeline/roll_splitter.py` — [OPT-262](https://linear.app/optionedge/issue/OPT-262/roll-detection-misses-puts-only-roll-on-multi-leg-iron-condor) lives here with zero direct coverage
* `src/pipeline/group_manager.py::_detect_roll_links` — [OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same) hinges on this with zero direct coverage

Both already have integration coverage via the orchestrator tests, but unit tests would catch regressions at a finer grain and faster.

### 5\. Property-based tests via Hypothesis

For invariants that should hold for any input shape:

* "every position_lot has a non-empty chain_id"
* "rolled_from_group_id, when set, points to a real group"
* "the sum of remaining_quantity across closing-event-bound lots is zero"
* "FIFO close ordering is stable across reprocesses"

Hypothesis generates many random inputs and shrinks failures to minimal counterexamples. Modest investment, big leverage on edge cases we wouldn't otherwise think of.

## Discipline carry-forward from [OPT-272](https://linear.app/optionedge/issue/OPT-272/test-strategy-coverage-growth-epic)

These remain in force:

* Bug fixes must include a regression test that fails without the fix
* CI runs on every push; merges blocked on red (already enforced via branch protection)
* New stale tests get repaired or deleted in the same PR that breaks them, never quarantined indefinitely
