# OPT-116: Gracefully handle closing transactions with no matching opening lot

When a user imports partial history (e.g., starting from a date that's after some positions were opened), there will be STC/BTC closing transactions for positions whose opening transactions were never imported. These orphaned closings have no matching lot to close against.

**Expected behavior:** These closing transactions should be silently ignored — no error, no phantom lots, no incorrect P&L.

**Areas to audit:**

* `process_equity_transactions()` in `ledger_service.py` — `close_lot_fifo` returns empty `affected` list when no matching lot exists; currently handled correctly (no error thrown)
* `order_processor.py` — `close_lot_fifo` called for option closings; same behavior
* Verify no warnings/errors are logged that would alarm users
* Verify P&L calculations don't produce incorrect values from unmatched closings

This is a known limitation of partial history import and should be documented as expected behavior, not a bug.

## Comments

### 2026-02-26 — Steve Johnson

**Audit complete — already handled gracefully.**

The core requirement (no error, no phantom lots, no incorrect P&L) is already met:

- **`close_lot_fifo()`** (`lot_manager.py:224-308`): Returns `(0.0, [])` when no matching open lot exists — no exception, no `LotClosing` record created
- **Main BTC/STC closing flow** (`position_ledger.py:159-185`): Calls `close_lot_fifo()` and continues silently if no match
- **Assignment/exercise paths** (`position_ledger.py:295-334, 461-500`): Have explicit deferred-retry logic with debug/warning logs
- **Equity netting** (`ledger_service.py:251`): Silently skips when no counterpart lot exists
- **No phantom lots** are ever created for unmatched closings
- **P&L is correct** — zero P&L for unmatched closings, no inflated numbers

Only gap is observability: no warning log when a regular BTC/STC closing finds no matching lot. This could be added later if needed for debugging partial imports, but is not a functional issue.
