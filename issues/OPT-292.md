# OPT-292: Ledger overview stats over-count P&L across date ranges

The Ledger page's date-filter drives both *which groups are shown* and *what the overview stats sum*. Today those two things use the same group-level filter, but the math should be at the event level. As a result, when a single group touches multiple months (open in one month, roll in another, close in a third), its full realized P&L is counted in **every** month-window it touches — so summing months individually produces a larger total than the same range queried as a single window.

## Reproduction

1. Pick any closed group that was rolled across at least two months — e.g. opened in January, rolled in February, closed in March.
2. Filter Ledger to January only — note the Realized total.
3. Filter to February only — note the Realized total.
4. Filter to March only — note the Realized total.
5. Filter Jan–Mar combined — note the Realized total.
6. `Jan + Feb + Mar ≠ Jan–Mar`.

The same group's realized $ rides along with whichever date-bucket matches; the matching is permissive enough that a multi-month group lands in three buckets but rolls up as one in the combined window.

## Diagnosis

`frontend/src/composables/useLedgerGroups.js:116–135` — a group is included if **any** of these dates falls in the range:

1. `opening_date`
2. `closing_date`
3. `last_activity_date`
4. Any lot's `entry_date`
5. Any lot's closing's `closing_date`

That OR is intentional for *visibility* — you want to see anything that touched the window. The bug is that the overview stats then walk the resulting `filteredGroups` and sum the **group-level rollups**:

| Stat | File:line | Bug |
|---|---|---|
| Realized | `pages/ledger/index.vue:119` | Group's full `realized_pnl` added once per window the group touches |
| Win % | `pages/ledger/index.vue:120,129` | Same group counted as a "win" in every window it appears in |
| Return % | tooltip at `pages/ledger/index.vue:246` | Both numerator (sum realized) and denominator (sum \|initial premium\|) use group rollups |

## Fix

Decouple the display filter from the totals math.

**Display list — unchanged.** Keep the inclusive OR-filter so the user still sees context (open positions in flight, rolled chains, partial closes touching the window).

**Overview stats — recompute from closing events.**

- Sum realized from `lot_closings.realized_pnl` (or `pnl_events` — already a denormalized fact table built for time-based P&L in OPT-176) where the closing's `closing_date` is in the window.
- Recompute Win % from the same event source: a "win" is a *group* whose closings inside the window sum to a positive realized — or simpler, count closing events with realized > 0 over total closing events. Pick whichever is easier to explain in the tooltip and stick with it.
- Recompute Return % numerator from the event-sourced realized; align the denominator to the same scope (capital deployed for the closings in the window, not initial premium of every group displayed).

With this split: **monthly sums equal the range sum exactly**, and the numbers reconcile no matter how the window is sliced.

## UX — the list/totals mismatch needs to be obvious

After the fix, the displayed group list and the overview totals are answering two different questions:

- **List:** "What groups touched this window?" (visibility)
- **Totals:** "What realized P&L happened in this window?" (event-sourced)

That's a feature, not a leak — but only if the user understands it. Without a label, somebody will spot a row in the list with $1,000 realized while the overview shows $400 and assume the page is broken.

Options for surfacing this, in increasing weight:

1. **Tooltip on each overview stat** explaining "counts only closings dated inside the window — list below shows all groups that touched the window."
2. **Subcaption line** under the overview row, e.g. `Totals reflect closing activity in window · List shows groups touched`.
3. **Per-row column** on each group showing the slice of its realized that landed in the window (so the list arithmetic also reconciles to the overview).

Option 3 is the most explicit but the most invasive — adds a column. Option 2 is the cleanest one-line affordance. Option 1 is the bare minimum.

Recommend starting with **option 2 + option 1 tooltips**, deferring option 3 unless users keep asking.

## Out of scope

- The "Reports" page already uses `pnl_events` correctly — no change there.
- Strategy targets / win-rate math under `frontend/src/composables/useLedgerGroups.js:188` is per-group (`realized_pnl ÷ basis`) and is correct *for that group*; only the *overview aggregate* needs the event-based source.

## Acceptance

- Sum of monthly Realized totals equals the same range queried as one window — test across at least three rolled-group examples.
- Win % and Return % are also stable when the window is sliced/unsliced.
- The list still shows multi-month groups in every window they touch; nothing visible disappears compared to today.
- A caption or tooltip makes the list-vs-totals scope explicit.
