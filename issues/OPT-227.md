# OPT-227: Insights Trade Questionnaire

"Can all of these things be determined from raw transaction data? Or must I solicit input from the user?"

Short answer:

About 70–85% of the dashboards I listed can be computed automatically from raw transaction data.

The remaining 15–30% require lightweight user input (or smart inference heuristics).

The trick for OptionLedger is deciding what to infer vs what to ask once vs what to ask every trade.

Here’s the practical breakdown 👇

\---

What you can compute automatically from transaction data

If you ingest full broker activity (fills, timestamps, symbols, quantities, strikes, expirations, prices), you can derive:

1\. Win rate

Fully automatic.

Requires:

entry fills

exit fills

\---

2\. Avg win / avg loss

Fully automatic.

Needs:

net credit/debit

closing P&L

\---

3\. Expectancy

Fully automatic once wins/losses known.

\---

4\. Strategy detection (usually automatic)

Example inference logic:

Pattern	Strategy

short call + long call same expiry	call spread

short put + long put same expiry	put spread

both sides	iron condor

single short put	CSP

single short call naked	short call

Most strategies can be classified algorithmically.

Edge cases needing help:

diagonals

calendars

ratio spreads

broken wing structures

Still mostly solvable.

\---

5\. Profit by symbol

Fully automatic.

\---

6\. Position size analysis

Fully automatic if you compute:

max risk per trade

Example:

Credit spread risk:

spread width − credit received

Iron condor risk:

max width − total credit

\---

7\. Days-in-trade analysis

Fully automatic:

exit date − entry date

\---

8\. Entry DTE (days to expiration)

Fully automatic from option metadata.

\---

9\. Held-to-expiration vs closed early

Fully automatic.

\---

10\. Rolling detection (mostly automatic)

A roll looks like:

close position A

open similar position B same symbol shortly after

Broker activity logs make this detectable.

Example signals:

same ticker

same strategy

same day

same quantity

High confidence roll detection.

\---

What you cannot reliably infer without user input

These are the high-value behavioral analytics traders actually want.

\---

1\. Trade intent (big one)

Example:

Was this:

short put

actually:

CSP income strategy?

wheel strategy entry?

synthetic limit order?

volatility trade?

Transactions don’t tell you.

Intent matters for analytics.

\---

2\. Entry reason

Example:

IV high

technical breakout

earnings play

theta decay harvest

delta targeting

portfolio hedge

This powers advanced edge detection.

Requires tagging (but only once per trade).

\---

3\. Adjustment classification

Example:

User rolls trade.

System sees:

close + reopen

But doesn't know:

defensive roll

profit extension roll

assignment avoidance roll

strategy shift

That insight requires tagging.

\---

4\. Earnings proximity logic

You can auto-detect earnings dates via API…

…but you still need confirmation whether trade was:

earnings play

avoid earnings

accidental overlap

Huge difference analytically.

\---

5\. Emotional mistakes (highest value journal insight)

Impossible to infer automatically:

revenge trade

oversized trade

rule violation

FOMO entry

late exit hesitation

These create the biggest improvements.

\---

The ideal OptionLedger design strategy

Ask the user exactly 3 lightweight things per trade

That unlocks almost everything:

Field 1

Strategy intent:

Income

Directional

Volatility

Wheel

Hedge

Earnings

Other

\---

Field 2

Entry reason:

High IV

Technical setup

Support/resistance

Delta target

Expiration cycle timing

Portfolio balance

\---

Field 3

Management plan (before entry)

Example:

Close at 50%

Hold to expiration

Roll at tested strike

Gamma scalp

Assignment acceptable

Now OptionLedger can later say:

\> Trades following your 50% exit rule outperform everything else by +0.21R

That’s powerful.

\---

The surprising truth about trading journals

Users hate typing after trades.

But they will answer 2–3 taps before entry if the payoff is better analytics later.

So the winning formula is:

Auto-detect everything possible

Ask only what unlocks behavioral edge detection

That’s how OptionLedger becomes smarter than spreadsheets without becoming annoying.

### 2026-03-22 — Steve Johnson

"OptionLedger syncs trades automatically via API and I can envision a questionnaire that queues up questions for every new trade entered."

That’s exactly the right architecture. An auto-sync + queued micro-questionnaire approach is how the best trading journals become sticky instead of burdensome.

The key is this:

\> Only ask questions that unlock analytics the system cannot infer.

If OptionLedger does this well, the questionnaire becomes a competitive advantage, not friction.

Here’s how I’d design it.

\---

The “Queued Questions” model (best practice structure)

Instead of asking everything immediately, attach a question lifecycle to each trade:

Timing	Ask

Immediately after entry	intent + plan

During trade	adjustment reason

At exit	outcome evaluation

Weekly review	behavior reflection

This keeps each interaction tiny (1–2 taps).

\---

Stage 1 — Questions to ask right after trade entry (highest ROI)

These unlock \~80% of behavioral analytics.

Question 1: Trade intent

Example UI:

What type of trade is this?

Choices:

Income

Directional

Volatility

Wheel

Earnings

Hedge

Other

Why this matters:

Later OptionLedger can say:

\> Your income trades outperform directional trades by 2.3×

That’s powerful insight.

\---

Question 2: Exit plan (before emotions exist)

Example UI:

Planned exit?

Choices:

Close at 50%

Close at 25%

Hold to expiration

Roll if tested

Assignment acceptable

Custom

This unlocks:

Rule-following analytics.

Example insight:

\> Trades where you followed your plan outperform rule-breaking trades by +0.31R

Extremely valuable.

\---

Question 3: Confidence level (1 tap slider)

Example:

Confidence level?

Low | Medium | High

Later insight:

\> High-confidence trades underperform your medium-confidence trades

Very common discovery.

\---

Stage 2 — Questions triggered only when adjustments happen

Triggered automatically if API detects roll / partial close / strike change.

Example:

Why did you adjust?

Choices:

Defense

Profit extension

Strike tested

Reduce risk

Free capital

Changed outlook

Now OptionLedger can answer:

\> Defensive rolls improve drawdown but reduce expectancy

Professional-level analytics.

\---

Stage 3 — Exit-time questions (tiny but powerful)

Ask exactly ONE:

Did you follow your plan?

Yes / No

If No → show quick follow-up:

Why not?

Choices:

Fear

News event

Time pressure

Changed outlook

Mistake

This enables:

Behavioral leak detection.

Example output later:

\> Trades exited early due to fear cost you $1,240 this quarter

That’s journal magic.

\---

Stage 4 — Weekly reflection (optional but elite feature)

Instead of tagging every trade emotionally (too much work), ask once weekly:

Example:

Choices:

Oversizing

\---

The secret sauce: conditional questioning

Never ask everything.

Example logic:

If strategy = iron condor

Ask:

Was this delta-balanced?

If earnings window detected automatically

Ask:

Was this an earnings play?

If roll detected

Ask:

Why did you roll?

Smart journals feel intelligent instead of intrusive.

→ Income

Total time:

≈ 4 seconds.

Massive analytics unlocked later.

Why this matters for OptionLedger specifically

Because your system already:

✔ auto-syncs trades

✔ knows strikes

✔ knows expirations

✔ knows fills

✔ detects rolls

That means your questionnaire should focus only on:

intent

plan

adjustment reason

rule-following behavior

If you design the queue correctly, OptionLedger becomes something rare:

A journal that explains why traders make or lose money, not just whether they did.

### 2026-03-22 — Steve Johnson

"What to do about all of the historical trades that won't have this info?"

This is one of the most important product-design questions for OptionLedger—and handled well, it can actually become a feature advantage instead of a limitation.

The short answer:

\> Don’t try to backfill everything. Infer what you can, estimate what’s reasonable, and progressively enrich the rest over time.

Here’s the right strategy used by serious analytics platforms.

\---

Step 1 — Auto-classify everything possible from historical data

Your historical trades already support most analytics:

You can compute:

✅ strategy

✅ entry DTE

✅ time in trade

✅ win rate

✅ expectancy

✅ symbol performance

✅ position sizing

✅ roll detection

✅ early exit vs expiration

✅ adjustment timing

✅ credit vs debit behavior

So Dashboard 1–5 from earlier still work immediately.

Users get value on day one.

That’s critical.

\---

Step 2 — Label missing behavioral fields as “Unknown” (not blank)

Instead of:

exit_plan = null

Use:

exit_plan = unknown

Why?

Because later analytics can say:

\> Trades tagged “Close at 50%” outperform untagged trades by 18%

This motivates tagging.

\---

Step 3 — Use inference heuristics for historical exit plans

You can often guess the exit plan probabilistically.

Example:

If trader repeatedly exits spreads at \~45–55% profit:

Infer:

likely 50% rule trader

If trader exits mostly near expiration:

Infer:

likely hold-to-expiration strategy

Then show:

\> Detected pattern: You usually close at \~50% profit. Confirm?

One tap confirmation = retroactive dataset upgrade.

Huge UX win.

\---

Step 4 — Add a “Retro Tagging Wizard” (high-value feature)

Prompt user once:

Help OptionLedger improve your past analytics

Ask:

Question 1

Most of your credit spreads were entered for:

Choices:

income

directional

mixed

Apply globally to matching trades.

\---

Question 2

Typical exit rule?

Choices:

25%

50%

expiration

variable

Apply retroactively.

Now suddenly:

12 months of trades become behavior-aware.

Users love this.

\---

Step 5 — Allow bulk tagging by filter (power-user magic)

Example:

User filters:

Strategy = iron condor

Symbol = SPX

Date = last year

Then taps:

Tag as Income strategy

Instant dataset upgrade.

Very powerful.

\---

Step 6 — Let analytics improve as tagging improves (progressive intelligence)

Show:

Behavioral accuracy: 42%

Then:

Tag 6 more trades to unlock management analytics

This gamifies completion.

Robinhood-style progress psychology works extremely well here.

\---

Step 7 — Use clustering to auto-tag historical trades (advanced but realistic)

Your system can detect patterns like:

Example cluster:

SPX

45 DTE

short strikes \~16 delta

exit near 50%

System suggests:

\> These 37 trades look like your standard income spreads. Tag them as Income trades?

One tap.

Massive upgrade.

\---

Step 8 — Most important insight: historical tagging doesn’t need perfection

Even:

70% labeled dataset

produces excellent analytics.

Trading journals aren’t accounting systems.

They’re decision-feedback systems.

Approximate intent still delivers value.

\---

The ideal onboarding flow for OptionLedger

When user connects broker:

Step 4:
