# OPT-272: Test strategy + coverage growth (epic)

## Why

In a single afternoon investigating [OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same) we surfaced four pipeline bugs ([OPT-244](https://linear.app/optionedge/issue/OPT-244/roll-not-detected-when-additional-leg-opened-alongside-roll) chain orphaning, [OPT-262](https://linear.app/optionedge/issue/OPT-262/roll-detection-misses-puts-only-roll-on-multi-leg-iron-condor) same-direction multi-leg pairing, the chain-strategy-label leak, and `position_ledger`'s `next(iter(affected_chains))` collapse) — none of which had automated coverage. We also found 6 stale tests against a removed schema that had been quietly failing for a month, and no CI running anything. This is the predictable cost of layered logic + heuristic interpretation + zero enforcement.

## Strategy: tier-based test surface

Reuse the tier model we adopted in [OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same):

**Tier 1 — deterministic (lots / chain_id).** Highest-density coverage. Unit tests for `position_ledger.py`, `roll_splitter.py`, `lot_manager.py`. Pin invariants like "every opening lot has a chain_id," "every roll preserves chain identity 1:1," "FIFO closing is stable across reprocesses." This is where mistakes propagate widest.

**Tier 2 — structural (groups).** Invariant + edge-case tests in `group_manager.py`. Cover: parallel ladder rungs at same expiration, strikes-converge-on-roll, multi-strike-same-day, partial close + open in one order. Assert "group contains lots from one chain" and "rolled_from set whenever chain matches across days."

**Tier 3 — heuristic (strategy_label).** Snapshot tests only. Allowed to drift. No downstream layer may depend on these labels (already enforced in [OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same)).

**Tier 4 — relational (roll chains, pnl_events, summaries).** End-to-end golden-snapshot tests. Take anonymized real transaction sequences (covered call ladder, iron condor, put spread roll, assignment + roll) and snapshot the resulting position_groups + chain_summaries.

## Sub-issues

- [ ] **CI setup** — GitHub Actions workflow running pytest on every push and PR. (Initial commit lands on the [OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same) branch.)
- [ ] **Fix the 6 stale tests** — either repair against current schema or delete. Currently quarantined via pytest config on the [OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same) branch.
- [ ] **Tier 1 coverage backfill** — `roll_splitter.py` (no tests today), `position_ledger.py` chain assignment (no tests today). Start with the parallel-ladder test landed alongside [OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same) (currently `xfail` until the chain-pairing fix lands).
- [ ] **Tier 2 coverage backfill** — `group_manager.py` routing edge cases. Cover Rule 0/1/2/3 explicitly. Cover `_detect_roll_links` chain matching.
- [ ] **Fixture library** — `tests/fixtures/` with canonical real-world transaction sequences (multi-fill rolls, parallel ladders, assignments, exercises, symbol changes). Grow incrementally as bugs surface.
- [ ] **Tier 3 snapshot suite** — recognizer outputs for canonical strategies, regenerate with `--update-snapshots`.
- [ ] **Tier 4 end-to-end snapshots** — take a frozen anonymized account dump, run pipeline, diff against golden output. Catches pipeline-wide regressions like today's chain collapse.
- [ ] **Property-based tests** (Hypothesis) for invariants. Small investment, big leverage on edge cases.

## Discipline

* New bug-fix work must include a regression test that fails without the fix (today's [OPT-270](https://linear.app/optionedge/issue/OPT-270/roll-chain-breaks-when-two-independent-positions-on-the-same) / parallel-ladder flow is the template).
* CI runs on every push; merges blocked on red.
* Don't allow stale tests to sit failing — repair, skip with explicit reference, or delete in the same PR that breaks them.

### 2026-04-25 — Steve Johnson

Two refactor opportunities surfaced during a /simplify code-review pass. Tracking here for follow-up — not blocking any current work, but worth a single focused PR each:

1. **Extract `_route_lot` shared helper.** `assign_lots_to_groups` (pure, ~L120-180) and `GroupPersister.process_groups` (DB, ~L770-820) now have nearly identical Rule 0 / Rule 1 / Rule 2 / Rule 3 routing blocks. They've drifted in lockstep through OPT-260, OPT-270, OPT-274 — but every change requires touching both. Extract a helper that takes the four indices (`aue_to_group`, `au_to_group`, `chain_to_group`, lots-by-group) and an `add_fn` callback, returning the assigned group key. Both call sites collapse to ~5 lines.

2. **Extract `_build_chain_queue` and `_assign_opens_to_chains` from `position_ledger.py`.** The OPT-273 ROLLING-order chain pairing block has three nesting levels (`if order_type==ROLLING / for tx / for lot / while`). Splitting into two pure helpers (`_build_chain_queue(closing_txs, get_open_lots) -> List[Tuple[chain_id, qty]]` and `_assign_opens_to_chains(opening_txs, chain_queue) -> Dict[tx_id, chain_id]`) gives each piece an independently testable surface and flattens the loop.

3. **Test fixture helper for open-then-roll scenarios.** Six new tests in `TestParallelLadderRollPreservation` and `TestRollMechanics` build very similar Day-1-open-then-Day-2-roll transaction lists. A `_open_then_roll(*, sym_old, sym_new, ...)` helper would cut ~150 lines from `test_orchestrator.py` and make each test's intent obvious. The mixed-quantity and put-spread tests have multi-leg shapes — leave those inline.

None of these are bug-driven; they're maintenance investments. Worth doing before the next pipeline-layer change to avoid drift between the two routing paths.
