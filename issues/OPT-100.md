# OPT-100: Frontend migration: Alpine.js → Vue 3

## Context

Alpine.js was the right choice for prototyping but won't scale to a commercial product. Key blockers: no unit testing story, CSP compliance requires `unsafe-eval`, no component model (single 1,500+ line functions), full DOM re-renders on every WebSocket update, and the Tailwind CDN build ships \~600KB per page.

Vue 3 is the recommended target — its directive syntax maps nearly 1:1 from Alpine (`x-data` → `setup()`, `x-for` → `v-for`, `x-show` → `v-show`, `Alpine.store()` → Pinia), making it the lowest-friction migration path for a team that already thinks in Alpine's model.

---

## Phase 1: Preparation (Do Now, While Still on Alpine)

These improvements are valuable regardless of migration timing and reduce the eventual migration scope:

1. **Extract inline components to separate** `.js` files — Move `positionsApp()`, `ledgerApp()`, etc. out of HTML and into `static/js/` files registered with `Alpine.data()`. This separates business logic from markup.
2. **Adopt** `Alpine.store()` for shared state — Replace the 33 `localStorage` usages that act as cross-page state with proper Alpine stores. This maps directly to Pinia later.
3. **Add Vite as a build step** — Solves the Tailwind CDN problem (600KB → \~10KB with PurgeCSS), enables proper JS modules, and Vite is the default build tool for Vue so it carries over directly.
4. **Extract pure business logic into standalone JS modules** — P&L calculations, strategy detection, sort comparators, formatting helpers. These can be unit tested today and reused unchanged in Vue.

---

## Phase 2: Migration (When to Switch)

**Trigger:** Begin the Vue migration when *any* of these conditions are met:

* The backend SaaS migration is complete (auth, multi-tenant, cloud DB) and you're ready to onboard users
* You need to add a major new frontend feature that would be painful in Alpine (e.g., drag-and-drop portfolio builder, complex form workflows, shared collaborative views)
* You find yourself fighting Alpine's limitations more than building features

**Don't trigger on:** Minor feature additions, bug fixes, or UI polish — those are fine to keep doing in Alpine.

**Migration approach:**

* Page-by-page migration, not a big-bang rewrite — Vue and Alpine can coexist during transition
* Start with the simplest page (Settings) to establish patterns, then Reports, Ledger, Risk, and finally Positions (most complex)
* Each page migration: extract Alpine component → convert to Vue SFC → add Pinia store if needed → add component tests
* Estimated 6-10 weeks for all 5 pages

---

## Tech Stack Target

* **Vue 3** with Composition API (`<script setup>`)
* **Pinia** for state management
* **Vite** for build tooling (introduced in Phase 1)
* **Vitest** for unit/component testing
* **Vue Router** if/when moving from MPA to SPA (optional, can stay MPA initially)
* Keep **ApexCharts** and **Chart.js** (both have Vue wrappers)
