# OPT-199: Redesign position grouping: strategy-only groups with roll chain visualization

## Background

The current grouping logic lumps all related trades into one mega-group. For recurring strategies like covered calls, this creates unwieldy groups spanning months with dozens of lots (e.g., IBIT covered calls dating back to May 2025 in one group). The Ledger becomes hard to navigate and individual trade decisions are buried.

Meanwhile, Reports (pnl_events) already handles cumulative P&L correctly, making the mega-group's accumulated P&L redundant for analytical purposes.

## Design Discussion

### Problem Statement

* Recurring income strategies (weekly covered calls, serial CSPs) create bloated groups
* Individual trade decisions are hard to review
* The Ledger P&L total was already removed because Reports handles time-bucketed P&L better
* Current grouping won't work well with SnapTrade (which may lack `order_id`)

### Proposed Grouping Model

**1. Strategy-only grouping**: Group options only enough to form a strategy — a Bull Put Spread (2 legs opened together) is one group. The *next* spread you open is a *new* group.

**2. Shares are independent**: Equity positions live in their own groups (or a separate Equity Positions page). They are NOT lumped in with option groups.

**3. Covered Call detection via account context**: A short call group is labeled "Covered Call" if the account holds sufficient shares of the underlying. Detection is contextual (query account inventory) not structural (same group). This already exists in `StrategyDetector._has_stock_coverage()`.

**4. Roll chains as relationships, not groups**: When Group A is rolled into Group B, they are linked but remain separate groups on the Ledger. Each has its own dates, P&L, and lifecycle. The chain is navigable, not forced into one blob.

### New Roll Definition (SnapTrade-compatible)

A roll is defined as: a lot close and lot open on the **same day**, for the same **account / underlying / option type**, with a different **strike and/or expiration**. This works with or without `order_id`, making it broker-agnostic.

### Roll Chain Visualization

A new page or popup (accessible from the Ledger) to visually display roll chains:

* Shows the progression: Group A → Group B → Group C (current)
* Each node shows dates, strikes, expiration, and realized P&L
* Cumulative chain P&L visible at a glance
* Navigable — click a node to jump to that group on the Ledger

### What This Solves

* **IBIT covered calls**: Each weekly call is its own group, clean and reviewable
* **WDC Bull Put Spread**: The roll from Mar20 to Apr17 shows as two linked groups, each with clear P&L
* **SnapTrade compatibility**: Roll detection works without `order_id`
* **Ledger usability**: Each entry represents one trade decision

### Competitive Context

* **OptionIncome**: Shows each trade cycle separately, has a separate Equity Positions page, detects covered calls via account context (same approach we're proposing)
* **Wingman**: Groups equities and options together (similar to our current approach)

### Open Questions

* Should equity positions get their own page, or be shown as a collapsible section on the Ledger?
* How should the roll chain view be accessed — dedicated page, popup/modal from Ledger, or inline expansion?
* When a roll chain is displayed, should it show the Roll Analysis metrics for the full chain?
* How to handle groups that were created under the old logic — migration path?

## Comments

### 2026-03-12 — Steve Johnson

## Pipeline Impact Analysis

Completed a comprehensive analysis of the current 7-stage pipeline to identify what needs to change for the grouping redesign. The good news: the changes are surprisingly contained.

### Files That Need Changes

**1. `src/pipeline/group_manager.py` — Major changes**

This is the primary file affected. Two key areas:

- **`assign_lots_to_groups()` (lines 53-200)**: The pure grouping function currently has 4 rules:
  - **Rule 0** (derived equity): Auto-groups shares with options → **REMOVE** (shares become independent)
  - **Rule 1** (chain-based): Groups by chain → **MODIFY** to group by *trade cycle* (same-day strategy legs only)
  - **Rule 2** (expiration match): Multi-leg strategy grouping → **KEEP** but scope to same-day entries
  - **Rule 2b** (equity match): Matches equities → **REMOVE** (no longer needed)
  - **Rule 3** (new group): Fallback → **KEEP** as-is

- **`GroupPersister.process_groups()` (lines 294-627)**: Needs new logic to detect roll relationships between groups and persist them to the new relationship table.

**2. Database schema — New table**

A new `roll_chains` (or `group_rolls`) table to link predecessor → successor groups:
- `id`, `predecessor_group_id` (FK → position_groups), `successor_group_id` (FK → position_groups), `roll_date`, `user_id`
- This is the backbone of the threaded/unthreaded Ledger view and roll chain visualization

**3. `src/pipeline/order_assembler.py` — Minor tweak**

Roll detection in `classify_order()` (lines 290-306) currently relies on order-level data. The new broker-agnostic roll definition (same day + same account/underlying/option type + different strike/expiration) needs to work without `order_id` for SnapTrade compatibility.

### Files That Stay Unchanged

- **`chain_graph.py`** — Union-find chain derivation still useful for linking related lots at the lot level
- **`lot_manager.py`** — Lot creation and FIFO closing are independent of grouping
- **`strategy_engine/recognizer.py`** — Strategy recognition works on lot legs, not groups — adapts automatically
- **`pnl_events.py`** — Reads from groups/lots, will reflect new grouping automatically
- **`orchestrator.py`** — Pipeline stages stay the same, just the grouping logic within them changes

### Implementation Approach

1. Add the `roll_chains` relationship table + Alembic migration
2. Rewrite `assign_lots_to_groups()` with simplified rules (strategy-only grouping)
3. Add roll chain detection logic in `GroupPersister` (same-day close+open matching)
4. Add API endpoint to query roll chains for a given group
5. Frontend: Ledger threaded/unthreaded view + roll chain visualization

### Key Insight

The current pipeline architecture cleanly separates concerns (lots → chains → groups → P&L events), which means the grouping redesign doesn't cascade into unrelated stages. The lot-level data model remains correct regardless of how lots are grouped.
